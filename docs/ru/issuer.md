# Инструменты выпуска (`tessera_issuer`)

## Обзор

Tessera проверяет сертификаты на устройстве, но сами сертификаты нужно чем-то
выпускать. Инструменты выпуска закрывают эту сторону: одно Rust-ядро собирает
`TBSCertificate` с расширениями Tessera, проверяет монотонное сужение рамок
делегирования **до** подписи и подписывает результат ключом выбранного бэкенда —
токена/HSM (PKCS#11), Vault Transit или локального PKCS#8-файла. С бэкендами
PKCS#11 и Vault инструмент — **не кастодиан**: приватные ключи через код выпуска
не проходят; файловый бэкенд — осознанный компромисс, ключ живёт в памяти
процесса выпуска (см. [threat-model.md §11](threat-model.md)).

Компоненты:

- **Ядро** (`tessera_issuer`, библиотека) — сборка TBS листа смены и CA
  организации, проверки рамок, случайные 128-битные серийники, выпуск CRL,
  журнал выпусков. Ядро pure-Rust и собирается под `wasm32` (для
  веб-кабинета); адаптеры подписи — за feature-флагами.
- **CLI `issuer`** — тот же код выпуска для автоматизации (тикет-системы,
  скрипты). Ни одна проверка в CLI не переопределяется: запрос, который отверг
  бы ядро, CLI отвергает точно так же.
- **Агент `issuer serve`** — локальный HTTP-сервер строго на `127.0.0.1`: мост
  «браузер ↔ токен» и раздача самого кабинета (по умолчанию).
- **Веб-кабинет** — один статический SPA (WASM-сборка того же ядра):
  выпуск через браузер без серверной части. Раздаётся локально самим агентом
  (`issuer serve`, кабинет встроен в бинарь) или отдельной
  статикой в закрытом контуре (см. раздел «Веб-кабинет» ниже).

Модель ролей — из предъявленного родительского сертификата: любой CA с рамками
делегирования (корень парка или CA организации) выпускает и подчинённые CA
организаций, и листы смен инженерам — строго в своих рамках делегирования,
которые на каждом шаге могут только сужаться. Отдельного «режима по
должности» нет.

Семантику самих расширений (`host_binding`, `user_binding`, `allowed_roles`,
`max_integrity`, `profile_version`, `delegation_constraints`) и их OID см. в
[cert-issuance.md](cert-issuance.md) — здесь описан инструмент, а не формат.
Поверхность атаки инструментов выпуска разобрана в
[threat-model.md §11](threat-model.md); дублировать её тут не будем.

## Быстрый старт CLI

Все выпускающие подкоманды выбирают бэкенд подписи флагами `--backend`
(`pkcs11` — по умолчанию, `vault`, `file`), `--key` (метка ключа CA;
для `file` — опционален, по умолчанию имя файла ключа), `--algorithm`
(`ecdsa-p256` — по умолчанию, `ecdsa-p384`, `rsa-sha256`; для `file`
алгоритм выводится из самого ключа, а флаг работает как сверка). Времена (`--not-before`,
`--not-after`, `--this-update`, …) задаются в Unix-секундах. Вход (`--parent`,
`--spki`, `--csr`, `--issuer`) принимается в PEM или DER — формат определяется по
содержимому. Выход — PEM, либо DER при `--der`.

PIN токена **никогда** не передаётся аргументом командной строки: PKCS#11-бэкенд
запрашивает его через pinentry на время операции, а при отсутствии pinentry —
из переменной `TESSERA_ISSUER_PIN` (см. [Бэкенды подписи](#бэкенды-подписи)).

### Выпуск CA организации

Под корневым сертификатом парка выпускается CA организации с назначением рамок
делегирования (роли, потолок уровня МКЦ, потолок TTL, требуемые метки):

```sh
issuer issue-ca \
    --backend pkcs11 --module /usr/lib/x86_64-linux-gnu/opensc-pkcs11.so \
    --key tessera-root --algorithm ecdsa-p256 \
    --parent root.pem \
    --spki org-ca.spki.der \
    --subject "CN=Org North CA,O=Org" \
    --not-before 1750000000 --not-after 1900000000 \
    --allow-role oper --allow-role serv \
    --max-level 5 --max-ttl 14400 \
    --require-tag region=north \
    --journal issuance.ndjson \
    --out org-ca.pem
```

Флаги `--allow-role`, `--require-tag` повторяются для нескольких значений.
Рамки выпускаемого CA обязаны быть ⊆ рамок родителя — иначе ядро отказывает
до подписи с указанием измерения (см. монотонное сужение в
[cert-issuance.md](cert-issuance.md)).

### Выпуск листа смены

Публичный ключ листа берётся из явного `--spki` (тогда `--subject` обязателен)
**или** из `--csr` (тогда субъект и ключ берутся из запроса). Флаги взаимно
исключающие.

Прямой путь (SPKI):

```sh
issuer issue-leaf \
    --backend pkcs11 --module /usr/lib/.../opensc-pkcs11.so \
    --key org-north-ca \
    --parent org-ca.pem \
    --spki ivanov.spki.der \
    --subject "CN=ivanov,O=Org" \
    --host "sha256:<host_id_hash>" \
    --user ivanov \
    --role oper \
    --not-before 1750000000 --not-after 1750086400 \
    --max-integrity-level 2 --max-integrity-categories 0x1 \
    --journal issuance.ndjson \
    --out ivanov.pem
```

Путь по CSR (см. [CSR-поток](#csr-поток)):

```sh
issuer issue-leaf \
    --backend pkcs11 --module /usr/lib/.../opensc-pkcs11.so \
    --key org-north-ca \
    --parent org-ca.pem \
    --csr ivanov.csr.pem \
    --host "sha256:<host_id_hash>" --user ivanov --role oper \
    --not-before 1750000000 --not-after 1750086400 \
    --journal issuance.ndjson \
    --out ivanov.pem
```

`--host`, `--user`, `--role` повторяются. `--max-integrity-level` опционален
(без него потолок целостности не задаётся); `--max-integrity-categories`
(битовая маска) учитывается только вместе с уровнем.

### Выпуск CRL

```sh
issuer issue-crl \
    --backend pkcs11 --module /usr/lib/.../opensc-pkcs11.so \
    --key org-north-ca \
    --issuer org-ca.pem \
    --this-update 1750000000 --next-update 1750604800 \
    --crl-number 7 --last-crl-number 6 \
    --revoke 2a:1750000500:1 \
    --revoke 3b:1750000600 \
    --journal issuance.ndjson \
    --out org-ca.crl
```

`--crl-number` обязан быть строго больше `--last-crl-number` (монотонность
`crlNumber` в стейте CA) — иначе отказ. Каждый `--revoke` — это
`serial_hex:unix_date[:reason_code]`, где `reason_code` — код причины RFC 5280
(0–6), опционален; флаг повторяется.

### Верификация журнала

```sh
issuer verify-journal --journal issuance.ndjson
```

Печатает одно из трёх состояний: цепочка цела и полностью подписана; цела, но
хвост не подписан (с номером `seq`, с которого); нарушена (с позицией первой
невалидной записи — тогда ненулевой код возврата). См.
[Журнал выпусков](#журнал-выпусков).

### Язык сообщений

Результатные сообщения оператору локализованы (RU/EN). Локаль: флаг `--lang`
(`ru`/`en`) → `TESSERA_ISSUER_LANG` → `LANG` → английский по умолчанию.
Совпадение по префиксу: любое значение, начинающееся на `ru`, выбирает русский.
Технические идентификаторы (субъект RFC 4514, OID, `crlNumber`, серийники) не
переводятся. См. [Локализация](#локализация).

## CSR-поток

CSR (PKCS#10) — равноправный с прямым SPKI источник ключа листа. Он снимает
необходимость передавать инструменту публичный ключ отдельно и даёт
proof-of-possession: инженер генерирует ключ на своём токене и подписывает им
запрос.

Сторона инженера — сформировать CSR ключом на токене:

```sh
issuer csr \
    --backend pkcs11 --module /usr/lib/.../opensc-pkcs11.so \
    --key ivanov-token-key --algorithm ecdsa-p256 \
    --subject "CN=ivanov,O=Org" \
    --spki ivanov.spki.der \
    --out ivanov.csr.pem
```

Инструмент только подписывает: публичный ключ инженера (`--spki`) подаётся
явно, запрос подписывается тем ключом токена, что адресует `--key`.
Proof-of-possession действителен, только если ключ токена соответствует
`--spki` — это ответственность инженера, ключи инструмент не генерирует.

Сторона оператора — `issue-leaf --csr` (см. выше). Что важно:

- Ядро проверяет самоподпись CSR (P-256/RSA, pure-Rust) **до** выпуска; битая
  самоподпись → отказ до подписи. CLI дополнительно печатает субъект CSR и
  статус самоподписи перед выпуском.
- Субъект и публичный ключ берутся из CSR. **Скоуп (рамки, привязки, роли)
  задаёт исключительно оператор** флагами — атрибуты CSR на состав расширений
  не влияют. Иначе CSR стал бы каналом «инженер сам запросил себе шире».

## Агент `issuer serve`

`issuer serve` — локальный HTTP-сервер строго на `127.0.0.1`: (1) мост
«браузер ↔ токен» — принимает готовый TBS от кабинета и возвращает подпись
(кабинет не может обратиться к PKCS#11 напрямую); (2) по умолчанию раздаёт сам
кабинет с того же адреса. Когда раздача активна, весь кабинет живёт в одном
подписанном бинаре — отдельный хостинг не нужен. Флаг `--no-cabinet` отключает
раздачу (чистый режим моста, когда SPA раздают снаружи).

Агент поддерживает те же бэкенды подписи, что и выпускающие подкоманды, —
`--backend pkcs11|vault|file` (по умолчанию `pkcs11`, поэтому старая форма
команды без `--backend` работает как прежде):

```sh
# Аппаратный токен или HSM (по умолчанию)
issuer serve \
    --module /usr/lib/x86_64-linux-gnu/opensc-pkcs11.so \
    --key tessera-ca --algorithm ecdsa-p256 \
    --port 0

# Vault / OpenBao Transit (токен Vault — из VAULT_TOKEN)
issuer serve --backend vault \
    --vault-addr https://vault.example:8200 \
    --key tessera-ca

# Ключ в локальном файле (PKCS#8; пароль — pinentry
# или TESSERA_ISSUER_KEY_PASSPHRASE)
issuer serve --backend file --key-file ca-key.p8
```

Для `pkcs11` обязательны `--module` и `--key`; `--token-label` выбирает токен,
если их несколько; `--pinentry` задаёт программу pinentry явно. Для `vault`
обязательны `--vault-addr`, `--key` и `VAULT_TOKEN` в окружении. Для `file`
обязателен `--key-file`; требования к файлу ключа — в
[«Ключ в файле»](#ключ-в-файле). Гейты агента (loopback, парный токен, Origin,
подтверждение оператора) одинаковы для всех бэкендов. `--port 0` (по умолчанию)
берёт эфемерный порт — фактический адрес печатается при старте, его и открывают в
браузере. При раздаче кабинета (по умолчанию, либо `--cabinet-dir <dist>`)
`--allow-origin` не требуется: агент сам входит в свой allowlist. Для чистого
режима моста (`--no-cabinet` или когда встроенного кабинета нет, кабинет
раздаётся отдельной статикой) `--allow-origin` обязателен (хотя бы один;
повторяется).

### Модель доступа

Основной гейт — **парный токен сессии**. Агент генерирует случайный токен при
старте, печатает его в stdout и (при раздаче кабинета) вшивает в отданную
страницу; кабинет передаёт его в заголовке `X-Tessera-Session` каждого запроса,
сравнение — за константное время. Запрос без валидного токена отвергается **до**
обращения к модулю подписи.

**Origin** — вторичный барьер против cross-origin: если запрос несёт заголовок
`Origin`, он должен быть в allowlist (иначе отказ); отсутствие `Origin`
(легитимный same-origin GET, который браузеры не сопровождают этим заголовком)
само по себе не отказ — гейтит токен. Cross-origin страница не прочитает вшитый
токен и не поставит `X-Tessera-Session` без preflight, который агент
контролирует; DNS-rebind несёт Origin атакующего и отсекается allowlist.

**Маршрутизация:** `POST /sign`, `GET /info` и — при раздаче — статика кабинета
фиксированным набором ассетов; всё прочее отвечает 404, статика не затеняет
подписные маршруты.

PIN по HTTP **не передаётся** — поля для него в протоколе нет. Запрос на подпись
несёт только идентификатор ключа и base64 от TBS; посторонние поля (например,
`pin`) serde игнорирует, до бэкенда они не доходят.

### Подтверждение операции

Токен сессии аутентифицирует кабинет; **саму операцию авторизует оператор**.
Перед обращением к модулю подписи агент:

1. Разбирает TBS тем же разделяемым кодом (`tessera_ext`), что и enforcement
   Engine. TBS, который не читается как операция выпуска, отвергается **до**
   запроса подтверждения — что нельзя показать, то нельзя подписать.
2. Показывает оператору тип операции (лист смены / CA организации / CRL),
   субъект, срок, роли/привязки/рамки — и подписывает только после явного «да».

Агент здесь — **доверенный дисплей**: даже подменённый SPA или посторонний
локальный процесс с валидным токеном не проведёт подпись, которую оператор не
подтвердил (см. [threat-model.md §11.1.1, §11.3](threat-model.md)).

Канал подтверждения/PIN — **pinentry** (протокол Assuan, прецедент gpg-agent),
когда доступен, с fallback на **терминал**. Так доверенный канал не привязан к
способу запуска агента. Ответы `y`/`yes`/`д`/`да` принимаются независимо от
локали. Сбой самого канала pinentry (не отказ оператора, а невозможность
запуска/протокольная ошибка) → откат на терминал; отказ оператора честно
трактуется как отказ.

### Раздача кабинета

`issuer serve` по умолчанию отдаёт встроенный в бинарь кабинет (SPA) с того же
`127.0.0.1`-адреса, что и `/sign`. Каноничный путь: запустил агент → открыл
напечатанный localhost-адрес → выпускаешь, офлайн, из одного подписанного бинаря.
`--cabinet-dir <dist>` раздаёт кабинет из внешнего каталога `dist/` (для
self-hosted и разработки) и перекрывает встроенный. `--no-cabinet` отключает
раздачу — агент работает чистым мостом, кабинет тогда раздаётся отдельной
статикой. Приоритет: `--no-cabinet` → мост; иначе `--cabinet-dir` → внешний
каталог; иначе встроенный кабинет (если бинарь собран с фичей `embed-cabinet`);
иначе — мост (без ошибки).

Отдавая страницу, агент вшивает в неё парный токен сессии и метку ключа — кабинет
преднастраивает подключение сам, оператору не нужно переписывать адрес и токен, а
блок настроек агента сворачивается до индикатора «подключён/не подключён».
Целостность кабинета обеспечивает подпись бинаря: из подменённого агента ключи
всё равно недостижимы — приватная криптография за токеном (см.
[threat-model.md §11](threat-model.md)).

Агент запускается на время сессии выпуска (foreground): меньше окно атаки.
Системный сервис и демон-автостарт не предусмотрены — агент запускают явно, когда
садятся выпускать; выпуск и есть повод его поднять.

## Бэкенды подписи

Ядро не знает, где ключ: подпись готового TBS уходит за единый интерфейс,
никакой ключевой материал через него не проходит. Бэкенд выбирается `--backend`.

### PKCS#11 (токен и HSM)

Бэкенд по умолчанию, один код для аппаратных токенов и HSM. Флаги: `--module`
(путь к `.so`/`.dylib`/`.dll` — обязателен), `--token-label` (выбор токена,
если их несколько), `--key` (метка `CKA_LABEL` ключа CA), `--pinentry`
(программа pinentry явно).

PIN запрашивается через pinentry на время операции (`Secret` + `zeroize`,
не в логах и не в argv); при отсутствии pinentry — из `TESSERA_ISSUER_PIN`.

Для проб и CI подойдёт **SoftHSM** как программный PKCS#11-модуль. ГОСТ-токены
работают через тот же адаптер, если токен отдаёт нужный PKCS#11-механизм.

### Vault / OpenBao Transit

Подпись готового TBS через HTTP-API Transit. Флаги: `--vault-addr`
(например `https://vault.example:8200` — обязателен), `--mount` (mount Transit,
по умолчанию `transit`), `--vault-key` (имя ключа Transit; по умолчанию равно
`--key`), `--ca-bundle` (PEM-бандл доверенных CA вместо системного стора — для
приватных Vault-CA), `--prehashed` (слать локально вычисленный дайджест с
`prehashed=true` — для ключей, настроенных на pre-hashed вход).

Токен Vault читается из переменной окружения `VAULT_TOKEN` (пустой/незаданный →
отказ), передаётся в заголовке `X-Vault-Token` и не логируется. Для ECDSA
адаптер запрашивает `marshaling_algorithm=asn1` (Vault возвращает DER-подпись).

> **Только Transit, не Vault PKI.** Движок Vault PKI непригоден для Tessera:
> `encoding/asn1` в Go не разбирает OID-дуги больше `int64`, а наши расширения
> сидят в арке `2.25.<UUID>` — через Vault PKI такой сертификат не выпустить.
> Поэтому Transit подписывает уже собранный нами TBS, а не строит сертификат.

Transit не проверяет, **что** подписывает, — все проверки выпуска выполняются до
подписи в ядре, на клиенте; кто может звать `sign`, ограничивает политика Vault.

### Ключ в файле

Бэкенд `file` подписывает ключом CA из локального файла: `--key-file <path>`.
Формат — PKCS#8 (PEM или DER), включая зашифрованный (`ENCRYPTED PRIVATE
KEY`, PBES2); типы ключей — ECDSA P-256/P-384 и RSA. ГОСТ-ключи файловым
бэкендом не поддерживаются — для ГОСТ остаётся PKCS#11. Другие форматы
конвертируются штатно:

```sh
# новый зашифрованный ключ P-256
openssl genpkey -algorithm EC -pkeyopt ec_paramgen_curve:P-256 \
    | openssl pkcs8 -topk8 -v2 aes-256-cbc -out ca-key.p8
chmod 600 ca-key.p8

# конвертация существующего ключа (SEC1/PKCS#1 → PKCS#8)
openssl pkcs8 -topk8 -v2 aes-256-cbc -in old-key.pem -out ca-key.p8
```

Правила бэкенда:

- Файл ключа обязан быть недоступен группе и остальным (`chmod 600`) — иначе
  отказ до чтения содержимого. Владение файлом и права каталога бэкенд не
  проверяет — держите ключ в своём каталоге с правами `700`.
- Пароль зашифрованного ключа запрашивается через pinentry, при его отсутствии
  берётся из `TESSERA_ISSUER_KEY_PASSPHRASE`; в аргументы командной строки и
  логи пароль не попадает, память затирается.
- Незашифрованный ключ принимается, но с предупреждением при каждом старте;
  рекомендация — зашифрованный PKCS#8.
- Алгоритм подписи выводится из самого ключа; `--algorithm`, не совпадающий с
  ключом, — ошибка. `--key` опционален (по умолчанию — имя файла) и служит
  идентификатором ключа в `/info`, сводке подтверждения и журнале.

Ключ в файле — осознанный компромисс для стендов, CI и малых инсталляций:
при компрометации хоста он, в отличие от токена/HSM/Vault, извлекаем. Для
прода рекомендованы PKCS#11 или Vault Transit (см.
[threat-model.md §11](threat-model.md)).

## Журнал выпусков

Каждая операция (выпуск листа, CA, CRL) — запись в NDJSON-журнале, связанная в
hash-chain: монотонный `seq`, хэш предыдущей записи, фиксированный genesis.
Журнал **fail-closed**: запись делается **до** выдачи артефакта, и если журнал
недоступен, операция отклоняется (сертификат без записи не выпускается). Путь
задаётся флагом `--journal` каждой выпускающей подкоманды.

Голова цепочки периодически подписывается через тот же интерфейс подписи (по
завершении сессии и по команде). `issuer verify-journal` различает три
состояния:

- **цела, хвост полностью подписан** — всё в порядке;
- **цела, неподписанный хвост с seq N** — цепочка не нарушена, но записи с `N`
  ещё не покрыты подписью головы;
- **нарушена в позиции N** — разрыв/подмена/переупорядочивание на записи `N`
  (ненулевой код возврата).

Журнал **вторичен**: первичная правда — аудит входов на самих устройствах;
журнал служит инвентаризации выпуска и разбору инцидентов.

## Веб-кабинет

Кабинет — один статический артефакт (SPA + WASM-ядро) **без серверной логики**:
все проверки и сборка TBS — на клиенте, весь стейт — файлы (родительский
сертификат, выпущенные сертификаты, журнал, CRL, снапшот инвентаря). Он работает
self-hosted и офлайн; единственное сетевое обращение — к локальному агенту
`issuer serve` на `127.0.0.1`. Данные выпуска на внешние адреса не отправляются.

Кабинет один на все роли выпуска: доступные операции выводятся из предъявленного
родительского сертификата. Любой CA с рамками делегирования (корень или CA
организации) выпускает и подчинённые CA организаций с назначением суженных
рамок, и листы смен в своих рамках — операция выбирается переключателем;
лист/непригодный сертификат → операции недоступны с объяснением. Отдельных
сборок «по должности» нет.

Кабинет — часть подписанного бинаря `issuer`: `issuer serve` по умолчанию
раздаёт его с `127.0.0.1`, отдельного хостинга нет. Артефакт двуязычный,
переключатель языка в UI. Целостность кабинета обеспечивает подпись самого
бинаря — отдельный хэш-пиннинг не нужен; приватная криптография всегда за агентом
и токеном, из подменённого SPA ключи недостижимы (см.
[threat-model.md §11](threat-model.md)).

### Сборка и раздача статики

`cabinet/build.sh` собирает каталог `dist/` — `index.html`, `main.js`,
`styles.css`, WASM-бинарь и манифест `SHA256SUMS`. Релизный бинарь встраивает
этот каталог (раздаётся по умолчанию). Для закрытого контура тот же
`dist/` можно раздать любым статическим веб-сервером или указать агенту флагом
`--cabinet-dir <dist>`; серверной части у кабинета нет.

### Порядок работы

1. На машине с токеном запустите агент с кабинетом:
   `issuer serve --module <…> --key <…>` и откройте
   напечатанный им адрес `http://127.0.0.1:<порт>` (см. [«Агент
   `issuer serve`»](#агент-issuer-serve)). Агент преднастраивает
   подключение сам — блок агента в кабинете показывает лишь статус
   «подключён».
2. Предъявите родительский сертификат — доступные
   операции выводятся из него: CA с рамками делегирования (корень
   или CA организации) → выпуск подчинённых CA организаций с
   назначением суженных рамок либо листов смен в своих рамках,
   операция выбирается переключателем (по умолчанию: корень —
   CA организации, CA организации — лист смены); лист или
   непригодный сертификат → операции недоступны, с объяснением
   причины.
3. Инвентарь устройств: соберите его прямо в кабинете (конструктор —
   устройства, пользователи, роли, метки; результат — «ручной»
   снапшот, выгружается файлом) либо загрузите готовый снапшот-файл.
   Подписанный снапшот кабинет проверяет (битая подпись → отказ) и
   помечает источник («подписанный»/«ручной») с возрастом. Инвентарь
   питает подсказки форм выпуска: устройства и пользователи
   предлагаются из него (введённое значение сохраняется), а набор
   ролей листа сужается до пересечения рамок родителя с ролями
   инвентаря. Инвентарь необязателен — без него поля заполняются
   вручную.
4. Ключ листа: локальная генерация или загрузка CSR. Из CSR кабинет
   показывает subject и статус самоподписи и подставляет атрибуты
   с пометкой «запрошено в CSR»; при битой самоподписи выпуск
   недоступен.
5. Подписывает агент на токене после подтверждения оператором
   (см. [«Подтверждение операции»](#подтверждение-операции)) — PIN
   не покидает машину с агентом. Готовый сертификат скачивается
   файлом, операция фиксируется в журнале выпусков.

Интерфейс кабинета разделён на две вкладки: **«Выпуск»** (родитель,
операция, инвентарь, агент, сводка) и **«Журнал»** (загрузка, проверка
цепочки и выгрузка журнала выпусков). Блоки родительского
сертификата и агента подписи снабжены контекстными справками — как
получить родителя и как поднять/настроить `issuer serve`.

## Локализация

Операторские поверхности инструмента (сводка подтверждения в pinentry/терминале,
сообщения `issuer serve`, вывод CLI, SPA кабинета) локализованы на
русский и английский без i18n-фреймворка (компактная таблица строк).
Кабинет выбирает язык по языку браузера с переключателем в UI; для CLI и агента
локаль разрешается один раз при старте:

1. явная настройка — флаг `--lang` (`ru`/`en`);
2. переменная `TESSERA_ISSUER_LANG`;
3. переменная `LANG`;
4. fallback — английский.

Совпадение по префиксу языка, регистронезависимо: `ru_RU.UTF-8` и `RU` дают
русский, `en_GB` — английский; нераспознанное значение просто проваливается к
следующему источнику. Переводятся только подписи полей — технические данные
(субъект RFC 4514, OID, `role_id`, серийники, `crlNumber`, таймстемпы)
воспроизводятся байт-в-байт в любой локали.
