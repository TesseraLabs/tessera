# Design: windows-tcb-service

## Context

Канон Windows-адаптера (концепт 2026-06-07): два компонента — CP (UI, ноль
enforcement) и TCB-служба (Engine). Research-проход №2 верифицировал примитивы:

- `LsaLogonUser(..., LocalGroups)` — канонический путь групп per-session:
  SID'ы добавляются в токен поверх WORLD + logon-type; требует
  `SeTcbPrivilege` (вызов из TCB/LocalSystem). Группы временные, живут
  только в токене сессии.
- `CreateRestrictedToken` — deny-only SID, удаление привилегий; только
  сужает; создаёт новый токен (живые потоки не сужаются).
- `SetTokenInformation(TokenIntegrityLevel)` — MIC проверяется до DACL,
  no write-up. Опровергнуто: «ровно 4 уровня»; «SetTokenInformation мутирует
  произвольные атрибуты» — группы/привилегии только LocalGroups /
  `AdjustTokenPrivileges`.
- Локальный логон офлайн — ок: LSA валидирует по SAM без внешнего authority.
- pGina-паттерн (синхронизация SAM-членства) отвергнут: стейт переживает
  logoff.

Модель УЗ без изменений: общая техническая УЗ = один локальный SAM-аккаунт;
личность инженера — удостоверение, роль — payload среза.

## Goals / Non-Goals

**Goals:**

- Контракт Engine-слоя Windows: верификация тем же ядром, enforcement
  payload'а Windows-роли, TTL, реакция на извлечение носителя, журнал, IPC.
- Инварианты платформенно-нейтральные: fail-closed на auth-пути, «группы на
  сессию, не в ОС», фиксация payload на момент открытия сессии, hash-chain.
- Портируемость `tessera_core` (verify-путь) на Windows-таргет без
  дублирования логики.

**Non-Goals:**

- CP/UI и сериализация учётных данных — change `windows-cp`.
- Доменная интеграция (доменные группы в токене, доменная техническая УЗ) —
  отдельный research-проход (вне скоупа). При этом **сосуществование с
  доменом — в скоупе**: SAM-модель обязана работать на domain-joined
  устройстве без зависимостей от контроллера домена на пути входа; конфликты
  с доменными политиками детектируются `doctor`, а не проявляются молчаливым
  отказом.
- Метод «код» на Windows — после портирования сертификатного пути.
- Ретроактивное сужение живой сессии — ограничение платформы, принято
  концептом (права применяются при открытии сессии).
- MSI/подпись/дистрибуция — change `windows-cp` (собирает оба компонента).

## Decisions

### D1. Служба — отдельный Rust-crate, ядро — то же

Новый crate (условно `tessera_win_service`): SCM-интеграция, device
notifications, таймеры TTL, named pipe сервер, FFI-обвязка LSA/token API.
Вся верификация — вызовы `tessera_core` (как PAM-cdylib на Linux). Ядро
компилируется под Windows-таргет: verify-путь (`trust`, `challenge`,
`revocation`, `role`) платформенно-нейтрален; Linux-специфика (nix, PAM,
mount) отрезается feature-гейтами/`cfg`. Альтернатива — перепись ядра под
Windows — отвергнута: двойная логика доверия = двойные баги (тот же довод,
что за единое ядро доверия во всех сборках).

### D2. Точка инъекции групп: LsaLogonUser из TCB-контекста

Канон концепта: токен сессии строится `LsaLogonUser` с `LocalGroups` =
SID'ы групп роли (резолв имён из `payload.windows.groups` через
`LookupAccountName` на момент открытия сессии — снапшот, как «фиксация
payload в сессии»). Вызов требует `SeTcbPrivilege` — отсюда LocalSystem.

Точная стыковка с Winlogon (чей токен становится интерактивной сессией) —
главный имплементационный риск: канонический интерактивный логон зовёт
`LsaLogonUser` из LogonUI/Winlogon, а не из нашей службы. Кандидаты
(проверяются spike-задачей до основной волны):

1. CP сериализует учётные данные технической УЗ, а группы в токен добавляет
   наш компонент на стороне LSA (authentication package / subauth) — канон
   MS-модели, но код в lsass = максимальный класс риска.
2. Служба сама зовёт `LsaLogonUser(Interactive, LocalGroups)` по запросу CP
   и передаёт токен Winlogon через возврат CP (паттерн «MFA-вендоры зовут
   LsaLogonUser прямо» — research: рабочее, но неканоничное отклонение;
   в нашей архитектуре вызов остаётся в службе, не в CP).

Выбор фиксируется результатом spike (задача 1.x) и вносится в спеку до
реализации волны 2. Инвариант спеки не зависит от выбора: группы роли
попадают в токен сессии через `LocalGroups` и не материализуются в SAM.

### D3. Сужение: integrity + отзыв привилегий на токене до старта оболочки

`SetTokenInformation(TokenIntegrityLevel)` с RID из whitelist-маппинга
(`untrusted|low|medium|high` → `SECURITY_MANDATORY_*_RID` — константы
адаптера) + `AdjustTokenPrivileges` (отзыв из `privileges_remove`).
`CreateRestrictedToken` — резерв для deny-only SID (open question
windows-payload). Применение — к токену до создания процессов сессии;
неуспех любой операции = отказ входа (fail-closed, диагностика в журнал),
не молчаливое сужение.

### D4. TTL: таймер службы + персист реестра сессий

`max_ttl` роли → deadline при открытии; по истечении — принудительный logoff
сессии (WTS API). Реестр сессий (кто, роль, deadline, serial носителя)
персистится (ProgramData, ACL SYSTEM-only) и восстанавливается при рестарте
службы: рестарт не «обнуляет» TTL. Дедлайны в прошлом при восстановлении →
немедленный logoff. Слабее cgroup `runtime_max_sec` (нет ядерного слоя) —
честное отличие концепта; компенсация: LocalSystem, protected service,
recovery-политика SCM (автоперезапуск).

### D5. Извлечение носителя: device notification → grace → действие

`RegisterDeviceNotification` на service status handle
(`SERVICE_CONTROL_DEVICEEVENT`, интерфейс smartcard/volume) с
WMI-поллингом как fallback. Семантика — порт session-monitoring: сопоставление
serial → сессия, grace-период, действие lock | logoff из конфигурации,
suspend-поведение. Machinery портируется по смыслу, не по коду.

### D6. IPC: тот же NDJSON, транспорт — named pipe

`\\.\pipe\tessera-engine`: NDJSON-фреймы, `MAX_FRAME_BYTES`, Hello/строгая
версия — без изменений (`tessera_proto` получает транспортную абстракцию).
Дескриптор безопасности pipe: доступ SYSTEM (LogonUI/CP работает в
winlogon-контексте) — обычные сессии инженеров к pipe не допускаются.
Новые сообщения CP-флоу: `ListRoles` (роли для комбобокса; отдаёт служба —
CP не читает role-store сам, единственный источник — Engine),
`AuthenticateRequest`/`AuthenticateResult` (материал удостоверения + выбранная
роль → вердикт). Коды ошибок — существующая таблица + новые по необходимости.

### D7. Журнал и конфигурация

Журнал hash-chain — та же библиотека logging-audit; файл в
`%ProgramData%\Tessera\` с ACL SYSTEM-only (аналог root:root 0600).
Конфигурация — тот же формат config.toml (платформенные пути); fail-closed
семантика конфигурации сохраняется.

## Risks / Trade-offs

- [Стыковка токена с Winlogon не решена до spike (D2)] → spike — первая
  задача; оба кандидата верифицированы research'ем как существующие пути;
  инварианты спеки не зависят от выбора.
- [Код с SeTcbPrivilege = аналог root-процесса логина (T3/T6)] → тот же
  режим, что для PAM-cdylib: Rust-обвязка над unsafe FFI, panic-guard на
  всех входных точках, никакого парсинга недоверенных данных в
  привилегированном контексте сверх необходимого.
- [TTL без ядерного слоя: инженер с правами внутри сессии не убьёт службу,
  но может пытаться мешать logoff] → LocalSystem + SCM recovery; отказ
  logoff фиксируется в журнале как security-событие; остаточный риск принят
  концептом (§3 «честные отличия»).
- [Named pipe доступен коду в SYSTEM-контексте, не только LogonUI] →
  порог = уже-SYSTEM атакующий, который и так выше нашей модели;
  дополнительная проверка образа клиента — best-effort, не граница доверия.
- [Резолв имени группы → SID на каждом входе: имя может не существовать] →
  fail-closed: нерезолвящаяся группа роли = отказ входа с диагностикой
  (не молчаливый вход без группы).
- [На domain-joined устройстве системный резолв имён (`LookupAccountName`)
  может уходить к контроллеру домена — скрытая сетевая зависимость и
  недетерминированное поведение офлайн] → резолв в спеке ограничен
  локальными данными (SAM/локальный кэш); реализация обязана использовать
  локально-скоупированный резолв, а не полагаться на системный порядок
  поиска; доменное имя = отказ независимо от доступности DC.
- [Доменные политики могут сломать компоненты Tessera снаружи: исключение
  credential provider'ов, deny-логон технической УЗ, WDAC/AppLocker против
  наших DLL] → `doctor` детектирует эти конфликты с указанием
  политики-источника; E2E-прогон на domain-joined стенде — обязательная
  часть верификации.

## Migration Plan

Новая платформа — миграции нет. Развёртывание: установка службы (в составе
MSI из `windows-cp`), `prepare` создаёт техническую УЗ/каталоги/ACL, `doctor`
верифицирует готовность. Rollback = остановка и удаление службы — Linux-пути
не затронуты. Протокол: AF_UNIX-контракт без изменений; named pipe — новый
параллельный транспорт (совместимость по compat-policy).

## Open Questions

- Механика стыковки с Winlogon (D2, кандидаты 1/2) — решается spike'ом.
- Нужен ли deny-only SID через `CreateRestrictedToken` в первой волне или
  достаточно integrity + отзыва привилегий (связан с open question
  `windows-payload`).
- Минимальная поддерживаемая версия Windows (10/11, LTSC) — зафиксировать
  при старте реализации.
