# ipc-protocol — delta (windows-tcb-service)

## ADDED Requirements

### Requirement: Транспорт Windows — named pipe

На Windows сервер протокола ДОЛЖЕН (MUST) слушать named pipe
(`\\.\pipe\tessera-engine`; имя настраивается конфигурацией). Фрейминг
(NDJSON, `MAX_FRAME_BYTES`, bounded read), Hello со строгой версией
протокола и коды ошибок ДОЛЖНЫ (MUST) совпадать с AF_UNIX-контрактом —
меняется только транспорт. Pipe ДОЛЖЕН (MUST) создаваться с дескриптором
безопасности, ограничивающим доступ контекстом SYSTEM; обычные сессии
инженеров доступа к pipe иметь НЕ ДОЛЖНЫ (MUST NOT). AF_UNIX-контракт
этим требованием не изменяется.

#### Scenario: Клиент вне SYSTEM-контекста
- **WHEN** процесс сессии инженера пытается открыть pipe
- **THEN** доступ отклоняется дескриптором безопасности до какого-либо обмена кадрами

#### Scenario: Несовпадение версии на named pipe
- **WHEN** первый кадр `Hello` по pipe несёт неподдерживаемую версию протокола
- **THEN** сервер отвечает `Error{1000 PROTOCOL_MISMATCH}` и закрывает канал — семантика идентична AF_UNIX

### Requirement: Сообщения CP-флоу

Протокол ДОЛЖЕН (MUST) поддерживать сообщения флоу Credential Provider —
расширение существующего каталога сообщений, доступное только на
Windows-транспорте:
запрос перечня ролей (ответ — роли локальной базы `os = windows` с
идентификатором, именем и уровнем для UI) и запрос аутентификации
(материал удостоверения и выбранная роль → вердикт: допуск с параметрами
сессии либо отказ с кодом ошибки). Сообщения CP-флоу ДОЛЖНЫ (MUST) быть
доступны только на Windows-транспорте; на AF_UNIX они НЕ ДОЛЖНЫ (MUST NOT)
приниматься (`1100 BAD_REQUEST`).

#### Scenario: CP-сообщение на AF_UNIX
- **WHEN** клиент AF_UNIX-сокета шлёт сообщение CP-флоу
- **THEN** сервер отвечает `1100 BAD_REQUEST`

#### Scenario: Отказ аутентификации
- **WHEN** запрос аутентификации не проходит верификацию (цепочка/challenge/revocation/binding/роль)
- **THEN** ответ несёт отказ с кодом ошибки без деталей, достаточных для перебора; полная причина — в журнале службы
