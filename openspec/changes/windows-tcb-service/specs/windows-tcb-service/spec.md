# windows-tcb-service Specification

## Purpose

Engine-слой Tessera на Windows: служба LocalSystem с `SeTcbPrivilege`,
которая верифицирует удостоверение тем же ядром `tessera_core` и применяет
payload Windows-роли к сессии (группы в токене, integrity level, отзыв
привилегий, лимиты, TTL), ведёт hash-chain журнал и обслуживает Credential
Provider по IPC.

## ADDED Requirements

### Requirement: Служба LocalSystem с SeTcbPrivilege

Engine на Windows ДОЛЖЕН (MUST) работать как служба под LocalSystem с
`SeTcbPrivilege`, автозапуском и recovery-политикой SCM (автоперезапуск).
Сессия инженера НЕ ДОЛЖНА (MUST NOT) иметь возможности остановить,
перезапустить или переконфигурировать службу. Вся верификация и enforcement
ДОЛЖНЫ (MUST) выполняться в службе; вне службы (в CP/LogonUI) enforcement-логика
существовать НЕ ДОЛЖНА (MUST NOT).

#### Scenario: Попытка остановки службы из сессии
- **WHEN** процесс сессии инженера пытается остановить службу через SCM
- **THEN** операция отклоняется контролем доступа; служба продолжает работать

#### Scenario: Падение службы
- **WHEN** служба аварийно завершилась
- **THEN** SCM перезапускает её по recovery-политике; реестр сессий восстанавливается из персиста

### Requirement: Верификация удостоверения тем же ядром

Служба ДОЛЖНА (MUST) верифицировать удостоверение инженера тем же путём
`tessera_core`, что и на Linux: цепочка доверия, challenge-response
(proof-of-possession), revocation (CRL/OCSP, fail-closed), host-binding /
user-binding, покрытие выбранной роли удостоверением. Верификация ДОЛЖНА
(MUST) быть полностью офлайн-способной: решение принимается по локальным
данным устройства; внешний authority для входа НЕ ДОЛЖЕН (MUST NOT)
требоваться (LSA валидирует техническую УЗ по локальному SAM).

#### Scenario: Отозванное удостоверение
- **WHEN** удостоверение числится в актуальном CRL устройства
- **THEN** вход отклоняется fail-closed с audit-событием — как на Linux

#### Scenario: Вход без сети
- **WHEN** устройство не имеет сетевой связности, носитель и локальные данные валидны
- **THEN** вход выполняется полностью офлайн

#### Scenario: Domain-joined устройство при недоступном контроллере домена
- **WHEN** устройство включено в домен, контроллер домена недоступен, носитель и локальные данные валидны
- **THEN** вход Tessera выполняется полностью офлайн — членство в домене не добавляет зависимостей на путь входа

### Requirement: Группы роли только в токене сессии

Служба ДОЛЖНА (MUST) доставлять группы роли (`payload.windows.groups`) в
токен сессии через механизм `LsaLogonUser LocalGroups`: SID'ы добавляются в
токен на входе и существуют только в нём; после logoff следов членства
оставаться НЕ ДОЛЖНО (MUST NOT). Синхронизация членства локального
SAM-аккаунта под роль (pGina-паттерн) ЗАПРЕЩЕНА — НЕ ДОЛЖНА (MUST NOT)
применяться ни как основной, ни как fallback-путь. Имена групп ДОЛЖНЫ (MUST)
резолвиться в SID на момент открытия сессии (снапшот payload); резолв ДОЛЖЕН
(MUST) выполняться только по локальным данным устройства (SAM/локальный кэш) —
обращение к контроллеру домена на пути входа выполняться НЕ ДОЛЖНО (MUST NOT).
Нерезолвящаяся локально группа — отказ входа (fail-closed), не молчаливый
вход без группы.

#### Scenario: Группы роли в токене
- **WHEN** инженер вошёл с ролью, несущей `groups = ["TesseraOper"]`
- **THEN** токен сессии содержит SID группы; членство в SAM-аккаунте технической УЗ не изменилось

#### Scenario: Logoff не оставляет следов
- **WHEN** сессия инженера завершилась
- **THEN** ни SAM, ни персист устройства не содержат следов членства роли; следующий вход строит токен заново

#### Scenario: Группа роли не существует на устройстве
- **WHEN** имя группы из payload не резолвится в SID
- **THEN** вход отклоняется с диагностикой и audit-событием

#### Scenario: Доменное имя группы на domain-joined устройстве
- **WHEN** payload несёт имя группы, отсутствующее в локальном SAM (например, доменной группы), даже при доступном контроллере домена
- **THEN** вход отклоняется fail-closed с диагностикой — резолв через контроллер домена на пути входа не выполняется (доменные группы — вне скоупа SAM-модели)

### Requirement: Сужение токена — integrity level и отзыв привилегий

Служба ДОЛЖНА (MUST) применять к токену сессии до старта процессов сессии:
integrity level из `payload.windows.integrity_level` (маппинг whitelist →
`SECURITY_MANDATORY_*_RID`) и отзыв привилегий из
`payload.windows.privileges_remove`. Сужение применяется при открытии сессии;
ретроактивное сужение живых потоков НЕ гарантируется (принятое ограничение
платформы). Роль НЕ ДОЛЖНА (MUST NOT) приводить к добавлению привилегий.
Неуспех любой операции сужения — отказ входа (fail-closed) с диагностикой,
НЕ ДОЛЖЕН (MUST NOT) происходить молчаливый вход с несуженным токеном.

#### Scenario: Роль с пониженным integrity level
- **WHEN** роль несёт `integrity_level = "low"`
- **THEN** токен сессии получает low mandatory label до старта оболочки; запись в объекты выше уровня блокируется MIC (no write-up)

#### Scenario: Сбой применения сужения
- **WHEN** отзыв привилегии или установка integrity level завершились ошибкой
- **THEN** вход отклоняется с audit-событием; сессия с несуженным токеном не открывается

### Requirement: Жёсткий TTL — forced logoff

Служба ДОЛЖНА (MUST) назначать deadline сессии из `max_ttl_seconds` роли на момент
открытия и по его истечении принудительно завершать сессию (logoff).
Реестр активных сессий (личность, роль, deadline, идентификатор носителя)
ДОЛЖЕН (MUST) персиститься и восстанавливаться при рестарте службы; рестарт
НЕ ДОЛЖЕН (MUST NOT) сбрасывать или продлевать TTL. Отказ выполнения logoff
ДОЛЖЕН (MUST) фиксироваться в журнале как security-событие с повторными
попытками.

#### Scenario: Истечение TTL
- **WHEN** deadline сессии наступил
- **THEN** служба выполняет принудительный logoff сессии и пишет audit-событие

#### Scenario: Рестарт службы не сбрасывает TTL
- **WHEN** служба перезапущена при активной сессии, чей deadline уже в прошлом
- **THEN** после восстановления реестра сессия немедленно завершается

### Requirement: Реакция на извлечение носителя

Служба ДОЛЖНА (MUST) детектировать извлечение носителя сессии (device
notification; WMI-поллинг как fallback), сопоставлять устройство с сессией
и по истечении настраиваемого grace-периода применять настроенное действие
(`lock` | `logoff`) — семантический порт session-monitoring. Возврат носителя
в течение grace ДОЛЖЕН (MUST) отменять действие.

#### Scenario: Носитель извлечён и не возвращён
- **WHEN** носитель сессии извлечён и grace-период истёк
- **THEN** к сессии применяется настроенное действие (lock или logoff), audit-событие пишется

#### Scenario: Носитель возвращён в grace
- **WHEN** тот же носитель возвращён до истечения grace
- **THEN** действие отменяется; сессия продолжается

### Requirement: Лимиты сессии через Job Object

При наличии `payload.windows.limits` служба ДОЛЖНА (MUST) помещать процессы
сессии в Job Object с лимитами: `memory_max` (лимит памяти) и `process_max`
(лимит числа активных процессов). Неуспех применения лимитов — отказ входа
(fail-closed), симметрично сужению токена.

#### Scenario: Превышение process_max
- **WHEN** процессы сессии достигли `process_max`
- **THEN** создание нового процесса в сессии отклоняется ОС (лимит Job Object)

### Requirement: Журнал hash-chain

Служба ДОЛЖНА (MUST) вести журнал той же hash-chain механикой logging-audit
(та же Rust-библиотека): события входа/отказа/enforcement/TTL/извлечения
носителя. Файлы журнала ДОЛЖНЫ (MUST) лежать в каталоге с ACL только для
SYSTEM (аналог root:root 0600); секреты в журнал попадать НЕ ДОЛЖНЫ (MUST NOT).

#### Scenario: Непрерывность цепочки
- **WHEN** журнал верифицируется после серии входов и отказов
- **THEN** hash-chain непрерывен; вмешательство в прошлые записи детектируется

### Requirement: IPC-сервер для Credential Provider

Служба ДОЛЖНА (MUST) обслуживать CP через named pipe (NDJSON-фрейминг и
строгая версия протокола — по ipc-protocol). Служба — единственный источник
перечня ролей для UI (CP не читает role-store). Сообщения CP-флоу:
запрос перечня ролей; запрос аутентификации (материал удостоверения +
выбранная роль) → вердикт. Любая ошибка канала или протокола на пути входа
ДОЛЖНА (MUST) приводить к отказу входа (fail-closed).

#### Scenario: Перечень ролей для комбобокса
- **WHEN** CP запрашивает роли до аутентификации
- **THEN** служба отдаёт роли из локальной базы (os = windows) с именами и уровнями для отображения

#### Scenario: Служба недоступна
- **WHEN** pipe не отвечает или закрыт при попытке входа
- **THEN** вход методом Tessera невозможен (fail-closed); стандартные методы Windows не затрагиваются

### Requirement: Диагностика устройства — doctor и prepare

CLI ДОЛЖЕН (MUST) предоставлять на Windows команды подготовки и диагностики
устройства: `prepare` (техническая УЗ, каталоги и ACL, регистрация службы)
и `doctor` (проверка готовности: служба, привилегии, ACL, база ролей,
доверие) — порт существующих команд по смыслу.

На устройстве, включённом в домен, `doctor` ДОЛЖЕН (MUST) дополнительно
проверять сосуществование с доменными политиками и диагностировать
конфликты: политика исключения credential provider'ов (CLSID Tessera CP в
списке исключённых), логон-ограничения технической УЗ (deny-политики
интерактивного входа), политики ограничения исполнения (WDAC/AppLocker),
блокирующие модули Tessera. Обнаруженный конфликт ДОЛЖЕН (MUST) выводиться
как непройденная проверка с указанием источника (доменная политика), а не
проявляться молчаливым исчезновением тайла или отказом входа без диагностики.

#### Scenario: doctor на неподготовленном устройстве
- **WHEN** `doctor` запущен до `prepare`
- **THEN** выводится перечень непройденных проверок с диагностикой; код возврата ненулевой

#### Scenario: Доменная политика исключила CP
- **WHEN** `doctor` запущен на domain-joined устройстве, где GPO исключает CLSID Tessera CP
- **THEN** проверка сосуществования не проходит с диагностикой «провайдер исключён политикой»; код возврата ненулевой

#### Scenario: Логон-запрет технической УЗ доменной политикой
- **WHEN** доменная политика запрещает технической УЗ интерактивный вход
- **THEN** `doctor` сообщает о конфликте с указанием политики-источника
