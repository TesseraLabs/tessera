# Design: role-format

## Context

Tessera v0.3.19: права сессии = `min(потолок МКЦ из серта, МНКЦ пользователя)`; именованного
объекта прав нет. Платформенная спека (tessera-ws) ввела роль как канонический объект Engine
и зафиксировала: явный выбор роли на логине без дефолта; роль ≠ учётка (одна учётка инженера,
роль — сессионный параметр); серт несёт ссылки (`role_id`), не копию прав; код Codes несёт
`role_id` во входе MAC. Design-сессия 2026-06-05 (см. research
`tessera-ws/specs/research/2026-06-05-session-privilege-levels.md`, проходы №3–5) дала формат.

Ограничения: офлайн-устройство (никаких онлайн-проверок); open-core — Engine+Login обязаны
работать без Control; PAM-стек Astra (порядок модулей — грабли pam_parsec_mac/pam_systemd);
`tessera_core` — sync, без tokio.

## Goals / Non-Goals

**Goals:**
- Декларативный формат роли (TOML-срез per-OS) и база ролей на устройстве.
- Два режима доверия: standalone (права ФС) / managed (манифест Control, anti-rollback).
- Явный выбор роли на логине: суффикс `user+role` + текстовый PAM-prompt; без дефолта.
- Покрытие: серт — membership по списку `role_id` из расширения; код — `role_id` в MAC (контракт на будущее, Codes ещё нет).
- Несравнимость ролей: никакой арифметики уровней в ядре.

**Non-Goals:**
- Доставка ролей (канал Control → устройство) — здесь только формат манифеста как контракт.
- Нативные UI выбора (GDM extension, fly-dm kgreeterplugin, CPFT_COMBOBOX) — опциональный полиш, отдельный change.
- Отзыв живой сессии, ротация ключей — research-проход №6.
- Linux/Windows payload-адаптеры — схема фиксирует секции, реализация astra-first.

## Decisions

1. **TOML, один файл = один срез роли** (`/var/lib/tessera/roles/<role>.toml`).
   Альтернативы: JSON (нет комментариев — отпадает для рукописного standalone-конфига),
   единый бандл CBOR (не инспектируем на офлайн-устройстве), SQLite (overkill).
   Подпись managed-режима — над сырыми байтами файла → нет проблемы каноникализации.
2. **Срез только своей ОС.** Поле `os` в файле; чужой `os` = ошибка валидации.
   Канон полного объекта (все ОС) живёт в Control; Windows-конфиг на линуксе не хранится.
3. **Роли несравнимы; membership-проверки.** `level` — атрибут отображения (порядок UI,
   соответствие уровню МКЦ на Astra), не иерархия. «Админ покрывает оператора» выражается
   списком в серте (`[oper, admin]`), не сравнением.
4. **role_id**: `[a-z][a-z0-9-]{0,15}` — suffix-safe (`ivanov+serv`), имя файла, вход MAC.
   Неизменяем; переименование = новая роль.
5. **Два режима доверия.** Standalone: каталог root:root 0755/0644, доверие = ФС (модель
   sudoers.d), манифест отсутствует. Managed (устройство enrolled): `manifest.toml`
   (`bundle_version` монотонный, `os`, `role → {version, sha256}`, подпись Control),
   fail-closed при любой невалидности, последний `bundle_version` персистится (anti-rollback,
   урок TUF: подписи срезов недостаточно против mix-and-match). Локальные правки в managed
   не действуют (hash не сойдётся) — by design.
6. **Выбор роли — до/вместе с аутентификацией, атомарно.** Суффикс парсится в начале
   `pam_sm_authenticate`; PAM_USER переписывается на каноническое имя (`pam_set_item`) ДО
   остальной работы; `+` запрещён в канонических именах. Резолв роли и валидация удостоверения —
   одна неразрывная стадия (урок polkit CVE-2021-3560), запрошенная роль входит в аудит-запись.
7. **Сосуществование с max_integrity.** Расширение `max_integrity` остаётся потолком МКЦ:
   эффективная метка = `mac_mask` роли, ограниченная потолком серта и МНКЦ (пересечение).
   Роль без покрытия маски потолком → отказ (а не молчаливое сужение) — явность важнее удобства.
8. **Сессия фиксирует копию payload** на момент открытия: смена/удаление роли не влияет на
   живую сессию (доживает TTL); `TTL = min(TTL удостоверения, session.max_ttl, глобальный дефолт)` —
   бессрочных сессий нет.
9. **Строгий парсинг**: неизвестные поля = ошибка (serde `deny_unknown_fields`); урок PwnKit —
   привилегированный код не делает молчаливых допущений о входе.

## Распределение open/commercial (утверждено 2026-06-05)

Принцип: **форматы и верификация — открытые; генерация, доставка и управление — платные.**

| Кусок | Лицензия |
|---|---|
| Схема TOML-среза, парсер, валидация, `role lint/list` | Открытая |
| Standalone-режим (доверие = ФС) | Открытая |
| Формат манифеста + верификация на устройстве (подпись, anti-rollback) | Открытая |
| Генерация/подпись/доставка манифеста, fleet-управление ролями | Коммерческая (Control) |
| `role_id` во входе MAC кода (формат — открытый контракт) | Коммерческая (Codes) |
| Расширение серта allowed-roles: OID, парсер, membership | Открытая |
| payload `linux` базовый: группы, sudo-роль, systemd-лимиты | Открытая |
| payload `astra` (МКЦ): схема + SPI + StubBackend | Открытая (как сейчас) |
| payload `astra`: ParsecBackend enforcement | Коммерческая (как сейчас) |
| payload `selinux`: схема полей + SPI | Открытая |
| SELinux-адаптер (env_params-пайплайн, резолв seusers→contexts) | Коммерческая |
| Нативный UI fly-dm (kgreeterplugin) | Коммерческая |
| Нативный UI GDM (choice-list extension) | Открытая (когда дойдём) |

Паттерн единый с mac-integrity: схема и SPI открыты, платформо-специфичный enforcement —
коммерческое расширение; базовый Linux-payload целиком открыт.

## Модель угроз managed-режима (решено 2026-06-05)

Канон — `tessera-ws/specs/threat-model.md` (§5.1); здесь сводка. Манифест + anti-rollback
защищают от: скомпрометированного канала доставки (полностью) и подлога файлов не-root
атакующим. Root устройства и полный офлайн-доступ к диску — вне модели (слой целостности
среды). Персист `bundle_version` — root-файл `/var/lib/tessera/bundle.version`; при потере
(новый диск) первый валидный манифест принимается как baseline (TOFU-даунгрейд осознанно,
audit `bundle_baseline_established`).

## Risks / Trade-offs

- [Суффикс ломает PAM-стеки, читающие PAM_USER до нас] → наш модуль первым в стеке (уже
  контролируем снипеты pam.d, спека pam-integration); таблица edge cases в тестах с первого дня.
- [Managed fail-closed: битый манифест = вход по кодам невозможен] → осознанный выбор
  (безопасность > доступность); серт-путь с физическим носителем как аварийный канал;
  аудит critical-событием. «Аварийную локальную роль» не делаем — дыра.
- [Расширение серта со списком ролей = перевыпуск при изменении НАБОРА ролей инженера] →
  права внутри роли меняются доставкой базы без перевыпуска; меняется только членство.
  Серты смены короткоживущие — ротация и так частая.
- [Несравнимость непривычна («у меня уровень 5, почему нельзя 3?»)] → UX-вопрос текста
  ошибки + выдача серта со списком; семантика честная (МКЦ-прецедент).
- [Два режима доверия = две ветки кода] → общая валидация схемы; режим — только источник
  доверия (одна точка ветвления при загрузке базы).

## Migration Plan

v0.3.x → role-format: (1) `role-store` + `role lint/list` без влияния на вход; (2) расширение
серта + покрытие в flow за feature-флагом конфига (`roles.enforce = false|warn|require`);
(3) `require` по умолчанию в следующем миноре. Откат: флаг в `false` — поведение v0.3.19.

## Open Questions

- Алгоритм подписи манифеста (Ed25519 vs ГОСТ для серт-ориентированных заказчиков) — решить
  при имплементации managed-режима; формат поля подписи закладывается расширяемым.
- Формат OID-расширения allowed-roles (SEQUENCE OF UTF8String vs одна строка с разделителем) —
  в дельта-спеке cert-scope-binding предложен SEQUENCE OF; подтвердить при имплементации.
- Текстовый prompt: показывать ли список доступных ролей (утечка имён ролей до аутентификации?)
  или только поле ввода — склоняемся к полю ввода без списка; решить в role-selection.
