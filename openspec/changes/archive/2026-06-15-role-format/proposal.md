# Proposal: role-format

## Why

Сегодня права сессии выводятся из пары «серт (потолок МКЦ) × МНКЦ пользователя» — нет именованного,
администрируемого объекта прав: нечего выбрать на логине, нечего доставлять с Control, не на что
ссылаться коду Tessera Codes. Платформенная спека (tessera-ws, 2026-06-05) зафиксировала канонический
объект **«роль»** и решение «явный выбор роли на логине, без дефолта»; design-сессия 2026-06-05
определила формат. Пора положить это в спеки tessera как контракт реализации.

## What Changes

- Вводится **роль** (role) — именованный OS-агностичный набор разрешений, активируемый на сессию;
  роли несравнимы (membership-проверки, без иерархии «выше/ниже»). Терминология: «профиль прав» →
  «роль» везде (глоссарий tessera-ws уже обновлён); в коде `role`, не `profile`.
- Новая **база ролей на устройстве**: `/var/lib/tessera/roles/*.toml` — срез только своей ОС
  (схема: `role/version/os/name/level/description/[payload]/[session]`); строгий парсинг
  (неизвестные поля = ошибка).
- **Два режима доверия**: standalone (open-core, доверие = права ФС, как `/etc/sudoers.d`) и
  managed (подписанный Control манифест: `bundle_version` + hash срезов, anti-rollback, fail-closed).
- **Выбор роли на логине**: обязательный и явный — суффикс учётки `ivanov+serv` (работает в любом
  DM и `ssh`) или текстовый PAM-prompt; без выбора вход не продолжается. PAM_USER канонизируется
  до остальных модулей стека.
- **BREAKING (схема серта):** leaf-серт получает расширение «список разрешённых `role_id`»;
  покрытие = «запрошенная роль ∈ списка». Существующее расширение `max_integrity` остаётся
  потолком МКЦ (эффективная метка по-прежнему ограничена сертом).
- Сессия фиксирует копию payload роли на момент открытия; TTL сессии = min(TTL удостоверения,
  `session.max_ttl` роли, глобальный дефолт).

## Capabilities

### New Capabilities

- `role-store`: формат TOML-среза роли, каталог базы на устройстве, валидация, два режима доверия
  (standalone/managed), манифест и anti-rollback, атомарное обновление набора.
- `role-selection`: выбор роли на логине (суффикс/prompt, без дефолта), канонизация PAM_USER,
  резолв роли из базы, проверка покрытия удостоверением, передача payload адаптеру, аудит-поля.

### Modified Capabilities

- `cert-scope-binding`: новое custom OID-расширение «allowed roles» (список `role_id`) в leaf-серте;
  правила сосуществования с `max_integrity` (потолок МКЦ применяется к метке, роли — к набору прав).
- `mac-integrity`: источник запрошенной метки сессии — `payload.mac_mask` выбранной роли;
  эффективная метка = пересечение с потолком серта и МНКЦ пользователя (вместо «метка из min(cert, МНКЦ)» без роли).
- `logging-audit`: словарь audit-событий ролей (target `role.audit`).
- `configuration`: секция `[roles]` (`enforce`/`dir`/`default_session_ttl`).
- `ipc-protocol`: опциональные поля `role`/`role_version` в сообщениях сессии.

## Impact

- `tessera_core`: новый модуль `role/` (схема, парсер, store, manifest); `x509/` — новое расширение.
- `pam_tessera`: парсинг суффикса + prompt в `pam_sm_authenticate`, передача роли во flow; `session.rs` — payload роли.
- `tessera_cli`: `role lint/list`, доставка managed-набора (заготовка под Control).
- `tessera_proto`: роль в IPC-сообщениях (session open).
- Документация/спеки tessera-ws — уже обновлены (глоссарий, платформенная спека §7).
- Codes/Control (закрытые, будущие): `role_id` во входе MAC кода; формат доставки манифеста — фиксируются здесь как контракт.
