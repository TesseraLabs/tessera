# Design: issuer-file-backend-and-serve-backends

## Context

Подпись выпуска идёт через трейт `SignatureBackend` (`sign.rs`): `algorithm(key_id)`
+ `sign(tbs_der, key_id)`, наружу — только подпись и алгоритм. Адаптеры: PKCS#11
(`pkcs11.rs`, фича `pkcs11`) и Vault Transit (`vault.rs`, фича `vault`). Агент
`Agent<B: SignatureBackend, C: Confirmer>` (`serve.rs`) уже генерик, но
`run_serve` (`cli.rs`) жёстко строит `Pkcs11Signer`; `ServeArgs` не имеет ни
`--backend`, ни vault-флагов. Канал подтверждения оператора (`confirm.rs`,
pinentry/терминал) от бэкенда не зависит. В зависимостях ядра уже есть `p256`,
`rsa`, `sha2` (проверка CSR), клиппи-политика запрещает `unwrap`/`panic`,
секреты — `secrecy` + `zeroize`.

Дизайн согласован с владельцем 2026-07-17 (брейншторм в сессии): файловый
бэкенд — прод-статус с допуском незашифрованного ключа; serve — все три
бэкенда; реализация RustCrypto-нативная; выбор бэкенда в serve — явным флагом.

## Goals / Non-Goals

**Goals:**

- Явный файловый бэкенд подписи (PKCS#8, включая EncryptedPrivateKeyInfo) для
  CLI-подкоманд и `issuer serve`.
- Vault Transit и файловый бэкенд в `issuer serve` — единая модель флагов с
  CLI-подкомандами, дефолт `pkcs11` без breaking-изменений.
- Полный e2e CLI-тестовый путь без внешних зависимостей (файловый бэкенд не
  требует SoftHSM/Vault).

**Non-Goals:**

- Генерация ключей (`issuer keygen`) — ключ создаётся `openssl genpkey`/
  `openssl pkcs8 -topk8`; команды приводятся в документации.
- Ed25519 (не экспонирован в `--algorithm`) и ГОСТ (нет в RustCrypto; остаётся
  за PKCS#11-адаптером).
- Форматы ключа помимо PKCS#8 (SEC1 «EC PRIVATE KEY», PKCS#1 «RSA PRIVATE
  KEY») — документируется конвертация в PKCS#8.
- Изменения кабинета (SPA) и протокола агента — кабинет не различает бэкенды.
- Ограничение глубины делегирования (отдельная тема, вне этого change).

## Decisions

### D1. RustCrypto-нативный `FileSigner` за фичей `file`

Новый модуль `file.rs` реализует `SignatureBackend` поверх
`p256::ecdsa::SigningKey` / `p384::ecdsa::SigningKey` /
`rsa::pkcs1v15::SigningKey<Sha256>`. Парсинг и расшифровка — крейт `pkcs8`
(фичи `encryption`, `pem`): PBES2 (AES-CBC + PBKDF2/scrypt). Новые зависимости
только `p384` и прямой `pkcs8` — то же RustCrypto-семейство, что уже в
workspace.

Альтернативы отвергнуты: обёртка над SoftHSM (тяжёлая runtime-зависимость,
мутирует стейт токена), внешний `openssl` (ключ через субпроцесс,
неконтролируемый zeroize).

### D2. Алгоритм выводится из ключа; `--algorithm` — кросс-проверка

Файл однозначно определяет тип ключа (кривая EC / RSA), поэтому
`FileSigner::algorithm()` возвращает выведенный алгоритм. Если оператор передал
`--algorithm`, не совпадающий с ключом, — типизированная ошибка (не молчаливое
переопределение). Это устраняет класс ошибок «подписали не тем алгоритмом, что
заявлен в TBS».

### D3. `--key` опционален для file-бэкенда; дефолт — basename файла

`KeyId` нужен агенту (`/info`, сводка подтверждения) и журналу операций, но у
файла нет пространства имён ключей. `--key` остаётся способом задать
человекочитаемый идентификатор; по умолчанию берётся имя файла без расширения.
`algorithm()`/`sign()` сверяют переданный `KeyId` с ожидаемым и отвечают
`UnknownKey` на чужой — та же семантика, что у `MockSigner`.

### D4. Права файла: отказ при `mode & 0o077 != 0`, плейн — с предупреждением

Ключ (и зашифрованный, и плейн) обязан быть недоступен группе/остальным —
прецедент ssh. Незашифрованный ключ принимается, но с локализованным
предупреждением в stderr (решение владельца: прод-статус, но плейн допустим —
удобство стендов/CI перевешивает). На Windows (кроссплатформенность стороны
выпуска) Unix-права недоступны — проверка выполняется только на Unix,
предупреждение о плейн-ключе остаётся везде.

### D5. Пароль ключа: pinentry → `TESSERA_ISSUER_KEY_PASSPHRASE`

Симметрия с PIN PKCS#11 (`--pinentry`, затем `TESSERA_ISSUER_PIN`): пароль
запрашивается через pinentry на время старта, fallback — переменная окружения.
`SecretString` + `zeroize`; расшифрованный DER-буфер зануляется сразу после
парсинга ключа. Ключ читается и расшифровывается один раз при построении
бэкенда (fail-fast на битом файле/пароле), не на каждую подпись: агент —
foreground-процесс на время сессии выпуска, держать `SigningKey` в памяти
процесса — неизбежное свойство файлового бэкенда, и это честно фиксируется в
threat-model.

### D6. `issuer serve --backend pkcs11|vault|file`, дефолт `pkcs11`

`ServeArgs` получает `--backend` и флаги Vault/file, единые по именам с
`BackendArgs` CLI-подкоманд. `run_serve` диспатчит построение бэкенда и передаёт
его в генерик `serve()`; `Agent`, гейты (токен, Origin), `Confirmer` — без
изменений. Требования флагов: `pkcs11` → `--module` + `--key`; `vault` →
`--vault-addr` + `--key` + `VAULT_TOKEN`; `file` → `--key-file`. Альтернатива
«выводить бэкенд из состава флагов» отвергнута: неоднозначно при комбинациях и
расходится с CLI.

Фичи Cargo: `file` (native-only), `serve` не тянет `file`/`vault` автоматически
— составом бинаря управляет релизный профиль (release-бинарь собирается со
всеми тремя).

### D7. Ошибки и локализация

Типизированные варианты ошибок файлового бэкенда: `NotFound`, `Permissions`
(с фактическим mode), `Malformed`, `WrongPassphrase`, `UnsupportedKeyType`,
`AlgorithmMismatch { key, requested }` → маппинг в `CliError::Backend`;
все операторские сообщения — через `l10n.rs` (RU/EN), включая предупреждение
о незашифрованном ключе.

## Risks / Trade-offs

- [Ключ извлекаем при компрометации хоста — ниже классом, чем токен/HSM/Vault]
  → честный абзац в threat-model §11 и в докe бэкенда: прод-рекомендация —
  PKCS#11/Vault; файловый бэкенд — для принятого риска (стенд, CI, малая
  инсталляция). Плейн-ключ дополнительно помечается предупреждением при каждом
  старте.
- [Новая крипто-поверхность: парсинг/расшифровка PKCS#8 в нашем процессе] →
  только аудитированные RustCrypto-крейты, никакого самописного ASN.1;
  негативные тесты (битый файл, неверный пароль, чужой тип ключа).
- [Расхождение `--algorithm` и ключа] → D2: явная ошибка, не переопределение.
- [Регресс текущих вызовов `issuer serve` без `--backend`] → дефолт `pkcs11`,
  контракт-тест на старую форму команды.
- [Windows: нет Unix-прав] → проверка прав платформо-зависимая (Unix), плейн-
  предупреждение — везде; зафиксировано в спеке кроссплатформенности.

## Migration Plan

Флаги аддитивны, дефолты сохранены — миграция не требуется. Откат = не
использовать новые флаги. Релизный бинарь включает фичу `file` со следующего
минорного релиза.

Нет — ключевые решения зафиксированы владельцем в брейншторме 2026-07-17.
