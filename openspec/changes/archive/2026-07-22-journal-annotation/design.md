# Design: journal-annotation

## Context

Журнал выпуска (`crates/tessera_issuer/src/journal.rs`) — hash-chained
append-only NDJSON: каждая строка несёт `seq`, `prev_hash`, `ts` и payload,
выбранный полем `op` (внутренне-тегированное перечисление `Payload`,
`#[serde(tag = "op")]`, встроенное в `Entry` через `#[serde(flatten)]`). Ядро
байт- и строкоориентированное, без часов и без своей персистентности
(`JournalStorage`), поэтому собирается под `wasm32`. Нужна нейтральная запись
для побочного контекста, не расширяющая ядро под каждый случай.

## Goals / Non-Goals

**Goals:**
- Один нейтральный op `annotation` (`kind` + `data`), непрозрачный для ядра.
- Полноправное участие в hash-chain: тамперинг рвёт цепочку на позиции.
- Структурная верификация без знания `kind`.
- Байтовая неизменность прежних op'ов (голдены).
- wasm-совместимость (без новых зависимостей: `serde_json` уже в ядре).

**Non-Goals:**
- Реестр или валидация `kind` в ядре — смысл принадлежит пишущему.
- Индексация/запрос аннотаций, схемы `data`, миграции.
- Какая-либо интерпретация `data` ядром.

## Decisions

1. **`data: serde_json::Value`, не сырая строка.** `serde_json` — базовая
   зависимость крейта (пути журнала и CSR), так что `Value` доступен без новой
   feature и на wasm. `Value` даёт структурную гарантию «валидный JSON-объект»
   бесплатно (в отличие от сырой строки, которую пришлось бы валидировать
   отдельно). Цена — `Entry`/`Payload` теряют derive `Eq`: `serde_json::Value`
   реализует только `PartialEq` (из-за `f64` в `Number`). Оба типа приватны,
   `Eq` нигде не требовался; `PartialEq` сохранён.
2. **Пустой `kind` отвергается дважды.** `append_annotation` возвращает
   `JournalError::EmptyAnnotationKind` до записи (fail-closed, состояние цепочки
   не трогается). `verify_lines` дополнительно отвергает пустой `kind`
   структурно — на случай строки, собранной в обход append'а. `kind` — тег
   пишущего; пустой не идентифицирует автора и бессмыслен.
3. **Аннотация — обычная запись цепочки.** В `verify_lines` она попадает в
   ветку неподписанного хвоста наравне с выпусками; отдельно обрабатывается
   только пустой `kind` (разрыв) и `head_signature` (закрытие хвоста). Хэш и
   `prev_hash` считаются как для любой строки.
4. **Вариант добавлен в конец `Payload`.** Существующие варианты и порядок их
   полей не тронуты, поэтому их NDJSON не меняется байт-в-байт. Зафиксировано
   голден-тестом прежнего op (`issue_leaf`) и голден-тестом самой аннотации.

## Совместимость верификаторов (compat-note)

Проверено фактически: `Payload` — `#[serde(tag = "op")]` без `#[serde(other)]`.
Верификатор прежней версии, встретив `op":"annotation"`, падает на разборе с
ошибкой `unknown variant \`annotation\`` (подтверждено прогоном изолированного
перечисления-двойника: `is_err = true`). В `verify_lines` этот путь уже ведёт к
`JournalStatus::Broken` на позиции строки (ветка `let Ok(entry) = … else {
return broken(...) }`). То есть старый верификатор НЕ пропускает аннотацию молча
— он сообщает о разрыве на её позиции. Это осознанный компромисс: строгий разбор
даёт fail-closed на неизвестном формате, ценой того, что аннотации, добавленные
новым кодом, «ломают» верификацию старым. Аннотации предназначены для журналов,
проверяемых кодом той же или более новой версии.

## Risks / Trade-offs

- **Старый верификатор видит разрыв на аннотации.** Это следствие строгого
  разбора (fail-closed), а не тихая деградация. Приемлемо: журнал проверяется
  своей версией инструмента; альтернатива (`#[serde(other)]`-пропуск) ослабила бы
  контроль формата для всех op'ов.
- **`data` неограничен по размеру/содержимому.** Ядро его не интерпретирует;
  дисциплина размера и запрет секретов — на стороне пишущего (как и для прочих
  записей: секреты MUST NOT попадать в журнал).
- **Потеря `Eq` на приватных `Entry`/`Payload`.** Влияния нет (типы приватны,
  `Eq` не использовался); `PartialEq` достаточен для тестов и сравнения строк.

## Migration Plan

Чистое добавление: новый вариант в конце enum, новый публичный метод, новый
вариант ошибки в `#[non_exhaustive]` перечислении. Существующие пути выпуска и
их сериализация не меняются; отката для формата не требуется, так как прежние
строки идентичны.
