# issuer-signing Specification

## Purpose

Сторона подписи выпуска: единый интерфейс «подписать байты TBS ключом слота»
(ключ не покидает токен/HSM/Vault), адаптеры PKCS#11 и Vault Transit, локальный
агент `issuer serve` (мост «браузер ↔ токен» на 127.0.0.1 с раздачей самого
кабинета), обязательное подтверждение операции оператором на доверенном канале,
кроссплатформенность стороны выпуска и локализация операторских поверхностей.

## MODIFIED Requirements

### Requirement: Локальный агент issuer serve

Агент `issuer serve` ДОЛЖЕН (MUST) слушать только 127.0.0.1. Основной гейт
запросов подписи/информации — **парный токен сессии**, выданный при старте
агента: запрос без валидного токена ДОЛЖЕН (MUST) отвергаться. Origin — вторичный
гейт против cross-origin: если запрос НЕСЁТ заголовок Origin, он ДОЛЖЕН (MUST)
входить в allowlist, иначе запрос отвергается; отсутствие Origin (легитимный
случай same-origin GET, который браузеры не сопровождают Origin) само по себе
НЕ отвергает запрос — его гейтит токен. Такое разделение безопасно: cross-origin
страница не может ни прочитать вшитый токен (same-origin policy), ни поставить
заголовок токена на cross-origin запрос без preflight, который агент
контролирует; DNS-rebind несёт Origin атакующего и отсекается allowlist. PIN
ДОЛЖЕН (MUST) вводиться на стороне агента; кабинет НЕ ДОЛЖЕН (MUST NOT) принимать,
хранить или передавать PIN.

Раздача статики кабинета (SPA) на том же 127.0.0.1-origin, что и `/sign`, —
поведение агента **по умолчанию**, когда кабинет доступен (встроен в бинарь
фичей сборки либо указан каталогом). Отдельный флаг отключает раздачу (чистый
режим моста — для случая, когда SPA раздаётся снаружи); если встроенного кабинета
нет и каталог не указан, агент падает в режим моста. Когда агент раздаёт кабинет:
(1) его собственный origin автоматически входит в allowlist (явный
`--allow-origin` не требуется); (2) агент вшивает парный токен сессии в отданную
страницу, чтобы оператор не переписывал адрес и токен вручную. Раздача статики
НЕ ДОЛЖНА (MUST NOT) ослаблять гейты `/sign`: парный токен и подтверждение
оператора остаются обязательными для подписи. Маршруты статики НЕ ДОЛЖНЫ
(MUST NOT) отдавать ничего за пределами набора ассетов кабинета (защита от path
traversal).

Токен, вшитый в отданную страницу, доступен по `GET /` до какого-либо гейта (иначе
браузер не получил бы его для первого запроса) — значит в режиме раздачи парный
токен НЕ является секретом против других локальных процессов на том же хосте,
дотягивающихся до loopback-порта. Анти-локальный-процесс-инвариант в этом режиме
держит НЕ секретность токена, а обязательное подтверждение оператора: локальный
сосед с прочитанным токеном может вызвать диалог подтверждения, но не завершить
подпись без явного «да» оператора. На однопользовательском рабочем месте (целевой
сценарий) сторонних локальных процессов нет.

#### Scenario: Чужой Origin
- **WHEN** запрос к агенту приходит с Origin, отсутствующим в allowlist
- **THEN** запрос отвергается до какого-либо обращения к PKCS#11-модулю

#### Scenario: Same-origin запрос без Origin
- **WHEN** кабинет, отданный агентом, шлёт same-origin GET (`/info`) — браузер не сопровождает его заголовком Origin — с валидным парным токеном
- **THEN** запрос обслуживается: отсутствие Origin само по себе не отказ, гейтом выступает токен

#### Scenario: Отсутствует парный токен
- **WHEN** запрос содержит допустимый Origin, но парный токен сессии отсутствует или неверен
- **THEN** запрос отвергается; событие фиксируется в выводе агента

#### Scenario: PIN запрашивается локально
- **WHEN** агент получает валидный запрос подписи, требующий входа в слот
- **THEN** PIN запрашивается в терминале/pinentry агента; в HTTP-обмене с кабинетом PIN не фигурирует

#### Scenario: Кабинет отдан агентом same-origin
- **WHEN** агент запущен с раздачей кабинета и оператор открывает его localhost-адрес
- **THEN** кабинет открывается с того же origin, что и `/sign`, с уже подставленными адресом агента и парным токеном; подпись всё равно требует токена в запросе и явного подтверждения оператора

#### Scenario: Статика не обходит гейты подписи
- **WHEN** к агенту, раздающему кабинет, приходит запрос `/sign` без парного токена (или запрос на файл вне набора ассетов кабинета)
- **THEN** `/sign` без токена отвергается как обычно, а запрос файла вне набора ассетов не отдаёт ничего из файловой системы

### Requirement: Кроссплатформенность стороны выпуска

CLI `issuer` и агент `issuer serve` ДОЛЖНЫ (MUST) работать на Linux, macOS и
Windows: платформо-специфичные механизмы (путь PKCS#11-модуля, права файлов,
запуск pinentry) ДОЛЖНЫ (MUST) быть абстрагированы, а инварианты безопасности
(127.0.0.1, Origin, парный токен, подтверждение операции) ДОЛЖНЫ (MUST)
сохраняться на каждой платформе. Агент работает как foreground-процесс на время
сессии выпуска; фоновая демонизация/автостарт стороной выпуска НЕ
предоставляются. Enforcement-сторона (агент Tessera на устройствах) остаётся
Linux-only и данным требованием не затрагивается.

#### Scenario: Выпуск с macOS/Windows рабочего места
- **WHEN** диспетчер запускает `issuer serve` на macOS или Windows с PKCS#11-модулем своего токена
- **THEN** цикл выпуска работает идентично Linux: те же проверки, тот же протокол кабинета, тот же порядок подтверждения
