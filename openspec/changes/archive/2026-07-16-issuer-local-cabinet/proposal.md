# Proposal: issuer-local-cabinet

## Why

Веб-кабинет выпуска ([[issuer-tooling]]) сейчас доставляется как хостируемый
статический SPA (issuer.tessera-access.com/.ru), который обращается к локальному
агенту `issuer serve` на 127.0.0.1 за подписью. Но у любого реального оператора
агент **уже** стоит локально (там PKCS#11/токен) — а раз агент локальный, он
может и сам отдать SPA. Хостинг доставляет ровно одну вещь (статику), которую
агент закрывает сам, и при этом:

- **противоречит целевой среде.** Основной сценарий — офлайн/изолированные парки
  (Astra, без исходящего интернета). Там хостируемый URL недостижим в принципе;
  весь нарратив продукта — «офлайн, self-hosted, ноль исходящих».
- **добавляет поверхность.** Cross-origin (CDN → localhost) — это ровно та
  сложность (Origin-allowlist, парный токен, «тот ли это кабинет из CDN»,
  SHA-256-пиннинг против supply-chain, см. threat-model §11.1.3), которую
  same-origin-раздача из агента схлопывает.
- **требует инфраструктуры без нужды** (два домена, серты, CDN, DNS,
  deploy-пайплайн, purge).

Раз агент запускают явно, чтобы получить кабинет, **фоновый демон/автостарт тоже
теряет смысл** — это возврат к тому, что спека и так называла базовым режимом
(«на время сессии, меньше окно атаки»).

## What Changes

- **`issuer serve` раздаёт кабинет same-origin.** Агент опционально отдаёт
  статику SPA (встроенную в бинарь) на том же 127.0.0.1-origin, что и `/sign`.
  Каноничный путь: `issuer serve …` → поднялся localhost с кабинетом → выпуск
  офлайн из одного подписанного бинаря.
- **Авто-инъекция парного токена.** Раз агент сам отдаёт HTML, он вшивает токен
  сессии (и свой origin) в отданную страницу — оператору не нужно переписывать
  адрес и токен. Токен по-прежнему гейтит `/sign`; подтверждение оператора
  по-прежнему авторизует операцию.
- **Убираем демон/автостарт.** `--daemon-token-file`, per-OS рантайм-файлы
  токена, примеры автостарта (systemd/launchd/Task Scheduler) удаляются. `serve`
  — foreground на время сессии. (Осознанное удаление отгруженного; пре-1.0,
  0.MINOR=мажор по [[tessera-compat-policy]].)
- **Ретайрим хостинг (ленивый).** Удаляем `deploy-issuer.yml`; доки переводим на
  локальный запуск; ссылку с лендинга уже сняли. Облачные ресурсы (бакеты/CDN/CF
  Pages/DNS `issuer.*`) гасим отдельным спокойным проходом (редирект → доки).

Демо-режим для лендинга (sandbox с mock-агентом) — НЕ в этот change, отложено.

## Impact

- Спека `issuer-signing`: MODIFIED «Локальный агент issuer serve» (раздача
  кабинета + инъекция токена + same-origin), MODIFIED «Кроссплатформенность»
  (убрать демон-токен-файл/автостарт; серв foreground), +сценарий. Заполнить
  Purpose (был TBD).
- Код `tessera_issuer`: `serve` раздаёт статику (embed за фичей `embed-cabinet` /
  `--cabinet-dir <path>`), инъекция токена; удаление демон-пути.
- CI/инфра: удалить `deploy-issuer.yml`; релизный билд встраивает cabinet/dist.
- Доки `docs/{ru,en}/issuer.md`, threat-model §11: локальный запуск вместо
  хостинга; убрать демон/автостарт-разделы.
- Инвариант «serverless» кабинета НЕ нарушается: агент раздаёт статику + локальный
  мост подписи, вся сборка/проверка TBS так же в WASM на клиенте, внешних
  обращений нет (127.0.0.1 = агент).
