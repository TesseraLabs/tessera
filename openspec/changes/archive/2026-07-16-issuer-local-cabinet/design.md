# Design: issuer-local-cabinet

`issuer serve` перестаёт быть только мостом подписи и становится способом
*запустить* кабинет: один подписанный бинарь раздаёт SPA и обслуживает `/sign` на
одном 127.0.0.1-origin. Хостируемый кабинет и фоновый демон уходят.

## 1. Раздача кабинета из агента

Текущий `serve` (`crates/tessera_issuer/src/serve.rs`, `tiny_http`) роутит только
`POST /sign` + `GET /info` (см. `match (input.method, input.path)`). Добавляем
раздачу статики:

- **Источник ассетов — две опции:**
  - **Встроенные** (`include_dir!` за cargo-фичей `embed-cabinet`): `cabinet/dist/`
    вшивается в бинарь. Релизный `issuer` собирается ПОСЛЕ `cabinet/build.sh`, с
    фичей → полностью автономный бинарь (офлайн, целостность даёт подпись бинаря,
    отдельный SHA-256-пиннинг не нужен).
  - **Каталогом** (`--cabinet-dir <path>`): раздать внешний `dist/` (работает и
    без фичи; для self-hosted/dev). Явный `--cabinet-dir` перекрывает встроенное.
  - Ни фичи, ни флага → сегодняшнее поведение (мост без кабинета), обратная
    совместимость.
- **Флаги (раздача по умолчанию):** `issuer serve` раздаёт кабинет **по
  умолчанию**, когда он доступен (встроен фичей `embed-cabinet` либо задан
  `--cabinet-dir <dist>`). `--no-cabinet` отключает раздачу (чистый режим моста —
  когда SPA раздают снаружи). Если встроенного кабинета нет и `--cabinet-dir` не
  задан — агент падает в мост автоматически. (`--module`/`--key` остаются
  обязательными — без них нечем подписывать; специфичны для развёртывания.)
  Прежний `--serve-cabinet` убран как избыточный.
- **Роутинг:** `GET /` → `index.html`; `GET /<asset>` для фиксированного набора
  (`main.js`, `main.js.map`, `styles.css`, `*.wasm`). Всё, что не в наборе, — 404;
  `/sign`,`/info` не затеняются. Никакого доступа к произвольным путям ФС: при
  каталоге — канонизировать и проверять, что путь внутри `dist/` (path-traversal),
  при встроенном — только по ключам `include_dir`. MIME по расширению (важно:
  `application/wasm`).
- **CSP:** кабинет несёт свою `<meta>`-CSP (`default-src 'self'`); при раздаче
  агент ДОЛЖЕН отдавать её и заголовком (в т.ч. `frame-ancestors 'none'`, который
  через `<meta>` не действует) — то, что кабинет-док помечал как задачу хостинга,
  теперь делает агент.
- **Гейты в same-origin (важно — найдено живым прогоном):** браузеры НЕ шлют
  заголовок `Origin` на same-origin GET, поэтому собственные `GET /info` кабинета
  приходят без Origin. Origin-allowlist (задумка cross-origin-хостинга) их
  отвергал → кабинет показывал «не подключён». Фикс: **токен — основной гейт**;
  Origin проверяется только если ПРИСУТСТВУЕТ (есть → в allowlist, нет → пропуск,
  гейтит токен). При раздаче агент **авто-добавляет свой bound-origin** в allowlist
  и `--allow-origin` больше не обязателен (при `--port 0` порт заранее неизвестен).
  Безопасность сохранена: cross-origin не прочитает токен и не поставит его
  заголовок без preflight; DNS-rebind несёт Origin атакующего → отсекается.

## 2. Инъекция парного токена (same-origin)

Раз агент отдаёт HTML, он вшивает в `index.html` при отдаче параметры сессии,
чтобы убрать ручной ввод адреса/токена:

- Агент подставляет в отдаваемый `index.html` небольшой инлайн-скрипт-конфиг
  (например `window.__TESSERA_AGENT__ = {origin, token}`) ЛИБО тег
  `<meta name="tessera-agent-token" content="…">`. Кабинет, увидев его,
  преднастраивает адрес агента (= свой origin) и токен; поля остаются видимыми и
  редактируемыми (не прячем — оператор должен видеть, к чему подключается).
- Инъекция идёт в CSP-совместимой форме (инлайн-скрипт → потребует хэш/`nonce` в
  script-src, либо использовать `<meta>`+чтение из JS — предпочтительно `<meta>`,
  без ослабления CSP).
- Безопасность: токен в странице эквивалентен тому, что агент сам её отдал
  (same-origin). Токен по-прежнему отсекает посторонние локальные процессы от
  `/sign`; подтверждение оператора не заменяется. Origin-allowlist при раздаче
  включает свой origin.
- Кабинет (`cabinet/src/state/sessionConfig.ts` + `ui/app.ts`) читает инъекцию
  при старте: если есть — предзаполняет `#agentSettings`/статус; если нет
  (внешний хостинг/dev) — как сейчас, ручной ввод. Изменение маленькое и
  обратно совместимое.

### 2a. Агент-секция при инъекции — только статус коннекта

Когда инъекция есть (агент сам отдал страницу), адрес/токен/ключ известны по
построению и редактировать их незачем. Агент-секция сворачивается до **одного
индикатора: подключён / не подключён**. Поля адрес/токен/ключ и «Сохранить»
скрыты (полная секция остаётся только в fallback без инъекции — внешний
self-hosted-static). Статус определяется авто-проверкой `GET /info` при загрузке
(тихо, без клика «Подключить»); это не меняет модель подписи — каждая операция
всё равно требует подтверждения оператора на стороне агента.

## 3. Удаление демона/автостарта

Из `serve`/CLI убираем: `--daemon-token-file`, запись токена в per-OS
рантайм-каталоги, вся логика демон-режима; из репозитория — примеры автостарта
`crates/tessera_issuer/examples/` (systemd/launchd/Task Scheduler). Токен сессии
печатается в stdout (как в базовом режиме) и/или вшивается в отданный кабинет
(п. 2). Кроссплатформенные абстракции токен-файла демона удаляются; остаются
абстракции пути PKCS#11 и pinentry.

## 4. Ретайр хостинга (ленивый)

- Удалить workflow `deploy-issuer.yml` (CI больше не деплоит issuer.*).
- Доки `docs/{ru,en}/issuer.md`: раздел хостинга → «локальный запуск»
  (`issuer serve --serve-cabinet`); удалить разделы демона/автостарта; раздел
  «Веб-кабинет» переписать под «кабинет = бинарь». threat-model §11: убрать
  supply-chain-довод про CDN-хэши (кабинет за подписью бинаря), same-origin.
- Ссылка с лендинга — уже снята.
- Облачные ресурсы (бакеты `issuer.tessera-access.ru`, CDN `bc8ro54hldkfbptepsfd`,
  CF Pages `tessera-issuer`, DNS `issuer.*`) — НЕ удаляем в этом change; отдельный
  спокойный проход: `issuer.* → 301` на доки, затем снос. (Вне scope спеки —
  операционный хвост.)

## 5. Границы

Не входит: демо-режим лендинга (sandbox/mock-агент); авто-открытие браузера
(`--open`) — приятно, но опционально, можно позже; изменения ядра выпуска, набора
операций, контракта подписи адаптеров; enforcement-сторона.

## 6. Совместимость

Удаление демон-режима и хостинга — видимая смена поверхности инструмента. Пре-1.0,
по [[tessera-compat-policy]] 0.MINOR трактуется как мажор — допустимо; отражаем в
changelog. `serve` без новых флагов сохраняет поведение «мост» (не ломает тех, кто
раздаёт SPA сам).
