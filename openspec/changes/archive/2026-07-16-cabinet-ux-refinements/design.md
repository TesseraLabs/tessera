# Design: cabinet-ux-refinements

Правки поверх отгруженного кабинета (`cabinet/`, vanilla-TS SPA + `el()`-хелпер,
i18n через `dict.ts`, тесты на `node:test`). Ни один serverless-инвариант и
контракт доверия не меняется: подпись — за агентом/токеном, стейт — файлы,
внешних сетевых обращений нет; CSP кабинета остаётся прежней (модалки и вкладки —
локальный DOM, без внешних ресурсов).

## 1. Инвентарь: конструктор + связь с формами (нормативная часть)

**Модель.** Инвентарь в состоянии App — уже существующий `AcceptedSnapshot`
(`core/snapshot.ts`: `{origin, payload:{hosts,users,roles,tags,generated_at}, ageSeconds}`).
Добавляется второй путь его получения:

- **Конструктор.** Списочные редакторы: устройства (пара `id` + необязательный
  `label`), пользователи (`stringListInput`), роли (`stringListInput`), теги
  (`tagListInput` без фиксированного префикса). Кнопка «Собрать» формирует
  `SnapshotPayload` и принимает его как `origin: "manual"`, `ageSeconds: 0`.
  Новый чистый хелпер `buildManualSnapshot(payload): SnapshotFile`
  (`core/snapshot.ts`) сериализует `{payload_json, signature_b64: null}` — так
  собранный инвентарь round-trip'ится через существующий `acceptSnapshot`
  (та же валидация, что для файла) и может быть скачан `.json`-файлом.
- **Файл.** Существующий путь `onSnapshotFileChosen` без изменений.
- Переключение режимов — радио/сегмент внутри секции инвентаря.

**Связь с формами.** Инвентарь (`#snapshot?.payload`) прокидывается в
`buildLeafForm`:

- `host_binding` / `user_binding` — новый виджет `suggestingStringListInput(add,
  remove, suggestions[])` в `ui/widgets.ts`: те же строки-строки, что
  `stringListInput`, но каждый `<input>` привязан к `<datalist>` из `suggestions`
  (CSP-safe, нативный комбобокс — выбор из инвентаря + свободный ввод сохраняется).
  Подсказки устройств — `payload.hosts.map(h => h.id)` (label в тексте опции),
  пользователей — `payload.users`.
- Роли — новый чистый хелпер `rolesForSelection(envelopeRoles, snapshotRoles?)`
  (`core/envelope.ts`): при наличии инвентаря возвращает пересечение, иначе —
  `selectableRoles(parent)` как сейчас. `roleCheckboxGroup` строится на нём;
  когда пересечение уже рамок родителя, под группой — пометка о сужении.
- Теги CA-формы (`require_tags`) получают datalist-подсказки ключей/значений из
  инвентаря (мелочь, необязательный полировочный штрих).

**Ограничение re-render.** `render()` пересобирает формы на каждое изменение
состояния (в т.ч. установку инвентаря) — незаполненная форма операции при этом
сбрасывается. Это существующая модель; штатный порядок (инвентарь готовят до
формы, docs шаг 3→4) её обходит. В рамках этой правки не расширяем.

## 2. Навигация (2 вкладки, ненормативно)

`#activeTab: "issue" | "journal"` в состоянии App (по умолчанию `issue`).
Таб-бар — под шапкой (заголовок + переключатель языка сохраняются глобально).
`render()`:

```
header
tab-bar [Выпуск][Журнал]
error-banner?
activeTab === "issue"
  ? [parent, operation?, snapshot, agent, summary?]
  : [journal]
```

Состояние (родитель, инвентарь, агент, `#pending`) живёт в App и переживает
переключение вкладок. Новые ключи i18n: `tab_issue`, `tab_journal`. CSS —
`.tab-bar`, `.tab-bar button.active` в духе существующего `.lang-switch`.

## 3. Справки (модалки, ненормативно)

Переиспользуемый `ui/modal.ts`: `openModal(title, bodyNodes[])` — оверлей + панель
+ кнопка закрытия, закрытие по Esc и клику по фону, возврат фокуса. Только
локальный DOM (CSP). Кнопка «?» рядом с заголовком блока родителя и блока агента.

- **Родитель.** Родитель кабинета — корень парка (создаётся владельцем PKI, вне
  ежедневного выпуска) ЛИБО CA организации (выдаётся под корнем командой
  `issuer issue-ca …`). Оператор получает файл родителя от уровня выше. Показать
  CLI-пример, принимаемые форматы (PEM/DER), ссылку на `docs/issuer.md`.
- **Агент.** Как установить/включить/настроить `issuer serve`: базовый запуск
  `issuer serve --allow-origin <origin> --module <path> --key <label>`; автостарт
  Linux (`systemd --user`), macOS (launchd LaunchAgent), Windows (Task Scheduler)
  из `crates/tessera_issuer/examples/`; парный токен сессии (stdout /
  `--daemon-token-file`); Origin-allowlist. Контент — из `docs/issuer.md`
  §«Агент issuer serve», без новых утверждений.

Тексты справок — ключи i18n (абзацы отдельными ключами); код-сниппеты — литералы
в билдере модалки (технические, не переводятся — как в остальном dict).

## 4. Агент: «Подключить» + статус (ненормативно)

`#agentStatus: "unknown" | "connecting" | "connected" | "error"` в состоянии App
(seed при `render()`, поэтому переживает перерендер и вкладки). Кнопки:

- «Сохранить» — persist настроек (как сейчас).
- «Подключить» — persist + `agentInfo(GET /info)` → `connected`/`error`; индикатор
  (точка + подпись) обновляется. Обновление индикатора — императивно на самом
  элементе (без полного `render()`, чтобы не терять форму); `#agentStatus` в
  состоянии — чтобы значение восстановилось при следующем `render()`/смене вкладки.
- Правка полей адреса/токена/ключа → статус в `unknown` (устарел), тоже
  императивно на индикаторе.

Контракт агента (`issuer-signing`) не трогается — это UI над существующими
`GET /info` и `POST /sign`.

## 5. Тесты

Чистая логика — по образцу существующих тестов `core/*`:

- `rolesForSelection`: пересечение, пустой инвентарь → рамки родителя, роль вне
  рамок отбрасывается.
- `buildManualSnapshot` + `acceptSnapshot` round-trip: собранный payload
  принимается как `manual`.
- `suggestingStringListInput.getValue`: свободный ввод сохраняется, пустые строки
  отфильтрованы (паритет со `stringListInput`).

Модалки/вкладки/индикатор — тонкий DOM-слой (как остальной `ui/*`, без юнитов);
покрываются визуальной проверкой в браузере.

## 6. Границы

Не входит: подпись снапшота в кабинете (остаётся операторским ключом организации,
собранный инвентарь — всегда ручной); правка re-render-модели форм; изменения
CLI/ядра/контракта агента; новые операции выпуска.
