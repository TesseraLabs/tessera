# Proposal: cabinet-ux-refinements

## Why

Первый проход веб-кабинета выпуска ([[issuer-tooling]], отгружен) собрал полный
поток на одной прокручиваемой странице, а инвентарь устройств принимал только
готовым снапшот-файлом. На практике оператору не хватает опор:

- **Инвентарь.** Требовать заранее сгенерированный снапшот-файл на каждый выпуск —
  трение: для разового листа проще собрать список устройств/пользователей/ролей
  прямо в кабинете. Спека сейчас жёстко фиксирует источник форм как «снапшот-файл»;
  ручное заполнение упомянуто, но не как структурированный конструктор, и инвентарь
  не связан с полями форм — host/user вводятся свободным текстом, роли берутся
  только из envelope. Инвентарь, который есть у оператора, никак не помогает
  заполнять форму без ошибок.
- **Ориентация.** Родительский сертификат и агент подписи — точки, где оператор
  чаще всего застревает («откуда взять родителя?», «как поднять агент?»), а
  подсказок в интерфейсе нет. Статус агента эфемерный (сбрасывается при
  перерендере), явной операции «подключиться» нет.
- **Раскладка.** Журнал выпусков — отдельная задача (загрузка/проверка/выгрузка),
  а не часть потока выпуска; на общей странице он мешает.

Правки не меняют serverless-инварианты и модель доверия (подпись всё так же за
агентом и токеном, стейт — файлы, внешних обращений нет).

## What Changes

Нормативная часть (одно требование спеки `issuer-cabinet`):

- **MODIFIED** `Инвентарь для форм`: источник инвентаря — снапшот-файл **или**
  конструктор в кабинете (in-app сборка неподписанного «ручного» снапшота из
  списков устройств/пользователей/ролей/тегов). Добавлена нормативная связь
  инвентаря с формами выпуска: устройства и пользователи предлагаются из инвентаря
  с сохранением свободного ввода; набор ролей в форме — пересечение рамок родителя
  (`envelope`) с ролями инвентаря; проверка подписи загруженного снапшота и запрет
  блокировать выпуск при отсутствии инвентаря — без изменений.

Ненормативная часть (UI-аффордансы, без новых MUST — детали в `design.md`):

- Навигация двумя вкладками: «Выпуск» (родитель → операция → инвентарь → агент →
  сводка) и «Журнал».
- Контекстные справки (модальные окна) у блока родительского сертификата и блока
  агента подписи: как получить/создать родителя, как установить/включить/настроить
  агент `issuer serve` (Linux/macOS/Windows). Контент опирается на `docs/issuer.md`.
- Агент подписи: явная кнопка «Подключить», индикатор состояния
  (подключён/не подключён), переживающий перерендеры и переключение вкладок.

## Impact

- Спека: `issuer-cabinet` — 1 модифицированное требование (+2 сценария), плюс
  заполнение `Purpose` (был `TBD` после архивации issuer-tooling).
- Код: `cabinet/` (SPA) — новый виджет пикеров с подсказками, конструктор
  инвентаря, модалки, таб-навигация, состояние статуса агента; чистые хелперы
  (пересечение ролей, сборка ручного снапшота) под тесты.
- Паритет CLI не затрагивается: конструктор — ввод инвентаря, не операция выпуска;
  проверки ядра и набор операций неизменны.
- Документация: `docs/{ru,en}/issuer.md` — порядок работы кабинета
  (конструктор инвентаря, вкладка журнала).
