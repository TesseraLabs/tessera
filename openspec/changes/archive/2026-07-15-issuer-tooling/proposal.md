# Proposal: issuer-tooling

## Why

Tessera проверяет сертификаты, но не даёт инструментов их выпускать: сегодня выпуск —
это ручной openssl-скриптинг, в котором легко ошибиться в custom-расширениях
(арка `2.25.<UUID>`) и невозможно проверить рамки делегирования до того, как
сертификат уедет в парк. Дизайн инструментов выпуска (brainstorming 2026-07-05,
tessera-ws) закрывает это: одно Rust-ядро выпуска и обёртки над ним — CLI для
автоматизации и один статический веб-кабинет без серверной логики; доступные
операции определяются предъявленным родительским сертификатом (корень парка →
выпуск CA организаций, CA организации → выпуск листов смен инженерам).
Приватные ключи никогда не покидают
токен/HSM/Vault: инструмент — не кастодиан.

## What Changes

- Новый крейт **`tessera_issuer`** в workspace: ядро выпуска сертификатов.
  - **Сборка TBS**: лист смены и CA-сертификат организации с custom-расширениями
    Tessera (host-binding/user-binding, теги, рамки делегирования — те же OID-дуги
    `2.25.<UUID>`, что проверяет Engine).
  - **Проверки на стороне выпуска**: монотонное сужение рамок делегирования
    (лист ⊆ рамки CA ⊆ рамки корня) — дублируют enforcement Engine, чтобы
    невалидный сертификат не выпускался вовсе, а не отбивался на устройстве.
  - **Выпуск по CSR (PKCS#10)** — равноправно с прямой передачей публичного
    ключа: инженер генерирует ключ на своём токене и отдаёт CSR, оператор
    проверяет самоподпись и выпускает; атрибуты CSR на скоуп не влияют
    (скоуп задаёт оператор).
  - **Серийные номера**: случайные 128-битные (RFC 5280) — несколько независимых
    диспетчеров выпускают без координации и без общего состояния.
  - **Выпуск CRL**: клиентская операция того же ядра — подписать новый CRL ключом CA
    с монотонным `crlNumber`.
- **Адаптеры подписи** за единым интерфейсом «подписать готовый TBS ключом слота»:
  PKCS#11 (токен и HSM — один код) и Vault/OpenBao **Transit** (подпись готового TBS;
  Vault PKI не используется — `encoding/asn1` в Go не парсит OID-дуги больше int64,
  наши расширения через него не выпустить).
- **Обёртки над ядром**:
  - CLI `issuer` (тот же крейт, bin) — сценарии автоматизации (тикет-системы);
    режим `issuer serve` — локальный агент на 127.0.0.1, мост «браузер ↔ токен»
    для веб-кабинета (прецедент Fortify).
  - WASM-сборка ядра для веб-кабинета: один статический SPA без сервера, весь
    стейт — файлы (родительский сертификат, выпущенные сертификаты, журнал, CRL);
    доступные операции выводятся из предъявленного родительского сертификата;
    работает self-hosted и офлайн.
- **Журнал выпусков**: локальный подписанный hash-chain (паттерн `logging-audit`);
  первичная правда — аудит входов на устройствах, журнал выпуска вторичен.
- Вне scope этого change: курьерский payload на носителе (отдельный change
  `courier-payload`), серверная сторона (снапшоты инвентаря, four-eyes,
  приём журналов), ГОСТ-fallback через браузерные плагины (Рутокен Плагин /
  КриптоПро ЭЦП Browser plug-in) — до подтверждения, что они умеют подпись
  произвольного TBS.

## Capabilities

### New Capabilities

- `cert-issuance`: сборка TBS с расширениями Tessera, проверки монотонности рамок
  делегирования на стороне выпуска, случайные 128-битные серийники, выпуск CRL
  с монотонным `crlNumber`.
- `issuer-signing`: единый интерфейс подписи готового TBS; адаптеры PKCS#11
  (токен/HSM) и Vault Transit; ключи не покидают модуль подписи; локальный агент
  `issuer serve` как мост браузер ↔ PKCS#11.
- `issuer-cabinet`: serverless-инварианты обёрток (CLI и единый веб-кабинет):
  вся логика на клиенте, стейт — файлы, источник инвентаря для форм — подписанный
  снапшот файлом; один кабинет на все роли выпуска — доступные операции
  определяются предъявленным родительским сертификатом (корень → выпуск CA
  организаций с рамками; CA организации → выпуск листов смен в этих рамках).
- `issuance-journal`: локальный подписанный hash-chain журнал операций выпуска
  (выпуск листа, выпуск CA, выпуск CRL), верификация цепочки.

## Impact

- **Код**: новый крейт `tessera_issuer` (lib + bin `issuer`) в workspace;
  cdylib/wasm32-таргет для веб-кабинета. Существующие крейты не трогаются;
  общие определения OID-расширений — кандидат на вынос в разделяемый модуль
  (сейчас продублированы в `tessera_core`), решение — в design.
- **Сборка/CI**: wasm32-таргет в CI, сборка статического SPA — отдельным
  артефактом (не входит в deb агента).
- **Зависимости**: семантика рамок делегирования и тегов — main-спеки
  `device-tags`, `cert-scope-binding`; формат журнала — паттерны `logging-audit`;
  формат CRL и `crlNumber` — `revocation`.
- **Связанные документы**: приватный дизайн 2026-07-05 (tessera-ws); обновление
  `commercial-split.md` и линейки Access/Codes — отдельным PR в tessera-ws,
  не этим change.
- **Threat-model**: новая поверхность — локальный агент `issuer serve`
  (127.0.0.1, CORS/origin-контроль), обращение с PIN токена, целостность
  WASM-артефакта; дополнение threat-model — задачей этого change.
