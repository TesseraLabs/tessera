# Design: cabinet-operation-choice

## Context

Реализовано 2026-07-17 (ветка cabinet-operation-choice) до оформления change —
этот документ фиксирует принятые решения постфактум, вместе с delta-спекой.
`renderOperationSection` в `cabinet/src/ui/app.ts` выбирал форму жёстко по
`info.kind`; wasm-API (`buildCaTbs`/`buildLeafTbs`) уже принимал любого
родителя с рамками.

## Goals / Non-Goals

**Goals:** выбор операции выпуска для любого CA-родителя с рамками; прежние
дефолты; чистая, тестируемая логика доступных операций.

**Non-Goals:** ограничение глубины делегирования (pathLen/max_depth — ядро
позволяет цепочку любой глубины, осознанно отложено); изменения wasm-ядра,
протокола агента, формата сертификатов.

## Decisions

- Чистая функция доступных операций и дефолта в `cabinet/src/ui/operations.ts`
  (`availableIssuanceOperations`, `defaultIssuanceOperation`) + unit-тесты;
  UI-состояние выбора — приватное поле App, переживает re-render, сбрасывается
  на дефолт при загрузке нового родителя.
- Переключатель — радио-пара в идиоме существующего snapshot-mode picker
  (стили `.operation-choice-picker`).
- l10n: три новых ключа в обоих словарях; описания родителя
  (`parent_kind_*_desc`, `help_parent_p1`) переформулированы под «оба типа
  выпускают обе операции».

## Risks / Trade-offs

- [Переключение операции сбрасывает полузаполненную форму] → то же поведение,
  что у существующего snapshot-переключателя (полный re-render) — принято.
- [Корень может выпускать листы напрямую, минуя орг-уровень] → это соответствие
  модели ядра; организационное разделение обязанностей — вне UI (фиксируется
  доками, не кодом).
