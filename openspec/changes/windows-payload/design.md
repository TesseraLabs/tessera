# Design: windows-payload

## Context

Схема среза роли (`role/schema.rs`) парсит `os = "windows"`, но payload для него
пуст by design: любое поле отвергается (`validate_payload_for_os`, ветка
`RoleOs::Windows`). Концепт Windows-адаптера (2026-06-07) зафиксировал состав
Windows-примитивов роли: локальные группы (в токен сессии через
`LsaLogonUser LocalGroups`), integrity level (`SetTokenInformation
(TokenIntegrityLevel)`), отзыв привилегий, лимиты сессии. Research-проход №2
(2026-06-05, 24 источника) подтвердил реализуемость и опроверг два допущения,
которые схема не должна закладывать: integrity-уровней НЕ «ровно 4 с
фиксированным маппингом»; `SetTokenInformation` НЕ мутирует произвольные
атрибуты токена.

Формат роли открыт для всех платформ (licensing-distribution: «схема парсится
целиком в открытой сборке»); enforcement для Windows не существует ни в какой
сборке — секция parse-only, как SELinux-секция до появления её адаптера.

## Goals / Non-Goals

**Goals:**

- Определить содержимое `[payload]` для `os = "windows"` — полный формат,
  достаточный для будущего адаптера (windows-tcb-service) без ревизии схемы.
- Сохранить инварианты схемы: строгий парсинг (`deny_unknown_fields`),
  кросс-ОС отторжение, полный парсинг в открытой сборке.
- Обратная совместимость: существующие astra/linux-срезы валидны без изменений.

**Non-Goals:**

- Enforcement (применение к сессии/токену) — change `windows-tcb-service`.
- AD/доменные сценарии (доменные SID, GPO) — отдельный research-проход.
- Локализация имён/описаний — YAGNI, как и раньше.
- Расширение `[session]` новыми Windows-полями — лимиты Windows живут в
  `[payload.windows.limits]`.

## Decisions

### D1. Вложенная секция `[payload.windows]`, а не плоские поля

Прецедент — `[payload.selinux]`: платформенно-специфичный блок с
подполями. Альтернатива (плоские поля `windows_groups`, `windows_integrity`
в `Payload`) отвергнута: засоряет общий struct, а поле `groups` уже занято
Linux-семантикой (supplementary-группы ≠ SAM-группы: другой резолвер, другой
формат имён). Разделение делает кросс-ОС валидацию тривиальной: одна проверка
`payload.windows.is_some()` вместо ревизии поля за полем.

### D2. Поля секции

```toml
[payload.windows]
groups = ["TesseraOper", "S-1-5-32-559"]   # имена локальных групп SAM и/или SID-строки
integrity_level = "medium"                  # untrusted | low | medium | high
privileges_remove = ["SeShutdownPrivilege"] # имена привилегий к отзыву

[payload.windows.limits]
memory_max = "512M"    # Job Object: лимит commit-памяти сессии
process_max = 256      # Job Object: ActiveProcessLimit
```

- `groups: Option<Vec<String>>` — элемент: имя локальной группы SAM или
  SID-строка (`S-1-…`). Валидация формата: непустая строка; строка,
  начинающаяся с `S-1-`, дополнительно проверяется на синтаксис SID
  (`S-1-<число>(-<число>)+`). Резолв имени → SID — задача адаптера (runtime),
  не схемы: схема не знает SAM устройства.
- `integrity_level: Option<String>` — закрытый whitelist
  `untrusted|low|medium|high`. Research опроверг «ровно 4 уровня», но роли
  нужен предсказуемый словарь: whitelist — это словарь **роли**, не модель ОС;
  маппинг в RID (`SECURITY_MANDATORY_*_RID`) — константы адаптера. `system`
  не входит: пользовательская сессия с system integrity — анти-паттерн,
  роль не должна уметь его запросить.
- `privileges_remove: Option<Vec<String>>` — имена привилегий
  (`Se…Privilege`). Семантика — отзыв (сужение), не выдача: добавить
  привилегии ролью нельзя (симметрия «сужение = `CreateRestrictedToken` /
  `AdjustTokenPrivileges`», добавление токеном не предусмотрено концептом).
  Валидация: непустая строка, префикс `Se`, суффикс `Privilege` — ловит
  опечатки без словаря всех привилегий (словарь версионируется Windows,
  фиксировать его в схеме хрупко).
- `limits` — свои поля, не переиспользование `LinuxLimits` (`nofile`/`nproc`
  бессмысленны на Windows). `memory_max` — строка в конвенции размеров,
  уже принятой в `[session].memory_max` (`512M`); `process_max: u32`.

### D3. `[session]` для Windows-срезов — только `max_ttl_seconds`

Поля `memory_max`/`tasks_max`/`cpu_weight`/`io_weight` в `[session]`
документированы как systemd per-session лимиты — на Windows они не имеют
носителя. Разрешить их «на всякий случай» = молчаливо-неприменяемая
конфигурация, что противоречит духу строгой схемы. Новая проверка
`validate_session_for_os`: для `os = "windows"` присутствие systemd-полей —
ошибка валидации. `max_ttl_seconds` универсален (жёсткий TTL — концептный
инвариант всех платформ). Для astra/linux поведение не меняется.

Альтернатива — перенести TTL в `[payload.windows]` — отвергнута: TTL
платформенно-нейтрален, его место в `[session]` на всех ОС.

### D4. Все поля опциональны, пустая секция валидна

`[payload.windows]` без полей — валидный срез (роль «просто личность и TTL»),
симметрично остальным ОС. Никаких обязательных полей: минимальный Windows-срез
не обязан перечислять группы.

## Risks / Trade-offs

- [Формат зафиксирован до первого адаптера — риск ревизии при имплементации
  windows-tcb-service] → состав полей выведен из research-верифицированных
  примитивов (LocalGroups, TokenIntegrityLevel, AdjustTokenPrivileges,
  Job Object), а не из желаемого UX; compat-policy разрешает добавление
  опциональных полей без слома.
- [Whitelist integrity_level может оказаться узким (нестандартные уровни)] →
  расширение whitelist'а — обратно-совместимое добавление; произвольные
  RID-числа осознанно не допущены (роль-файл читают люди).
- [Валидация SID/привилегий синтаксическая, не семантическая: опечатка в имени
  группы доживёт до runtime] → это симметрично Linux (`groups` там тоже не
  резолвятся при lint); семантическая проверка — `doctor` будущего адаптера.

## Migration Plan

Чистое добавление: новые срезы с `[payload.windows]` валидны после релиза,
старые срезы всех ОС не затронуты. Rollback = откат релиза (новая секция
перестанет парситься — Windows-срезов в проде не существует, ломать нечего).

## Open Questions

- Нужен ли `deny_sids` (deny-only SID для `CreateRestrictedToken`) как
  отдельное поле — отложено до windows-tcb-service: возможно, отзыв привилегий
  покрывает потребность сужения, а deny-only SID выводится адаптером.
- Доменные SID: `S-1-5-21-<домен>-<RID>` синтаксически неотличим от
  локального машинного `S-1-5-21-<машина>-<RID>` — схема пропускает оба и
  различать их не пытается (различение возможно только резолвом на
  устройстве). Семантика доменных групп в payload — вне скоупа
  (AD — отдельный research-проход); поведение адаптера с нерезолвящимися
  именами зафиксировано в `windows-tcb-service` (fail-closed, резолв только
  по локальным данным).
