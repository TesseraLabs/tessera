# Proposal: issuer-file-backend-and-serve-backends

## Why

Сторона выпуска поддерживает два хранилища ключа CA — PKCS#11 и Vault Transit, — но
агент кабинета `issuer serve` жёстко собран вокруг PKCS#11: запустить кабинет поверх
Vault нельзя, хотя дизайн issuer-tooling это обещал. Кейса «ключ в локальном файле»
нет вовсе — сейчас он закрывается только обходом через SoftHSM, что неудобно для
стендов, CI и малых инсталляций, принявших риск файлового ключа.

## What Changes

- Новый бэкенд подписи `file`: ключ CA в локальном файле PKCS#8 (PEM/DER),
  включая зашифрованный (EncryptedPrivateKeyInfo, PBES2/scrypt). Прод-статус:
  зашифрованный ключ — рекомендация, незашифрованный допускается с проверкой
  прав файла и предупреждением. Алгоритмы: ECDSA P-256/P-384, RSA-SHA256
  (паритет с `--algorithm`); ГОСТ файловым бэкендом не поддерживается — только
  через PKCS#11.
- `issuer serve` получает выбор бэкенда `--backend pkcs11|vault|file`
  (по умолчанию `pkcs11` — текущие вызовы работают без изменений) и флаги
  Vault/file, единые с выпускающими подкомандами. Инварианты агента
  (127.0.0.1, парный токен, Origin-allowlist, подтверждение оператора) от
  бэкенда не зависят.
- CLI-подкоманды `issue-*`/`csr` получают `--backend file` c флагом
  `--key-file <path>`.
- Документация: подраздел «Ключ в файле» в бэкендах подписи, примеры запуска
  `issuer serve` для всех трёх бэкендов, абзац в threat-model §11 о сниженном
  классе защиты файлового ключа.

Breaking-изменений нет: все флаги аддитивны, дефолты сохраняют текущее
поведение (compat-policy соблюдена).

## Capabilities

### New Capabilities

Нет — файловый бэкенд и выбор бэкенда в serve входят в существующую
capability `issuer-signing`.

### Modified Capabilities

- `issuer-signing`: (1) новое требование «Файловый бэкенд подписи» — формат
  ключа, обязательная проверка прав файла, обращение с паролем и ключом в
  памяти, вывод алгоритма из ключа; (2) требование «Локальный агент issuer
  serve» — агент ДОЛЖЕН (MUST) поддерживать выбор бэкенда подписи из трёх
  адаптеров с сохранением всех гейтов; секрет бэкенда (PIN, пароль ключа,
  токен Vault) остаётся на стороне агента; (3) Purpose — «токен/HSM/Vault»
  дополняется файловым хранилищем.

## Impact

- `crates/tessera_issuer`: новый модуль `file.rs` за cargo-фичей `file`;
  `cli.rs` — `BackendKind::File`, `--key-file`, `--backend` в `ServeArgs`,
  генерализация `run_serve`; `l10n.rs` — новые сообщения RU/EN.
- Зависимости: + `p384`, + прямой `pkcs8` (фичи `encryption`, `pem`) — оба
  RustCrypto, уже в семействе workspace.
- Тесты: e2e CLI-путь без внешних зависимостей (в отличие от gated
  `pkcs11-tests`/`vault-tests`); сценарий Vault-в-serve в `vault-tests`.
- Документация: `docs/{ru,en}/issuer.md`, `docs/{ru,en}/threat-model.md`.
- Кабинет (SPA) не меняется: он говорит с агентом одним протоколом
  независимо от бэкенда.
