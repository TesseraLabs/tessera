# role-selection Delta Specification

## MODIFIED Requirements

### Requirement: Роль, требующая недоступный enforcement

Вход по роли, чей payload требует недоступный в сборке backend, ДОЛЖЕН (MUST) быть отклонён с явной диагностикой — НЕ молчаливое сужение прав (открытая сборка: `mac_mask` без ParsecBackend, `selinux` без SELinux-адаптера).

Linux-payload (`groups`, `sudo_role`, `limits`, `[session]`-лимиты) ДОЛЖЕН (MUST)
**реально применяться** к пользовательской сессии в открытой сборке через
открытый Linux-backend (capability `linux-session-enforcement`), а не только
парситься и валидироваться. Роль, чей payload полностью покрыт доступными
backend'ами (Linux-примитивы — всегда в открытой сборке; `mac_mask` — при
ParsecBackend), ДОЛЖНА (MUST) применяться целиком. Примитив payload, который
не удалось применить, ДОЛЖЕН (MUST) приводить к отказу входа с диагностикой и
audit deny — fail-closed, без частичного молчаливого применения.

#### Scenario: Роль с mac_mask на открытой сборке
- **WHEN** запрошена роль с `mac_mask` на сборке со StubBackend
- **THEN** отказ входа с диагностикой «роль требует МКЦ-enforcement», audit deny

#### Scenario: Роль с группами и sudo на открытой сборке
- **WHEN** запрошена роль с `groups`/`sudo_role`/`limits` на открытой сборке вне Astra
- **THEN** Linux-backend применяет группы, sudo-роль и лимиты к сессии; вход разрешён с полным набором прав роли

#### Scenario: Неуспех применения Linux-примитива
- **WHEN** применение группы/RLIMIT/systemd-лимита из payload завершается ошибкой
- **THEN** отказ входа с типизированной диагностикой и audit deny — без частичного применения

### Requirement: Фиксация payload в сессии и TTL

Сессия ДОЛЖНА (MUST) фиксировать копию payload роли на момент открытия: изменение или новая версия среза в хранилище НЕ влияет на уже открытую сессию.
Снапшот ДОЛЖЕН (MUST) включать полный enforcement-payload роли — `mac_mask`
И Linux-примитивы (`groups`, `sudo_role`, `limits`, `[session]`-лимиты), — а не
только `mac_mask`, чтобы применение в PAM-фазах не перечитывало хранилище.

#### Scenario: Изменение среза после открытия сессии
- **WHEN** срез роли в хранилище изменён или его версия повышена после открытия сессии
- **THEN** сессия продолжает работать на payload-копии момента открытия; новая версия действует со следующего входа

#### Scenario: Снапшот полного payload
- **WHEN** сессия открывается по роли с группами, sudo, лимитами и (на Astra) mac_mask
- **THEN** `SessionRolePayload` фиксирует все эти поля; применение в `setcred`/`open_session` использует снапшот, не хранилище
