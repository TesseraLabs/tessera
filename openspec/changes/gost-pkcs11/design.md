# Design: gost-pkcs11

## Context

Tessera v0.3.19, PKCS#11-путь (режим B): приватный ключ неизвлекаем, подпись `C_Sign` на
устройстве, верификация на хосте публичным ключом из найденного на токене серта
(sign.rs:83–127). Для RSA/ECDSA работает; для GOSTR3410 `select_mechanism` возвращает
типизированный `MechanismNotSupported` с комментарием «cryptoki 0.7 does not expose
CKM_GOSTR3410{,_2012_*}» (mechanism.rs:160–166).

Ограничения: собственная криптография запрещена (верификация — только gost-engine);
`tessera_core` sync; fail-closed на auth-пути; целевое железо — JaCarta-2 GOST
(требует RW-сессию, legacy-экземпляры требуют `pkcs11_locking_mode = mutex`) и
Рутокен ЭЦП 2.0/3.0.

## Goals / Non-Goals

**Goals:**
- Подпись 32-байтового nonce GOST-ключом (2012-256/512) на токене, верификация на хосте.
- Выбор механизма строго по атрибутам ключа и заявленным токеном механизмам (fail-closed).
- Минимальная новая поверхность в зависимостях (supply-chain: cargo deny/audit).

**Non-Goals:**
- GOST R 34.10-2001 (legacy, вне scope — как Ed25519 в challenge-response).
- Генерация/импорт ключей на токен, форматирование токена (это инструменты вендора).
- Извлечение ключа или хост-подпись как fallback (ломает инвариант режима B).
- CI с реальным железом — отдельный change `ci-hardening`.

## Decisions

1. **Лестница разблокировки cryptoki** (порядок предпочтения, выбор финализируется
   первой задачей имплементации):
   - **(1) Upgrade cryptoki** до версии, экспонирующей GOST-механизмы (проверить changelog
     upstream; при отсутствии — оценить стоимость PR в parallaxsecond/rust-cryptoki).
     Плюс: типизированный API, нулевой unsafe у нас. Минус: ждать релиза/ревью upstream.
   - **(2) Vendor-defined-механизм**: численные CKM-константы TC26 прокидываются через
     механизм-обёртку cryptoki (если API допускает произвольный `CK_MECHANISM_TYPE`).
     Плюс: без смены версии. Минус: константы и параметры — на нашей совести.
   - **(3) Raw FFI через `cryptoki-sys`** только для `C_SignInit`/`C_Sign` GOST-ветки,
     остальное — через типизированный cryptoki. Последний резерв: локальный unsafe-островок,
     ревью по чек-листу FFI.
2. **Точные CKM/CKA-константы — по официальным заголовкам вендоров** (Рутокен SDK,
   JaCarta SDK; спецификация TC26 для PKCS#11). В спеку численные значения не зашиваются —
   они фиксируются в коде одним модулем констант с ссылками на источник. Известная вилка:
   «хэш на токене» (`CKM_GOSTR3410_WITH_GOSTR3411_12_*`) vs «хэш на хосте + raw-подпись»
   (`CKM_GOSTR3410`/`_512` по предвычисленному Streebog-дайджесту через gost-engine).
3. **Выбор режима подписи**: предпочтителен «хэш на токене» (меньше нашего кода на криптопути);
   если токен заявляет только raw-механизм — Streebog считается на хосте gost-engine'ом
   (дайджест не секрет, инвариант «приватный ключ не покидает токен» сохраняется).
   Выбор — по `C_GetMechanismList`: ни одного GOST-механизма → `MechanismNotSupported`.
4. **Определение 256 vs 512** — по атрибутам объекта ключа (`CKA_GOSTR3410_PARAMS` /
   `CKA_KEY_TYPE` + длина), сверка с типом публичного ключа из серта (NID 979/980);
   расхождение серт↔ключ → отказ (fail-closed), как и текущая верификация «ключом ИЗ серта».
5. **Верификация на хосте**: публичный ключ из серта, gost-engine, Streebog-256/512 —
   симметрично PKCS#12-пути (challenge/gost.rs). Формат подписи токена (64/128 байт r||s,
   порядок компонент по ГОСТ vs ожидание OpenSSL/gost-engine) — главный известный риск
   несовместимости; фиксируется round-trip тестами «подписал токен → верифицировал engine»
   на реальном железе и зеркальными unit-тестами на перекодировку.
6. **Сессия/локинг без изменений**: RW-сессия и `pkcs11_locking_mode` уже покрывают
   капризы JaCarta-2 GOST; GOST-ветка не вводит собственных режимов.

## Risks / Trade-offs

- **Зоопарк прошивок**: разные ревизии JaCarta/Рутокен заявляют разные наборы GOST-механизмов.
  Митигация: выбор по `C_GetMechanismList` + матрица ручных прогонов по железу в runbook'е.
- **Несовместимость формата подписи** (порядок байт r||s) — ловится только интеграционным
  тестом с железом; до его прогона фичу не объявлять рабочей (критерий приёмки в tasks).
- **Upstream-зависимость** (вариант 1): сроки релиза cryptoki вне нашего контроля; поэтому
  лестница, а не ставка на один вариант.
- **unsafe FFI** (вариант 3): минимизируется площадью (две функции), компенсируется ревью
  и тем, что вызовы идут в уже загруженный вендорский модуль — новых границ доверия нет.

## Open Questions

- Поддерживает ли актуальный cryptoki vendor-defined `CK_MECHANISM_TYPE` без форка —
  проверить на этапе 1.
- Нужна ли поддержка `CKA_ID`-несовпадений у старых JaCarta (серт и ключ с разными ID) —
  пока вне scope, текущий `find_private_key_for_cert` остаётся как есть.
