# token-pkcs11 Delta Specification

## MODIFIED Requirements

### Requirement: Механизмы подписи

Выбор механизма подписи ДОЛЖЕН (MUST) быть таким: RSA → `Sha256RsaPkcsPss` (salt 32, MGF1-SHA256);
EC P-256/P-384 → `EcdsaSha256/384` (raw r||s перекодируется в DER); GOSTR3410 → GOST-механизм
2012-256/512. Для GOST вариант (256/512) ДОЛЖЕН (MUST) определяться по атрибутам объекта ключа
(`CKA_GOSTR3410_PARAMS` / тип+длина) и сверяться с типом публичного ключа из серта (NID 979/980) —
расхождение серт↔ключ → отказ. Предпочтителен механизм «хэш на токене»
(`CKM_GOSTR3410_WITH_GOSTR3411_2012_*`-семейство); если токен заявляет только raw-механизм —
Streebog-дайджест считается на хосте через gost-engine; если `C_GetMechanismList` не содержит
ни одного GOST-механизма → `MechanismNotSupported`. Численные CKM/CKA-константы фиксируются
в коде одним модулем со ссылками на источники (Рутокен SDK / JaCarta SDK / TC26).
Верификация — публичным ключом ИЗ серта, не из заявленного токеном (sign.rs:119–125);
для GOST — через gost-engine.

#### Scenario: GOST-ключ на токене
- **WHEN** ключ токена имеет тип GOSTR3410 и токен заявляет GOST-механизм
- **THEN** выбирается GOST-механизм 2012-256/512 по атрибутам ключа; подпись выполняется `C_Sign` на устройстве

#### Scenario: Расхождение варианта ключа и серта
- **WHEN** атрибуты ключа токена дают 2012-512, а публичный ключ серта — NID 979 (2012-256)
- **THEN** отказ (fail-closed) — ключ не является парным сертификату

#### Scenario: Токен без GOST-механизмов
- **WHEN** `C_GetMechanismList` токена не содержит GOST-механизмов
- **THEN** `MechanismNotSupported` → отказ; молчаливый fallback на хост-подпись запрещён
