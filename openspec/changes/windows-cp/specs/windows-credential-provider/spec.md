# windows-credential-provider Specification

## Purpose

UI-слой Windows-адаптера: Credential Provider V2 (COM DLL в LogonUI) —
тайл входа Tessera, выбор метода и роли, сбор PIN, обмен с TCB-службой и
сериализация учётных данных технической УЗ строго после вердикта службы.
Enforcement-логики в CP нет.

## ADDED Requirements

### Requirement: Тайл и поля LogonUI

CP ДОЛЖЕН (MUST) реализовывать Credential Provider V2 и декларировать поля
тайла: выбор метода (комбобокс: серт-носитель; позиция метода «код»
зарезервирована и неактивна до его портирования), выбор роли (комбобокс),
ввод PIN (парольное поле), статусный текст. Отрисовку выполняет LogonUI;
CP собственный UI рисовать НЕ ДОЛЖЕН (MUST NOT). Дефолтной роли НЕ ДОЛЖНО
(MUST NOT) быть — роль выбирается инженером явно; порядок ролей в комбобоксе —
по `level` (только порядок отображения, не сравнение прав).

#### Scenario: Тайл при валидной базе ролей
- **WHEN** LogonUI строит тайл Tessera на подготовленном устройстве
- **THEN** комбобокс ролей заполнен ролями от службы (имя из `name`, порядок по `level`), роль по умолчанию не предвыбрана

#### Scenario: Тайл на domain-joined устройстве
- **WHEN** подготовленное устройство включено в домен и доменные политики не исключают провайдера
- **THEN** тайл Tessera строится и работает идентично устройству вне домена; членство в домене само по себе не меняет поведение CP

#### Scenario: Метод «код» до портирования
- **WHEN** инженер открывает комбобокс метода
- **THEN** позиция «код» присутствует, но неактивна; активен метод серт-носителя

### Requirement: Ноль enforcement и локальных данных в CP

CP НЕ ДОЛЖЕН (MUST NOT) содержать enforcement-логики и вызовов управления
токенами (`LsaLogonUser`, `SetTokenInformation`, `CreateRestrictedToken`,
`AdjustTokenPrivileges`), НЕ ДОЛЖЕН (MUST NOT) читать локальные данные
Tessera (base ролей, конфигурацию, журнал) и НЕ ДОЛЖЕН (MUST NOT) выполнять
криптографическую верификацию доверия. Единственные источники данных CP —
ввод инженера и IPC-ответы TCB-службы.

#### Scenario: Перечень ролей — только от службы
- **WHEN** CP заполняет комбобокс ролей
- **THEN** данные получены сообщением CP-флоу от службы; файлы role-store из CP не читаются

### Requirement: Вердикт службы до сериализации

CP ДОЛЖЕН (MUST) передавать службе материал для аутентификации (данные
носителя, выбранную роль, PIN) и выполнять сериализацию учётных данных
(`GetSerialization`) только после вердикта-допуска. При отказе или
недоступности службы сериализация выполняться НЕ ДОЛЖНА (MUST NOT) —
CP возвращает «нет учётных данных» со статусным текстом. Секрет технической
УЗ ДОЛЖЕН (MUST) поступать одноразово в вердикте допуска и зануляться сразу
после сериализации; персистентного секрета в CP или его конфигурации быть
НЕ ДОЛЖНО (MUST NOT). PIN ДОЛЖЕН (MUST) зануляться немедленно после передачи
службе.

#### Scenario: Успешный вход
- **WHEN** служба ответила допуском на запрос аутентификации
- **THEN** CP сериализует учётные данные технической УЗ для стандартного пути LogonUI; одноразовый секрет зануляется после сериализации

#### Scenario: Отказ верификации
- **WHEN** служба ответила отказом (revocation/binding/challenge/роль)
- **THEN** сериализация не происходит; тайл показывает статус отказа без деталей, пригодных для перебора

#### Scenario: Служба недоступна
- **WHEN** pipe недоступен или версия протокола не совпала
- **THEN** вход методом Tessera невозможен (fail-closed); тайлы стандартных провайдеров Windows работают как обычно

### Requirement: Устойчивость кода в LogonUI

Каждая COM-точка входа CP ДОЛЖНА (MUST) быть закрыта panic-guard'ом
(аналог PAM-канона): паника внутри CP НЕ ДОЛЖНА (MUST NOT) обрушивать
LogonUI или мешать стандартным провайдерам. Деградация всегда локальна:
неисправен только тайл Tessera.

#### Scenario: Паника внутри CP
- **WHEN** в обработчике поля CP возникает panic
- **THEN** panic-guard гасит её, тайл Tessera переходит в состояние ошибки; стандартный вход продолжает работать

### Requirement: Дистрибуция и регистрация

Поставка ДОЛЖНА (MUST) быть единым MSI-пакетом: TCB-служба, CP DLL, CLI;
установка регистрирует CLSID CP и запускает `prepare`, деинсталляция —
дерегистрирует CLSID, останавливает и удаляет службу (журнал сохраняется).
Все исполняемые артефакты и MSI ДОЛЖНЫ (MUST) быть подписаны (Authenticode)
по supply-chain требованиям существующего канала поставки.

#### Scenario: Деинсталляция возвращает штатный вход
- **WHEN** MSI деинсталлирован
- **THEN** CLSID дерегистрирован, тайл Tessera исчез, служба удалена, журнал сохранён; стандартный вход Windows не затронут

#### Scenario: Неподписанный артефакт
- **WHEN** релизный pipeline собирает MSI
- **THEN** сборка без валидной подписи всех артефактов не публикуется
