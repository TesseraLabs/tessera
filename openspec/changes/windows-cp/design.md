# Design: windows-cp

## Context

Канон Windows-адаптера: CP — чистый UI/сериализация, ноль enforcement
(подтверждено research-проходом №2 исходниками multiOTP CP и pGina: ноль
вызовов LsaLogonUser/SetTokenInformation/CreateRestrictedToken внутри CP).
Модель UI Windows решает проблему выбора роли на логине нативно: CP
декларирует поля (`CREDENTIAL_PROVIDER_FIELD_TYPE`), LogonUI рендерит их
сам — «Credential providers are not responsible for drawing their own UI».
Engine-сторона (верификация, вердикт, enforcement) — `windows-tcb-service`;
обмен — named pipe + сообщения CP-флоу (delta `ipc-protocol` там же).

## Goals / Non-Goals

**Goals:**

- Контракт CP: тайл, поля (метод, роль, PIN), взаимодействие со службой,
  сериализация только после вердикта, fail-closed деградация.
- Жёсткая граница: в CP нет ни enforcement, ни чтения локальных данных
  Tessera, ни криптографии доверия — всё в службе.
- Дистрибуция: MSI с обоими компонентами, подпись, регистрация.

**Non-Goals:**

- Метод «код» (qr-login) на Windows — позиция в комбобоксе резервируется,
  реализация — отдельный change после портирования основного пути.
- Credential provider filter (скрытие стандартных провайдеров, lockdown) —
  отдельное будущее решение; первая волна не меняет доступность
  стандартных методов входа.
- Unlock-сценарии RDP/удалённого доступа — вне скоупа этого change.
- AD/доменные устройства.

## Decisions

### D1. CP V2 как COM DLL, логика — в Rust

CP реализует `ICredentialProvider` + `ICredentialProviderCredential2` (V2 —
корректная работа с user tiles Windows 8+). Обвязка COM — на Rust
(`windows`-крейт генерирует COM-интерфейсы); C++-шелл — запасной вариант,
если COM-реализация V2-интерфейсов на Rust упрётся в грабли (решается в
первой задаче; предпочтение Rust — единый язык, единые паник-гарантии,
класс риска кода в LogonUI тот же T3/T6). Вся не-COM логика (протокол,
состояние полей) — обычный Rust-модуль с unit-тестами без COM.

### D2. Поля тайла

| Поле | Тип | Содержимое |
|---|---|---|
| Метод | `CPFT_COMBOBOX` | «Носитель (сертификат)» \| «Код» (позиция зарезервирована, disabled до портирования qr-login) |
| Роль | `CPFT_COMBOBOX` | Роли от службы: имя (`name`) + упорядочение по `level` (порядок в UI, не сравнение прав) |
| PIN | `CPFT_PASSWORD_TEXT` | PIN носителя; в память CP попадает только на время обмена, зануляется после |
| Статус | `CPFT_SMALL_TEXT` | Диагностика (носитель не найден / служба недоступна / отказ) |

Перечень ролей запрашивается у службы при построении тайла и обновляется
при событиях CP (`ICredentialProviderEvents`); дефолтной роли нет —
инженер выбирает явно (инвариант role-selection: «без дефолта»).

### D3. Последовательность входа: вердикт до сериализации

1. Инженер выбирает метод и роль, вводит PIN → CP шлёт службе запрос
   аутентификации (материал носителя + роль + PIN-подтверждение владения —
   challenge-response выполняет служба/ядро, не CP).
2. Служба верифицирует и отвечает вердиктом.
3. Только при допуске CP выполняет `GetSerialization` →
   `KERB_INTERACTIVE_UNLOCK_LOGON` (техническая УЗ) — дальше стандартный
   путь LogonUI/Winlogon; параметры сессии (роль, TTL) уже зафиксированы
   службой на её стороне.
4. Отказ → `CPGSR_NO_CREDENTIAL_NOT_FINISHED` + статус-текст; сериализация
   не происходит.

Секрет технической УЗ, которым CP сериализует логон, CP получает от службы
в вердикте допуска (одноразово, зануляется после сериализации) — постоянного
секрета в CP/реестре нет; хранение и ротация секрета УЗ — обязанность службы
(зафиксировано в спеке как требование, конкретная механика ротации — деталь
реализации `windows-tcb-service`).

Примечание: если spike D2 из `windows-tcb-service` выберет вариант
«authentication package», шаг 3 сериализует пакет для нашего AP вместо
`KERB_INTERACTIVE_UNLOCK_LOGON` — контракт CP (вердикт до сериализации,
ноль enforcement) не меняется.

### D4. Fail-closed без вмешательства в чужие тайлы

Служба недоступна / pipe отклонил / версия протокола не совпала → тайл
Tessera показывает статус и не сериализует; стандартные провайдеры Windows
работают как обычно. Готовность устройства (device readiness) и
lockdown-политики «только Tessera» — осознанно вне первой волны: до них
CP не должен уметь блокировать чужой вход.

### D5. Дистрибуция: один MSI на оба компонента

MSI (WiX): служба + CP DLL + CLI; регистрация CLSID CP
(`HKLM\...\Authentication\Credential Providers\{CLSID}`); Authenticode-подпись
DLL/EXE/MSI (WHQL не нужен — CP не драйвер); post-install — `prepare`,
uninstall — дерегистрация CLSID и остановка службы с зачисткой (журнал
сохраняется). Канал поставки и подпись — те же supply-chain требования,
что для существующих пакетов (T5).

## Risks / Trade-offs

- [Код в LogonUI: краш CP валит вход на устройстве целиком] → panic-guard на
  каждой COM-точке входа (аналог PAM-канона), деградация «тайл не работает,
  стандартный вход жив»; никакого паникующего кода на пути чужих тайлов.
- [Секрет технической УЗ проходит через память CP] → одноразовая выдача в
  вердикте, zeroize сразу после `GetSerialization`, никакой персистенции в
  CP; альтернатива (CP знает постоянный секрет) отвергнута как размазывание
  границы доверия.
- [COM на Rust для CP V2 — меньше протоптанных троп, чем C++] → первая
  задача — walking skeleton тайла; при граблях — C++-шелл (D1), решение
  фиксируется до основной волны.
- [Комбобокс ролей до аутентификации раскрывает имена ролей устройства
  любому у консоли] → осознанно: имена ролей — не секрет (симметрично
  Linux-prompt); чувствительные имена — вопрос гигиены наполнения role-store.
- [На domain-joined устройстве доменная политика может исключить наш CP
  (политика исключения провайдеров) или заблокировать DLL (WDAC/AppLocker) —
  тайл молча исчезает] → детект — `doctor` (windows-tcb-service, domain-чеки);
  сам CP этому противостоять не может и не должен (домен-админ — владелец
  устройства, вне модели угроз); задача Tessera — диагностируемость, не
  сопротивление.

## Migration Plan

Новая платформа — миграции нет. Установка/удаление — MSI (D5); rollback =
деинсталляция, стандартный вход Windows не затрагивается ни в какой момент.

## Open Questions

- Механика выдачи/ротации секрета технической УЗ службой (одноразовый
  секрет в вердикте — контракт; генерация/ротация — при реализации
  `windows-tcb-service`).
- Локализация строк тайла (RU/EN) — по существующей практике продукта на
  момент реализации; в схеме роли локализации нет (YAGNI, `name` — одна
  строка).
