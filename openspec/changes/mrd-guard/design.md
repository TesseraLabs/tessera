# mrd-guard — design

## Context

Astra SE несёт два независимых мандатных механизма PARSEC: МКЦ (целостность,
Biba) и МРД (конфиденциальность, Белла—ЛаПадулы; активен на уровне защищённости
«Смоленск»). Tessera работает только с осью целостности: `IntegrityLabel`,
`mac_mask` роли, потолок `pam_cert_max_integrity`.

Текущее состояние:

- libpdp записывает parsec-метку только целиком (обе оси одним триплетом),
  частичной записи одной оси в API нет — поэтому контракт обращения с осью
  конфиденциальности при записи метки должен быть зафиксирован явно, до сих
  пор его не было;
- `startup_check` различает только «kernel parsec активен / нет»
  (`KernelParsecState` через `tessera_mac_parsec::probe_runtime()`), ось
  конфиденциальности не видит.

Текстовый формат триплета известен и проверен C-репродьюсером на Astra 1.8.4.

## Goals / Non-Goals

**Goals:**

- Зафиксировать контракт: системы с активным МРД не поддерживаются в v0.x;
  поведение на них — определённое и безопасное, а не случайное.
- Guard №1: startup-check `mac_mrd_active` — оператор узнаёт о неподдерживаемой
  конфигурации до боевой эксплуатации (ERROR при `runtime=required`).
- Guard №2: любые записи меток (процесс, файл, fd) сохраняют существующее
  поле конфиденциальности — даунгрейд невозможен даже там, где детект не
  сработал.

**Non-Goals:**

- Назначение/выбор уровня конфиденциальности сессии (МРД как фича) — SPI
  остаётся осью целостности, `IntegrityLabel` не меняется.
- Поддержка уровня защищённости «Смоленск» как платформы.
- UI выбора уровня секретности, расширения сертификата для секретности.

## Decisions

1. **Probe МРД по образцу `KernelParsecProbe`.** Enum
   `MrdState { Active, Inactive, Unknown }` живёт в `tessera_core::mac`
   (рядом с `MacRuntime` — коммерческий crate должен возвращать этот тип,
   а на `tessera_cli` он не завязан); в открытом `startup_check` — тип
   probe-функции `MrdProbe` и `real_mrd_probe()`: за feature `astra-mac`
   вызов `tessera_mac_parsec::probe_mrd()`, иначе `Unknown`. Альтернатива —
   расширить `MacBackend` — отвергнута: probe нужен до построения
   backend'а, и существующий паттерн (`real_kernel_parsec_probe`) уже
   отделяет runtime-детект от SPI.

2. **Матрица политики** (симметрична существующему `mac_runtime`-чеку):
   `required`: Active → ERROR (демон не стартует), Unknown → WARN,
   Inactive → INFO; `auto`: Active → WARN, иначе INFO; `disabled`: всегда
   INFO. Обоснование ERROR только при `required`: это режим строгих
   инвариантов; при `auto`/`disabled` установка либо деградирует осознанно,
   либо не пишет метки вовсе.

3. **Способ детекта — внутри коммерческого crate, подтвердить на Astra VM.**
   Кандидаты в порядке предпочтения: запрос libpdp (максимальный/текущий
   уровень конфиденциальности системы), чтение состояния parsec из
   sysfs/procfs, exec `astra-mac-control status` (последний — нежелателен:
   внешний бинарь в auth-инфраструктуре). До подтверждения на VM API
   probe'а фиксируем, реализацию — нет.

4. **Сохранение поля 0 — read-modify-write.** `apply_session` /
   `set_file_label` / `set_fd_label` перед записью читают текущую метку
   цели (`pdp_get_pid` / `pdp_get_lpath` / `pdp_get_fd` — публичный
   libpdp-API) и переносят её поле 0 в новую метку. Альтернатива —
   частичная запись только оси целостности — в libpdp-API отсутствует
   (запись только триплетом). `IntegrityLabel` в открытом ядре не трогаем:
   конфиденциальность — деталь коммерческого backend'а, SPI-контракт
   формулируем как «MUST NOT изменять поле конфиденциальности».

5. **Границы репозиториев.** Открытый репо: enum/probe-тип в
   `startup_check`, сам чек с матрицей, контракт в спеке `mac-integrity`,
   абзац в `docs/mac-integrity.md`. Коммерческий (`tessera-enterprise`,
   отдельный трек): `probe_mrd()`, правки кодека и FFI. Тексты в открытом
   репо — только факты (никаких рыночных обоснований).

## Risks / Trade-offs

- [Детект МРД различается между версиями Astra 1.7/1.8] → реализация после
  проверки на VM; `Unknown` — безопасный fallback (WARN при `required`).
- [Read-modify-write не атомарен] → приемлемо: метка собственного процесса
  меняется в момент открытия сессии (до exec шелла), файловые метки — при
  провижининге; конкурентного писателя меток в этих точках нет.
- [ERROR при `required` на МРД-машине ломает старт демона] → намеренно:
  fail-closed вместо неопределённого поведения; оператор явно выбирает
  `auto`/`disabled`, принимая WARN.
- [Поле 0 из «мусорной» текущей метки переносится в новую] → на не-МРД
  системах поле 0 всегда 0 — поведение не меняется; на МРД-системах
  перенос и есть цель.

## Sequencing (2026-07-05)

Группа 2 (открытый репо) реализована без VM: `real_mrd_probe` в открытой
сборке возвращает `Unknown`, семантика детекта отложена. Группа 3
(коммерческий crate) идёт следом: (а) детект МРД ждёт C-репродьюсера на VM
(задача 1.1); (б) коммерческая сборка требует версии открытого ядра,
уже содержащей `MrdState`, — `probe_mrd()` добавляется после её выхода.

## Open Questions

- Точный API libpdp для детекта активного МРД и чтения текущей метки
  процесса/файла — подтвердить C-репродьюсером на Astra VM (1.8.4),
  как делали для текстового формата.
- Нужен ли отдельный audit-код (`mac.audit`) для события «поле 0 сохранено
  ненулевым» — полезно для диагностики на неподдерживаемых системах.
