# Delta: role-selection

## MODIFIED Requirements

### Requirement: Явный выбор роли, без дефолта

Вход ДОЛЖЕН (MUST) требовать явного указания роли; дефолтной роли НЕ ДОЛЖНО
(MUST NOT) существовать. Без указания роли вход НЕ ДОЛЖЕН (MUST NOT)
продолжаться.

**Целевой способ указания — имя ролевой учётной записи** (модель «роль =
ролевая учётная запись», принята 2026-06-19): инженер входит в ролевую УЗ с
именем роли (`oper@host`, включая `ssh oper@host`); запрошенная роль равна
имени УЗ входа. Допуск проверяется расширением `pam_cert_user_binding`
(список разрешённых ролевых УЗ — см. `cert-scope-binding`); личность
инженера — в сертификате и журнале выпусков, не в учётной записи. Ролевые УЗ
провижинятся Census (вход в них закрыт всеми путями, кроме
cert-аутентификации Tessera).

Суффикс `<user>+<role>` и текстовый PAM-prompt — **депрекейт-механизмы
совместимости**: они НЕ ДОЛЖНЫ (MUST NOT) подаваться в документации и
интерфейсах как основной способ. KNOWN GAP: код v0.4.x реализует только
суффикс/prompt — перепроводка role-selection на модель ролевых УЗ (≈8 точек,
CVE-2021-3560-чувствительная) — отдельный высокорисковый change, не
выполнен; до него суффикс остаётся рабочим механизмом.

Нативные UI (GDM choice-list, fly-dm kgreeterplugin, Windows CPFT_COMBOBOX) —
опциональный слой поверх, их отсутствие не влияет на работоспособность.

#### Scenario: Вход в ролевую учётную запись (целевой)
- **WHEN** инженер подключается `ssh oper@device`, предъявляя лист, чей `user_binding` содержит `oper`
- **THEN** запрошенная роль = `oper` (имя УЗ входа), допуск и активация роли валидируются атомарно с аутентификацией; отдельного указания роли не требуется

#### Scenario: Ролевая УЗ вне user_binding
- **WHEN** инженер подключается `ssh admin@device`, а `user_binding` листа содержит только `oper` и `serv`
- **THEN** вход отклоняется (`UserNotAllowed`), audit deny

#### Scenario: Логин без роли в окружении без conversation (депрекейт-путь)
- **WHEN** инженер вводит персональное имя без суффикса и PAM-стек не может показать prompt
- **THEN** вход отклоняется с диагностикой «роль не указана», audit deny

#### Scenario: Выбор по суффиксу через SSH (депрекейт-путь)
- **WHEN** инженер подключается `ssh ivanov+serv@device`
- **THEN** запрошенная роль = `serv`, PAM_USER канонизируется в `ivanov` (KNOWN GAP: путь жив до перепроводки)

### Requirement: Канонизация PAM_USER

Депрекейт-механизм совместимости (см. KNOWN GAP выше). Пока суффиксный путь
жив, модуль ДОЛЖЕН (MUST) парсить суффикс в начале `pam_sm_authenticate` и
переписывать PAM_USER на каноническое имя (`pam_set_item`) ДО любых обращений
к учётной записи и до остальных модулей стека. Символ `+` ДОЛЖЕН (MUST) быть
запрещён в канонических именах учётных записей (проверка при провижининге).
Синтаксические ошибки (`ivanov+`, `ivanov+a+b`, пустое имя, role_id вне
`^[a-z][a-z0-9-]{0,15}$`) — отказ до аутентификации. В аудит ДОЛЖНЫ (MUST)
идти каноническое имя и запрошенная роль отдельными полями. В целевой модели
(вход в ролевую УЗ) канонизация не применяется: PAM_USER уже каноничен и
равен имени роли.

#### Scenario: Двойной суффикс
- **WHEN** введено `ivanov+a+b`
- **THEN** синтаксическая ошибка, отказ до аутентификации, audit deny с сырой строкой

#### Scenario: Пустой суффикс
- **WHEN** введено `ivanov+`
- **THEN** синтаксическая ошибка, отказ до аутентификации

### Requirement: Резолв и покрытие — атомарно с аутентификацией

Резолв роли, валидация удостоверения и проверка покрытия ДОЛЖНЫ (MUST) выполняться
одной неразрывной стадией в `pam_sm_authenticate` — между валидацией
удостоверения и фиксацией роли в сессии НЕ ДОЛЖНО (MUST NOT) быть точки подмены роли
(урок polkit CVE-2021-3560). Покрытие: для сертификата — запрошенная роль ∈ списку расширения
allowed-roles (membership, без сравнения уровней); для кода — `role_id` входит во вход MAC
(контракт Tessera Codes). В целевой модели запрошенная роль равна имени ролевой
УЗ входа, и покрытие дополнительно гарантировано допуском `user_binding`
(значения совпадают на выпуске). Роль не найдена в базе / не покрыта
удостоверением — отказ, audit deny с (инженер, роль, причина).

#### Scenario: Роль не покрыта сертификатом
- **WHEN** запрошена `admin`, а allowed-roles серта = `[oper, serv]`
- **THEN** отказ входа, audit deny (membership-проверка, никакого «уровень выше — пропустим»)

#### Scenario: Роль отсутствует в базе устройства
- **WHEN** запрошена роль, которой нет в role-store
- **THEN** отказ «роль неизвестна»; в депрекейт-prompt-режиме допускается повторный выбор (в целевой модели повторный выбор = новая попытка входа в другую ролевую УЗ)
