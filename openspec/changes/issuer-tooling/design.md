# Design: issuer-tooling

## Context

Engine проверяет custom-расширения Tessera (host-binding/user-binding, теги, рамки
делегирования — арка `2.25.<UUID>`), но инструмента, который такие сертификаты
выпускает, в репозитории нет. Выпуск через openssl-конфиги хрупок: расширения
собираются руками, монотонность рамок никто не проверяет до попытки входа,
серийники и CRL требуют дисциплины оператора.

Дизайн-основа — brainstorming 2026-07-05 (tessera-ws). Ключевая правка
относительно исходного дизайна: **кабинет один**, а не два (владелец парка /
обслуживающая организация) — доступные операции выводятся из предъявленного
родительского сертификата.

Ограничения:
- Инструмент не кастодиан: приватные ключи живут в токене/HSM/Vault и не
  экспортируются.
- Веб-кабинет — статика без серверной логики: работает self-hosted и офлайн,
  весь стейт — файлы.
- Семантика расширений уже зафиксирована main-спеками (`cert-scope-binding`,
  `device-tags`, `revocation`, `trust-chain-validation`) — выпуск обязан
  порождать ровно то, что Engine принимает.

## Goals / Non-Goals

**Goals:**
- Одно Rust-ядро выпуска (`tessera_issuer`): сборка TBS, проверки монотонности
  рамок на стороне выпуска, случайные серийники, выпуск CRL.
- Единый интерфейс подписи готового TBS; адаптеры PKCS#11 (токен/HSM) и
  Vault/OpenBao Transit.
- Обёртки: CLI `issuer` (автоматизация, тикет-системы) и один статический
  веб-кабинет (WASM-сборка того же ядра); мост браузер ↔ токен — локальный
  агент `issuer serve`.
- Локальный подписанный hash-chain журнал операций выпуска.

**Non-Goals:**
- Курьерский payload на носителе (CRL/конфиги/deb) — отдельный change
  `courier-payload`.
- Серверная сторона: снапшоты инвентаря из Control, four-eyes, приём журналов
  выпуска — вне репозитория и вне scope.
- ГОСТ-fallback через браузерные плагины (Рутокен Плагин, КриптоПро ЭЦП Browser
  plug-in) — до подтверждения, что они умеют подпись произвольного TBS (raw),
  а не только CMS/документов.
- Изменения Engine/агента на устройстве — их спеки остаются как есть.

## Decisions

### D1. Один крейт `tessera_issuer`, ядро wasm-совместимое

Новый крейт в workspace: lib (ядро) + bin `issuer` (CLI и `serve`).
Ядро — sync, без tokio (паритет с `tessera_core`), собирается в wasm32:
сборка TBS, DER-кодирование расширений, проверки монотонности, CRL, журнал.
Адаптеры PKCS#11 и Vault Transit — за feature-флагами, в wasm не собираются
(в браузере подпись уходит в локальный агент по HTTP).

Альтернатива «отдельный репозиторий» отклонена: определения OID-расширений и
семантика рамок должны эволюционировать вместе с Engine, в одном PR.

### D2. Общие определения расширений — новый крейт `tessera_ext`

OID-константы, DER-схемы расширений и предикат монотонного сужения рамок
выносятся в новый крейт `tessera_ext` (pure Rust, без openssl/cryptoki/nix,
собирается под wasm32), чтобы проверка на стороне выпуска и enforcement Engine
были одним кодом, а не копией. Состав: OID-константы (бывший `x509/oids.rs`),
pure-Rust DER-энкодер OID для арок `2.25.<u128>` (замена openssl-канонизации
в `der_helpers.rs`), минимальный DER TLV reader/writer, структуры расширений
(включая `DelegationConstraints`) и предикат монотонного сужения рамок.
`tessera_core` переходит на `tessera_ext` без изменения поведения и публичного
API (re-export'ы по старым путям).

Вариант «фичегейт в `tessera_core`» отклонён при осмотре кода: крейт жёстко
зависит от openssl (типы `Certificate`/`VerifiedX509`) и системных крейтов —
в wasm32 он не собирается ни в какой конфигурации фич.

### D3. Подпись: trait «подписать готовый TBS ключом слота»

Единый trait (вход: байты TBS, идентификатор ключа; выход: подпись + алгоритм).
Реализации:
- **PKCS#11** (cryptoki): токены и HSM — один код; ГОСТ-токены работают через
  тот же адаптер, если токен отдаёт PKCS#11-механизм.
- **Vault/OpenBao Transit**: подпись готового TBS по HTTP. Vault PKI НЕ
  используется: Go `encoding/asn1` не парсит OID-дуги больше int64 — наши
  расширения арки `2.25.<UUID>` через Vault PKI не выпустить.
- **Remote (браузер)**: WASM-ядро шлёт TBS локальному агенту `issuer serve`,
  тот подписывает через PKCS#11-адаптер.

Ядро не знает, где ключ; никакой ключевой материал через интерфейс не проходит.

### D4. Браузер ↔ токен: локальный агент, не браузерный плагин

`issuer serve` — HTTP-сервер строго на 127.0.0.1 (прецедент Fortify):
- allowlist Origin'ов (origin кабинета + self-hosted значения из конфига),
  запросы без/с чужим Origin отвергаются — защита от CSRF/DNS rebinding;
- парный токен сессии: кабинет получает его из вывода `issuer serve`
  (оператор копирует), агент требует его в каждом запросе;
- PIN токена вводится на стороне агента (терминал/pinentry), через браузер
  НЕ передаётся;
- **подтверждение каждой операции**: агент разбирает TBS (`tessera_ext`),
  показывает оператору тип/subject/срок/рамки в своём интерфейсе и подписывает
  только после явного согласия — агент является доверенным дисплеем; токен
  аутентифицирует кабинет, подтверждение авторизует операцию (защита от
  зловредного локального софта и подменённого SPA даже при краже токена);
- **канал подтверждения/PIN** — pinentry-диалог (Assuan, прецедент gpg-agent),
  когда доступен; терминальный prompt — fallback. Это отвязывает доверенный
  канал от способа запуска агента;
- **модель процесса** — user-scope, НИКОГДА не системный сервис под root
  (PKCS#11 — пользовательский ресурс). Базовый режим — запуск на время сессии
  выпуска (меньше окно атаки: вне сессии стучаться некуда); user-scope
  демонизация (`systemd --user` autostart + pairing-токен файлом 0600 в
  `$XDG_RUNTIME_DIR` вместо вывода в терминал) — поддерживаемый режим для
  постоянных рабочих мест диспетчеров.

Свой браузерный плагин отклонён: зоопарк браузеров (включая Chromium-Gost)
делает поддержку несоразмерной задаче. WebCrypto не даёт PKCS#11.

### D5. Один кабинет; режим — из родительского сертификата

Кабинет один (один SPA, одна сборка). После загрузки родительского сертификата
кабинет читает его BasicConstraints и рамки делегирования:
- корень парка → доступен выпуск CA организаций (с назначением рамок);
- CA организации → доступен выпуск листов смен строго внутри рамок этого CA;
- лист/непригодный сертификат → операции выпуска недоступны, с объяснением.

Формы и валидация — одни и те же; различие только в целевом профиле
сертификата. Это убирает дублирование UI и рассинхронизацию проверок.

### D6. Серийные номера — случайные 128-битные

RFC 5280 позволяет до 20 октетов; берём 128 бит CSPRNG (положительное число,
старший бит гасится). Несколько независимых операторов выпускают без
координации и общего состояния — коллизии статистически исключены.

### D7. CRL — клиентская операция; `crlNumber` монотонен per-CA

Выпуск CRL — та же операция подписи ядром: список отозванных серийников из
стейт-файлов + `crlNumber` строго больше последнего выпущенного (хранится в
стейте CA и фиксируется в журнале). Backstop против «зажатого» CRL — короткие
TTL листов смен (существующая семантика `revocation`), поэтому редкая ручная
генерация приемлема.

### D8. Журнал выпусков — hash-chain, подпись доступным ключом

Каждая операция (выпуск листа, выпуск CA, выпуск CRL) — запись с хэшем
предыдущей (genesis — фиксированный); периодическая подпись головы цепочки
через тот же trait подписи (D3). Формат записей — паттерн `logging-audit`
(NDJSON, монотонные seq). Журнал вторичен: первичная правда — аудит входов
на устройствах; журнал служит инвентаризации и разбору инцидентов выпуска.

### D9. Инвентарь для форм — подписанный снапшот файлом

Формы кабинета (выбор устройств/тегов/ролей) кормятся снапшотом-файлом
(подписанный экспорт или заполненный вручную). Источник сменный, форма одна;
живой источник при связности — вне scope этого change (серверная сторона).
Кабинет обязан показывать происхождение и возраст снапшота.

### D10. CSR-режим — равноправный источник ключа листа

Помимо прямой передачи публичного ключа (SPKI) оператором, лист выпускается
по CSR (PKCS#10): инженер генерирует ключ на своём токене, отдаёт оператору
CSR; ядро проверяет самоподпись (proof-of-possession) до выпуска. Атрибуты
CSR — только advisory (prefill форм с пометкой), скоуп задаёт исключительно
оператор — иначе CSR стал бы каналом «инженер сам запросил себе шире».
Оба режима равноправны и не исключают друг друга. Разбор и верификация CSR —
pure Rust (x509-cert `CertReq`, верификация P-256/RSA), wasm-совместимо.
CSR-режим снимает часть открытого вопроса №3: в нём кабинету не нужно право
записи на токен инженера — сертификат возвращается файлом.

### D11. Сторона выпуска — кроссплатформенная (Linux / macOS / Windows)

Инструменты выпуска работают там, где сидит диспетчер; enforcement на
устройствах остаётся Linux-only. Ядро уже pure Rust; `cryptoki` грузит
`.so`/`.dylib`/`.dll`; `tiny_http`/`ureq` кроссплатформенны. Абстрагируются:

| Механизм | Linux | macOS | Windows |
|---|---|---|---|
| Рантайм-каталог токена | `$XDG_RUNTIME_DIR/tessera-issuer/` | `~/Library/Application Support/tessera-issuer/` (0700) | `%LOCALAPPDATA%\tessera-issuer\` |
| Защита токен-файла | права 0600 | права 0600 | дефолтные ACL профиля пользователя (каталог не расшарен) |
| Автостарт (опция) | `systemd --user` unit | launchd LaunchAgent plist | Task Scheduler (логон-триггер) |
| Подтверждение/PIN | pinentry → терминал | pinentry-mac → терминал | pinentry (Gpg4win) → терминал |

В `tessera_issuer` не должно появиться unix-only зависимостей (`nix`/`rustix`)
вне `#[cfg(unix)]`-гейтов; unix-права — через `std::os::unix`, Windows-ветка
полагается на ACL пользовательского профиля. CI собирает и тестирует
`tessera_issuer` (+`tessera_ext`) на ubuntu/macos/windows-раннерах — остальной
workspace (PAM/udev) на Windows не собирается и не должен.

### D12. Локализация — RU/EN без i18n-фреймворка

Операторские поверхности (сводка подтверждения, сообщения serve, вывод CLI,
SPA кабинета) локализуются на русский и английский. Механика — компактная
таблица строк (enum-ключи → две локали) в `tessera_issuer` и словарь в SPA;
никакого fluent/gettext — для двух локалей и десятков строк это лишние
зависимости и процессы. Локаль: конфиг → `TESSERA_ISSUER_LANG` → `LANG` →
fallback EN; в кабинете — переключатель + язык браузера. Технические
идентификаторы (role_id, OID, RFC 4514 subject, поля протокола) не
переводятся. Доки уже двуязычные (docs/{ru,en}).

## Risks / Trade-offs

- [Проверки монотонности задублируются с Engine] → D2: один разделяемый
  предикат; contract-тесты «что выпустил issuer — принял Engine» в CI.
- [Локальный агент = новая поверхность на 127.0.0.1] → Origin-allowlist +
  парный токен + PIN только на стороне агента; дополнение threat-model —
  задача этого change.
- [Подмена статики кабинета при хостинге] → SRI/фиксированные хэши артефакта
  в релизе, self-hosted как первоклассный сценарий; подпись ключом не
  требуется — целостность артефакта проверяется на уровне дистрибуции.
- [Подменённый SPA или зловредный локальный процесс подсовывает агенту иной
  TBS (в пределах рамок)] → закрыто D4: обязательное подтверждение операции
  оператором на стороне агента с показом разобранных полей TBS перед PIN;
  дополнительно ущерб ограничен монотонностью рамок.
- [`crlNumber` при нескольких операторах одного CA] → монотонность
  обеспечивается стейтом владельца ключа CA; параллельный выпуск CRL двумя
  операторами одного ключа — организационно запрещён и виден в журнале.
- [Vault Transit не проверяет, ЧТО подписывает] → все проверки —
  до подписи, в ядре; политика Vault ограничивает, кто может звать sign.
- [WASM-сборка тянет криптографию в браузер] → в браузере только хэш TBS и
  сборка DER; приватная криптография — всегда за trait подписи вне браузера.

## Migration Plan

Новый компонент: миграций нет. Появление крейта не меняет deb-пакет агента;
CLI/кабинет распространяются отдельными артефактами релиза. Откат — не
использовать инструмент (ручной выпуск остаётся возможным).

   кастомные critical-расширения; если выпуск обязан идти через внешний УЦ,
   роль инструмента сжимается до подготовки запроса + проверки монотонности.
