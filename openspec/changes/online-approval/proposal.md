# Proposal: online-approval

## Why

Code-метод (`qr-login-core`) требует ручного ввода кода с телефона — на connected-устройствах
это лишний шаг: сервер уже принял решение о выдаче, но доставить его может только человек.
Дизайн онлайн-подтверждения (brainstorming 2026-07-04, tessera-ws) добавляет **второе
завершение той же попытки**: устройство при связности регистрирует попытку на сервере, после
approve (SSO/2FA инженера + four-eyes по политике) грант приезжает по сети и вход продолжается
автоматически. Ручной ввод кода остаётся вырожденным случаем при обрыве сети.

Это НЕ третий метод входа и НЕ online-режим проверки: тот же nonce, тот же challenge, тот же
крипто-контракт MAC; офлайн-методы никогда не ждут сеть, недоступность сети убирает только
онлайн-завершение (fallback на ручной ввод — бесплатный, nonce общий).

## What Changes

- Новая capability **online-approval**: онлайн-завершение попытки code-метода.
  - **Регистрация попытки**: при связности `pam_tessera` после генерации challenge шлёт
    `ATTEMPT{challenge}` sync-агенту по локальному IPC; агент доставляет на сервер
    (outbound long-poll). Отсутствие/недоступность агента = штатный офлайн-режим,
    попытка не блокируется и не деградирует.
  - **Ожидание на двух источниках**: `pam_tessera` мультиплексирует ввод кода (prompt/оверлей)
    и `GRANT` от агента; первое валидное завершение выигрывает, nonce консьюмится один раз.
  - **Грант = полный MAC** (без `truncate_N`) по той же канонической формуле контракта
    (`qr-login-core` §8): один верификатор на оба представления, brute-force неравенство
    применимо только к ручному вводу.
  - **Агент не доверен**: грант проверяется локально Engine как и код; подмена бессмысленна,
    replay гасится consumed-state nonce, задержка/дроп = DoS только онлайн-завершения.
  - **Lifecycle**: `ATTEMPT_CANCEL` при отмене/таймауте попытки; грант после консьюма nonce
    или TTL отбрасывается с аудит-событием; ротация nonce (`REFRESH`) обновляет попытку
    на сервере.
- **Модификация qr-login-method** (pending change `qr-login-core`): попытка получает второе
  завершение; терминальные состояния и consumed-state расширяются сценариями гонки
  «грант vs ручной ввод».
- **Модификация logging-audit**: события — попытка зарегистрирована онлайн, грант получен,
  грант отброшен (после консьюма/TTL/невалидный MAC), завершение попытки (онлайн | ручной ввод).

## Capabilities

### New Capabilities

- `online-approval`: регистрация попытки через sync-агент, мультиплекс двух завершений,
  верификация полного MAC, lifecycle попытки на сервере (`ATTEMPT`/`GRANT`/`REFRESH`/
  `ATTEMPT_CANCEL`), fail-closed и деградация в офлайн-завершение.

### Modified Capabilities

- `qr-login-method`: попытка code-метода допускает два завершения (код руками | грант по сети);
  consumed-state и terminal-states покрывают гонку завершений. (Capability вводится pending
  change `qr-login-core` — этот change зависит от него, delta применяется поверх.)
- `logging-audit`: новые события онлайн-завершения (регистрация попытки, грант
  получен/отброшен, способ завершения).

## Impact

- **Код**: `pam_tessera` (мультиплекс источников, IPC-клиент агента), `tessera_core`
  (верификация full-MAC — снятие truncate, гонка консьюма nonce), `tessera_proto`
  (сообщения ATTEMPT/GRANT/REFRESH/ATTEMPT_CANCEL — расширение IPC-контракта
  по паттернам `qr-login-overlay` §socket).
- **Зависимости (блокеры)**:
  1. `qr-login-core` — общий крипто-контракт (Change-0, MAC-каноникализация с Codes)
     и сам code-метод;
  2. **sync-агент** — компонент из дизайна hybrid-fleets ещё не существует; этот change
     специфицирует только auth-сторону IPC. Канал агент↔Control — отдельный change
     (sync-agent) + серверный блок. Онлайн-подтверждение — третий заказчик агента
     (после выгрузки аудита и синхронизации параметров) — аргумент поднять его приоритет.
- **Инвариант платформы**: формулировка hybrid-fleets «online-режима проверки нет и не будет»
  уточняется в tessera-ws (`tessera-platform.md`, `2026-06-07-hybrid-fleets-design.md`):
  офлайн-методы никогда не ждут сеть; связность может только добавлять завершения.
  Приватные спеки правятся отдельным PR в tessera-ws, не этим change.
- **Сервер (вне scope)**: pending-attempts реестр, экран approve, список попыток +
  number-matching (запасной биндинг без камеры) — сторона Codes/`issuance-signals`.
- **Threat-model**: sync-агент входит в auth-путь как недоверенный транспорт — дополнение
  §5.11 (подмена/replay/задержка гранта, флуд ATTEMPT, QoS: login-трафик приоритетнее
  батчей аудита).
