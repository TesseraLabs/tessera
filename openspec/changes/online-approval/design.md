# Design: online-approval

Дизайн-сессия: brainstorming 2026-07-04 (tessera-ws). Контекст: дисплей- и крипто-слой
code-метода — `qr-login-core`/`qr-login-overlay`; sync-агент и инвариант офлайн-семантики —
`tessera-ws/specs/2026-06-07-hybrid-fleets-design.md`; серверные сигналы выдачи —
`tessera-ws/specs/2026-06-07-issuance-signals-design.md`.

## Context

Code-метод: устройство показывает QR → инженер сканирует → Codes (SSO/2FA, four-eyes) →
телефон показывает `truncate_N(MAC(...))` → ручной ввод → локальная проверка Engine.
На connected-устройствах ручной ввод — лишний шаг: решение о выдаче уже принято сервером.

Sync-агент (дизайн hybrid-fleets, компонент ещё не реализован) — единственная сетевая точка
устройства: pull-only outbound mTLS к Control, long-poll; транспорт не доверен, всё содержимое
верифицируется локально. Запланированные классы трафика: манифесты/CRL/политики (вниз),
подписанные команды (вниз), аудит/doctor батчами (вверх). Онлайн-подтверждение добавляет
четвёртый класс: интерактивный login round-trip (попытка вверх, грант вниз).

Роль-учётки не несут личность (`oper@host` — роль, не человек) → сервер не знает, кому слать
push; биндинг попытки к инженеру создаёт сам инженер: скан QR (основной канал) или список
pending-попыток + number-matching на сайте Codes (запасной, без камеры).

## Goals / Non-Goals

**Goals:**

- Второе завершение попытки code-метода: грант по сети → авто-`PAM_SUCCESS`, без ручного ввода.
- Один крипто-контракт: грант = полный MAC по канонической формуле (`qr-login-core` §8);
  `truncate_N` — только представление для ручного ввода.
- Единая попытка: один nonce, оба завершения конкурируют, первое валидное консьюмит nonce.
- Деградация без ослабления: обрыв сети в любой точке → ручной ввод жив (тот же nonce);
  отсутствие агента → штатное поведение code-метода.
- Sync-агент остаётся недоверенным транспортом: ничего не решает, ничего не проверяет.

**Non-Goals:**

- Отдельный «онлайн-метод входа» — не вводится (завершение, не метод).
- Online-режим проверки cert-пути или ослабление офлайн-семантики — исключено.
- Сам sync-агент (канал агент↔Control, enrollment, QoS-механика) — отдельный change;
  здесь только auth-сторона его локального IPC.
- Серверная сторона (pending-реестр, экран approve, number-matching UI) — Codes/issuance.
- Push-нотификации инженеру — сторона Codes, вне device-scope.

## Decisions

1. **Завершение, не метод** (отвергнуто: третий метод с выбором на экране).
   Один nonce/challenge/UX; инженеру не нужно заранее знать о связности устройства;
   fallback бесплатный. Отдельный метод дублировал бы nonce-машинерию и терминальные состояния.

2. **Транспорт — sync-агент, не прямая сеть из PAM** (отвергнуто: HTTPS-клиент в
   `pam_tessera`, как pam_duo). TLS/DNS/парсинг в pre-auth root-контексте PAM — большая новая
   поверхность атаки; вторая точка выхода в сеть ломает модель «вся сеть устройства = агент».
   `pam_tessera` говорит с агентом по локальному unix-socket по паттернам socket-контракта
   `qr-login-overlay` (SO_PEERCRED, 0600, framing, bounded, per-attempt lifecycle).

3. **Грант = полный MAC** (отвергнуто: короткий код — запас прочности калиброван под ручной
   ввод, короткому секрету нечего делать в сетевом канале; отвергнуто: подписанный
   grant-конверт — второй верификатор, распространение/ротация публичного ключа, расхождение
   с контрактом §8). Полный MAC: нулевая новая криптография, один верификатор, brute-force
   неравенство остаётся только про ручной ввод.

4. **Мультиплекс в `pam_tessera`**: ожидание на источниках {PAM prompt | socket оверлея |
   socket агента} — poll/select, без потоков; первое валидное завершение консьюмит nonce
   атомарно, проигравшее отбрасывается с аудит-событием. Authority nonce — только
   `pam_tessera` (инвариант qr-login сохраняется).

5. **Биндинг инженера — со стороны телефона**: скан QR (физическое присутствие у экрана,
   phishing-резистентно) — основной; список pending + number-matching — запасной. Push
   «вслепую» отвергнут: роль-учётка не идентифицирует человека, push-fatigue.

6. **Идентичность и four-eyes**: self-approve через SSO/2FA телефона обязателен; второй
   апрувер — политика роли/уровня (механизм issuance-signals, не дублируется).

7. **Уточнение инварианта платформы** (правка приватных спек отдельным PR): было
   «online-режима проверки нет и не будет»; становится «офлайн-методы никогда не ждут сеть;
   связность может только добавлять завершения; их недоступность возвращает офлайн-завершение,
   не блокирует вход и не ослабляет проверку». Формула hybrid-fleets «атака на сеть возвращает
   в офлайн-модель, а не в ослабленную» сохраняется дословно.

## Поток (happy path)

```
pam_sm_authenticate:
  nonce ← CSPRNG; challenge = (device_id, nonce, role_id, level)
  QR на экран (режимы 1/2/3 — без изменений)
  агент доступен? → ATTEMPT{attempt_id, challenge} → агент → long-poll → PDP
  ждать: { код (prompt/оверлей) | GRANT{attempt_id, mac} | timeout | cancel }
    код:   truncate_N(MAC') == код?      → консьюм nonce → PAM_SUCCESS
    GRANT: MAC' == mac (полный)?         → консьюм nonce → PAM_SUCCESS
    оба невалидны / timeout / cancel     → fail-closed, ATTEMPT_CANCEL агенту
  TOCTOU-перечитка уровня перед PAM_SUCCESS (как в qr-login-core)
```

Ротация nonce (§7 qr-login): `REFRESH` уходит и оверлею, и агенту; сервер держит то же
множество живых nonce, что и устройство.

## Risks / Trade-offs

- [Sync-агент входит в auth-путь] → trust-граница не меняется (транспорт, ничего не решает);
  threat-model §5.11 дополнить: подмена гранта (гасится MAC), replay (consumed-state nonce),
  задержка/дроп (= DoS только онлайн-завершения, офлайн живо), флуд ATTEMPT (rate-limit PDP
  per device + bounded очередь агента).
- [Гонка «грант vs ручной ввод»] → атомарный консьюм nonce в одном процессе (`pam_tessera`
  владеет попыткой); проигравшее завершение — определённое терминальное состояние + аудит.
- [Агент завис/умер посреди ожидания] → IPC non-blocking, таймауты; отсутствие ответа агента
  никогда не блокирует ручной ввод.
- [Компрометация агента] → максимум: DoS онлайн-завершения + знание challenge/гранта попытки;
  грант привязан к (device, nonce, role, level) — использование требует физического присутствия
  у той же попытки; эскалации нет.
- [Число живых nonce × два канала доставки] → brute-force неравенство §9 qr-login не меняется:
  сетевой канал несёт полный MAC (полная энтропия), короткий код по-прежнему только руками.
- [Зависимость от нереализованного sync-агента] → change реализуем частично: IPC-контракт и
  мультиплекс тестируются mock-агентом; сетевая часть ждёт change `sync-agent`.

## Migration Plan

Аддитивно, breaking-изменений нет: без агента (или `connectivity.mode = "offline"`)
поведение code-метода байт-в-байт текущее. Конфиг: онлайн-завершение вкл/выкл per-device
(секция connectivity профиля). Rollback = выключить секцию/убрать агент.

## Open Questions

1. Формат `attempt_id` и его связь с `nonce_id` IPC-контракта оверлея (один реестр попыток).
2. TTL pending-попытки на сервере vs TTL nonce на устройстве — кто короче, поведение на
   рассинхроне (устройство — авторитет своей попытки; сервер бракует раньше — норм).
3. Транспортная сериализация ATTEMPT/GRANT: расширение NDJSON `tessera_proto` v2 или отдельная
   схема соседнего socket? (Решить при имплементации IPC, вместе с change `sync-agent`.)
4. Number-matching: длина сверочного кода и его рендер в режимах 1/2/3 (серверная сторона —
   Codes; устройству — только показать).
