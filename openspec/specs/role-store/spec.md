# role-store Specification

## Purpose

Локальная база ролей устройства: формат среза роли (TOML), загрузка каталога, доверие в
standalone-режиме (права ФС) и managed-режиме (подписанный manifest.toml + anti-rollback),
атомарное обновление набора. Схема парсится целиком в открытой сборке (включая payload-секции,
чей enforcement коммерческий); недоступен только сам enforcement.

Код: `crates/tessera_core/src/role/{schema,store}.rs`; CLI — `tessera-cli role lint` / `role list`.
## Requirements
### Requirement: Формат среза роли

Срез роли ДОЛЖЕН (MUST) быть TOML-файлом `/var/lib/tessera/roles/<role>.toml` со схемой:
`role` (id, `^[a-z][a-z0-9-]{0,15}$`), `version` (u32), `os` (`astra|linux|windows`),
`name` (строка UI/аудита), `level` (u8, порядок в UI; на Astra соответствует уровню МКЦ),
`description` (опц.), `[payload]` (содержимое зависит от `os`), `[session]` (опц.).
`name` — одна строка (локализация — YAGNI, не закладывается). Поля `[session]` —
закрытый whitelist: `max_ttl` (duration) и ресурсные лимиты сессии `memory_max`, `tasks_max`,
`cpu_weight`, `io_weight` (маппятся в systemd per-session лимиты через `pam_set_data`
до pam_systemd). Парсинг ДОЛЖЕН (MUST) быть строгим (`deny_unknown_fields`): неизвестные
поля и битые типы — ошибка валидации; границы парсера: файл ≤ 64 KiB, база ≤ 256 ролей
(превышение = ошибка валидации, не молчаливое усечение). `role` ДОЛЖЕН (MUST) совпадать
с именем файла и неизменяем (переименование = новая роль). Роли несравнимы: ядро
НЕ ДОЛЖНО (MUST NOT) сравнивать роли или уровни «больше/меньше».

#### Scenario: Неизвестное поле в файле роли
- **WHEN** срез содержит поле, отсутствующее в схеме
- **THEN** срез отвергается с ошибкой валидации и audit-событием; остальные роли базы продолжают работать

#### Scenario: Срез чужой ОС
- **WHEN** `os` среза не совпадает с ОС устройства (файл скопирован с другой платформы)
- **THEN** срез отвергается, audit-событие; роль считается отсутствующей

#### Scenario: role не совпадает с именем файла
- **WHEN** `serv.toml` содержит `role = "oper"`
- **THEN** срез отвергается с ошибкой валидации

### Requirement: База ролей и доверие standalone

База ролей ДОЛЖНА (MUST) жить в `/var/lib/tessera/roles/` (root:root, каталог 0755, файлы 0644).
В standalone-режиме (устройство не enrolled) доверие = права файловой системы (модель
`/etc/sudoers.d`); подписи и манифест НЕ ДОЛЖНЫ (MUST NOT) требоваться. `tessera-cli role lint`
ДОЛЖЕН (MUST) валидировать базу; `tessera-cli role list` — показывать роли с версиями.
Схема ДОЛЖНА (MUST) парситься целиком в открытой сборке, включая payload-секции, чей
enforcement коммерческий (mac, selinux) — формат открыт всегда.

#### Scenario: Standalone-устройство без манифеста
- **WHEN** устройство не enrolled и в каталоге нет manifest.toml
- **THEN** база валидна, роли загружаются по правам ФС

#### Scenario: Lint роли с коммерческим payload в открытой сборке
- **WHEN** `tessera-cli role lint` на открытой сборке встречает роль с `mac_mask`
- **THEN** схема валидируется полностью без ошибок (формат открыт; недоступен только enforcement)

### Requirement: Managed-режим — манифест и anti-rollback

При enrolled-устройстве Engine ДОЛЖЕН (MUST) требовать `manifest.toml`: `bundle_version`
(монотонный u64), `os`, перечень `role → {version, sha256}` (hash над сырыми байтами файла),
подпись Control над байтами манифеста. База принимается только если: подпись валидна ключом
Control И `bundle_version` ≥ последнего принятого (персистится) И hash каждого среза сходится.
Любая невалидность — отказ ВСЕЙ базы (fail-closed) с audit-событием critical. Локальные правки
файлов в managed-режиме не действуют (hash расходится). Аварийная локальная роль НЕ ДОЛЖНА
(MUST NOT) существовать.

#### Scenario: Rollback-атака
- **WHEN** подложен старый корректно подписанный манифест с `bundle_version` меньше персистированного
- **THEN** база отвергается целиком, audit critical, вход по кодам невозможен

#### Scenario: Mix-and-match срезов
- **WHEN** при валидном манифесте подменён один срез (hash не сходится)
- **THEN** база отвергается целиком (fail-closed), audit critical

#### Scenario: Локальная правка в managed
- **WHEN** админ устройства руками правит срез на enrolled-устройстве
- **THEN** hash расходится с манифестом, база отвергается — централизованная истина сохраняется

### Requirement: Атомарное обновление набора

Обновление базы ДОЛЖНО (MUST) быть атомарным: новый набор пишется во временный каталог,
валидируется (и верифицируется манифестом в managed), затем подменяется `rename()`.
Частично записанный набор НЕ ДОЛЖЕН (MUST NOT) быть наблюдаем Engine.

#### Scenario: Сбой посреди обновления
- **WHEN** питание пропало между записью срезов нового набора
- **THEN** действующая база — прежний каталог; временный каталог игнорируется/зачищается

### Requirement: Инициализация baseline bundle_version при enrollment

При первом managed-enrollment устройство ДОЛЖНО (MUST) инициализировать персист `bundle_version`
значением из принятого manifest как baseline для anti-rollback `role-store`. Дальнейшие обновления
набора ролей ДОЛЖНЫ (MUST) подчиняться существующему anti-rollback (монотонный `bundle_version`,
fail-closed при откате). Отсутствие предыдущего baseline (чистый клон после flip) НЕ ДОЛЖНО
(MUST NOT) трактоваться как «любой bundle_version принимается» после первичной инициализации.

#### Scenario: Первый manifest задаёт baseline
- **WHEN** на свежеразвёрнутое устройство импортируется первый подписанный manifest с `bundle_version = N`
- **THEN** `N` фиксируется как baseline; последующий manifest с `bundle_version < N` отвергается
