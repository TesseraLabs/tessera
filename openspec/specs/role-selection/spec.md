# role-selection Specification

## Purpose

Выбор роли на логине в `pam_tessera`: явное указание роли (без дефолта) через суффикс
имени учётки `<user>+<role>` или текстовый PAM-prompt, канонизация PAM_USER, атомарный
резолв и проверка покрытия удостоверением, фиксация payload роли в сессии с ограниченным TTL.

Код: `crates/pam_tessera/` (`pam_sm_authenticate`, суффикс-парсер, prompt, резолв из role-store),
`crates/tessera_core/src/role/`.

## Requirements

### Requirement: Явный выбор роли, без дефолта

Вход ДОЛЖЕН (MUST) требовать явного указания роли; дефолтной роли НЕ ДОЛЖНО (MUST NOT)
существовать. Механизмы указания (DM-агностичные, оба обязательны в реализации):
суффикс имени учётки `<user>+<role>` и текстовый PAM-prompt (PAM_TEXT_INFO + PAM_PROMPT_ECHO_ON),
если суффикс не указан. Без указания роли вход НЕ ДОЛЖЕН (MUST NOT) продолжаться.
Нативные UI (GDM choice-list, fly-dm kgreeterplugin, Windows CPFT_COMBOBOX) — опциональный
слой поверх, их отсутствие не влияет на работоспособность.

#### Scenario: Логин без роли в окружении без conversation
- **WHEN** инженер вводит `ivanov` без суффикса и PAM-стек не может показать prompt
- **THEN** вход отклоняется с диагностикой «роль не указана», audit deny

#### Scenario: Выбор по суффиксу через SSH
- **WHEN** инженер подключается `ssh ivanov+serv@device`
- **THEN** запрошенная роль = `serv`, PAM_USER канонизируется в `ivanov`

### Requirement: Канонизация PAM_USER

Модуль ДОЛЖЕН (MUST) парсить суффикс в начале `pam_sm_authenticate` и переписывать PAM_USER
на каноническое имя (`pam_set_item`) ДО любых обращений к учётке и до остальных модулей стека.
Символ `+` ДОЛЖЕН (MUST) быть запрещён в канонических именах учёток (проверка при провижининге).
Синтаксические ошибки (`ivanov+`, `ivanov+a+b`, пустое имя, role_id вне `^[a-z][a-z0-9-]{0,15}$`)
— отказ до аутентификации. В аудит ДОЛЖНЫ (MUST) идти каноническое имя и запрошенная роль
отдельными полями.

#### Scenario: Двойной суффикс
- **WHEN** введено `ivanov+a+b`
- **THEN** синтаксическая ошибка, отказ до аутентификации, audit deny с сырой строкой

#### Scenario: Пустой суффикс
- **WHEN** введено `ivanov+`
- **THEN** синтаксическая ошибка, отказ до аутентификации

### Requirement: Резолв и покрытие — атомарно с аутентификацией

Резолв роли, валидация удостоверения и проверка покрытия ДОЛЖНЫ (MUST) выполняться
одной неразрывной стадией в `pam_sm_authenticate` — между валидацией
удостоверения и фиксацией роли в сессии НЕ ДОЛЖНО (MUST NOT) быть точки подмены роли
(урок polkit CVE-2021-3560). Покрытие: для сертификата — запрошенная роль ∈ списку расширения
allowed-roles (membership, без сравнения уровней); для кода — `role_id` входит во вход MAC
(контракт Tessera Codes). Роль не найдена в базе / не покрыта удостоверением — отказ,
audit deny с (инженер, роль, причина).

#### Scenario: Роль не покрыта сертификатом
- **WHEN** запрошена `admin`, а allowed-roles серта = `[oper, serv]`
- **THEN** отказ входа, audit deny (membership-проверка, никакого «уровень выше — пропустим»)

#### Scenario: Роль отсутствует в базе устройства
- **WHEN** запрошена роль, которой нет в role-store
- **THEN** отказ «роль неизвестна», в prompt-режиме допускается повторный выбор

### Requirement: Фиксация payload в сессии и TTL

Сессия ДОЛЖНА (MUST) фиксировать копию payload роли на момент открытия: изменение или
удаление роли в базе НЕ ДОЛЖНО (MUST NOT) влиять на живую сессию (доживает свой TTL; отзыв
живой сессии — отдельный механизм). TTL сессии = `min(TTL удостоверения, session.max_ttl
роли, глобальный дефолт конфига)`; бессрочная сессия НЕ ДОЛЖНА (MUST NOT) возникать.

#### Scenario: Роль изменена при живой сессии
- **WHEN** Control доставил новую версию роли во время открытой сессии
- **THEN** сессия продолжает работать на payload-копии момента открытия; новая версия действует со следующего входа

#### Scenario: Бессрочный серт без max_ttl
- **WHEN** удостоверение без TTL и роль без `session.max_ttl`
- **THEN** TTL сессии = глобальный дефолт конфига Engine (не бесконечность)

### Requirement: Роль, требующая недоступный enforcement

Если payload роли требует недоступный в сборке backend, вход по этой роли ДОЛЖЕН (MUST) быть
отклонён с явной диагностикой — НЕ молчаливое сужение прав (открытая сборка: `mac_mask`
без ParsecBackend, `selinux` без SELinux-адаптера). Роли, чей payload полностью
покрыт доступными backend'ами (группы/sudo/лимиты), работают в открытой сборке полностью.

#### Scenario: Роль с mac_mask на открытой сборке
- **WHEN** запрошена роль с `mac_mask` на сборке со StubBackend
- **THEN** отказ входа с диагностикой «роль требует МКЦ-enforcement», audit deny; роль только с группами/sudo работает
