# issuer-cabinet Specification

## Purpose

Веб-кабинет выпуска: один статический SPA (WASM-сборка ядра `tessera_issuer`)
без серверной логики. Вся сборка TBS и проверки — на клиенте, стейт — файлы,
работает self-hosted и офлайн; доступные операции выводятся из предъявленного
родительского сертификата. Родство с CLI `issuer` — общее ядро и одни и те же
проверки. Этот раздел фиксирует поведение кабинета: serverless-инварианты,
режим-из-родителя, инвентарь для форм, источник ключа листа, локализацию и
паритет с CLI.

## Requirements
### Requirement: Serverless — логика на клиенте, стейт в файлах

Веб-кабинет ДОЛЖЕН (MUST) быть статическим артефактом (SPA + WASM-ядро) без
серверной логики: все проверки и сборка TBS выполняются на клиенте, весь стейт —
файлы (родительский сертификат, выпущенные сертификаты, журнал, CRL, снапшот
инвентаря). Кабинет ДОЛЖЕН (MUST) полностью работать self-hosted и без сети
(кроме обращений к локальному агенту подписи на 127.0.0.1). Кабинет НЕ ДОЛЖЕН
(MUST NOT) отправлять данные выпуска на какие-либо внешние адреса.

#### Scenario: Выпуск без сети
- **WHEN** кабинет открыт с локальной копии статики без доступа в интернет
- **THEN** полный цикл выпуска (загрузка родителя, сборка TBS, подпись через локальный агент, сохранение артефактов) завершается успешно

#### Scenario: Никаких внешних обращений
- **WHEN** кабинет выполняет любую операцию выпуска
- **THEN** сетевые обращения ограничены локальным агентом на 127.0.0.1

### Requirement: Один кабинет — режим из родительского сертификата

Кабинет ДОЛЖЕН (MUST) быть единственным (один SPA, одна сборка) и выводить
доступные операции из предъявленного родительского сертификата. Родитель-CA с
рамками делегирования (корень парка или CA организации) даёт обе операции
выпуска — подчинённого CA организации с назначением суженных рамок и листа
смены строго внутри рамок родителя; операция выбирается переключателем.
По умолчанию ДОЛЖНА (MUST) предлагаться операция, соответствующая типу
родителя: корень парка → выпуск CA организации, CA организации → выпуск листа
смены. Отдельных сборок или режимов «по ролям пользователей» НЕ ДОЛЖНО
(MUST NOT) существовать; формы и валидация — общие.

#### Scenario: Предъявлен корень парка
- **WHEN** оператор загружает корневой сертификат парка
- **THEN** кабинет предлагает переключатель операций с выбранным по умолчанию выпуском CA организаций с назначением рамок; выпуск листа доступен переключением

#### Scenario: Предъявлен CA организации
- **WHEN** оператор загружает CA организации
- **THEN** кабинет предлагает переключатель операций с выбранным по умолчанию выпуском листов смен; выпуск подчинённого CA доступен переключением; формы ограничены рамками этого CA (значения вне рамок не предлагаются и отвергаются)

#### Scenario: Смена родителя сбрасывает выбор операции
- **WHEN** оператор с выбранной вручную операцией загружает другой родительский сертификат
- **THEN** выбор операции сбрасывается на умолчание для типа нового родителя

#### Scenario: Предъявлен лист или непригодный сертификат
- **WHEN** загруженный сертификат не является CA (или не содержит рамок делегирования)
- **THEN** операции выпуска недоступны, кабинет объясняет причину

### Requirement: Инвентарь для форм — снапшот файлом с проверкой подписи

Источником данных форм (устройства, теги, роли) ДОЛЖЕН (MUST) служить инвентарь,
полученный одним из двух способов: загрузкой снапшот-файла ЛИБО сборкой в самом
кабинете (конструктор). Собранный в кабинете инвентарь эквивалентен
неподписанному (ручному) снапшоту. Если загруженный снапшот подписан, подпись
ДОЛЖНА (MUST) проверяться; снапшот с невалидной подписью ДОЛЖЕН (MUST)
отвергаться. Кабинет ДОЛЖЕН (MUST) отображать происхождение инвентаря
(подписанный экспорт | заполнен вручную) и его возраст. Отсутствие инвентаря
НЕ ДОЛЖНО (MUST NOT) блокировать выпуск — поля заполняются вручную с явной
пометкой.

При наличии инвентаря формы выпуска ДОЛЖНЫ (MUST) использовать его как источник
подсказок, не ограничивая оператора: устройства (`host_binding`) и пользователи
(`user_binding`) предлагаются из инвентаря, но оператор МОЖЕТ (MAY) ввести
значение, которого в инвентаре нет. Если инвентарь перечисляет роли, набор ролей
в форме листа ДОЛЖЕН (MUST) быть пересечением рамок родителя (`allow_roles`
envelope) с ролями инвентаря, и сужение относительно рамок родителя ДОЛЖНО (MUST)
быть заметно оператору; пустой список ролей в инвентаре роли НЕ ограничивает
(эквивалент «инвентарь роли не задаёт») — предлагаются полные рамки родителя.
Инвентарь НЕ ДОЛЖЕН (MUST NOT) расширять выбор за пределы рамок родителя —
envelope остаётся потолком, инвентарь только сужает и подсказывает.

#### Scenario: Битая подпись снапшота
- **WHEN** загружен снапшот, заявленный как подписанный, с невалидной подписью
- **THEN** снапшот отвергается, формы не заполняются его данными

#### Scenario: Ручной снапшот
- **WHEN** загружен неподписанный снапшот (ручной)
- **THEN** формы заполняются, источник помечен как ручной, возраст показан

#### Scenario: Сборка инвентаря конструктором
- **WHEN** оператор собирает инвентарь в кабинете (устройства, пользователи, роли, теги) без загрузки файла
- **THEN** инвентарь принимается как ручной (неподписанный), помечен как заполненный вручную, и питает подсказки форм наравне с загруженным снапшотом

#### Scenario: Инвентарь питает пикеры форм, envelope — потолок
- **WHEN** предъявлен CA организации и загружен (или собран) инвентарь, роли которого — подмножество рамок CA
- **THEN** форма листа предлагает устройства и пользователей из инвентаря (с возможностью ввести своё), а список ролей равен пересечению рамок CA с ролями инвентаря; роль вне рамок CA не предлагается, даже если присутствует в инвентаре

### Requirement: Источник ключа листа — SPKI или CSR

Кабинет и CLI ДОЛЖНЫ (MUST) принимать оба источника ключа выпускаемого листа:
публичный ключ напрямую (SPKI) и CSR (PKCS#10). При загрузке CSR ДОЛЖНЫ (MUST)
отображаться subject и результат проверки самоподписи; предзаполнение форм из
CSR-атрибутов ДОЛЖНО (MUST) быть явно помечено как «запрошено в CSR» и
редактируемо оператором. CSR с невалидной самоподписью НЕ ДОЛЖЕН (MUST NOT)
допускаться к выпуску.

#### Scenario: Выпуск по CSR в кабинете
- **WHEN** оператор загружает CSR-файл инженера
- **THEN** кабинет показывает subject и статус самоподписи, формы предзаполняются с пометкой источника, скоуп задаёт оператор

#### Scenario: CSR с битой самоподписью в кабинете
- **WHEN** загруженный CSR не проходит проверку самоподписи
- **THEN** выпуск недоступен, причина показана оператору

### Requirement: Локализация кабинета

Веб-кабинет ДОЛЖЕН (MUST) поддерживать русский и английский языки интерфейса
(переключатель + автовыбор по языку браузера, fallback — английский).
Технические идентификаторы и значения полей сертификатов НЕ ДОЛЖНЫ (MUST NOT)
переводиться.

#### Scenario: Кабинет в русской локали
- **WHEN** браузер оператора сообщает язык `ru`
- **THEN** интерфейс кабинета открывается на русском; значения полей (subject, роли, OID) отображаются как есть

### Requirement: Паритет CLI и кабинета

CLI `issuer` ДОЛЖЕН (MUST) покрывать те же операции выпуска, что кабинет
(выпуск CA, выпуск листа, выпуск CRL, верификация журнала), тем же ядром и с
теми же проверками. Расхождений в проверках между обёртками НЕ ДОЛЖНО
(MUST NOT) существовать: проверка живёт в ядре, обёртки её не переопределяют.

#### Scenario: Одинаковый отказ в CLI и кабинете
- **WHEN** один и тот же запрос с нарушением монотонности рамок подан через CLI и через кабинет
- **THEN** оба пути отвергают его одной и той же проверкой ядра с эквивалентным объяснением
