# issuer-signing Specification

## Purpose

Сторона подписи выпуска: единый интерфейс «подписать байты TBS ключом слота»
(ключ не покидает токен/HSM/Vault), адаптеры PKCS#11, Vault Transit и файловый
(PKCS#8; ключ в памяти процесса — осознанный компромисс), локальный
агент `issuer serve` (мост «браузер ↔ токен» на 127.0.0.1 с раздачей самого
кабинета по умолчанию), обязательное подтверждение операции оператором на
доверенном канале, кроссплатформенность стороны выпуска и локализация
операторских поверхностей.

## Requirements
### Requirement: Единый интерфейс подписи готового TBS

Все операции подписи ДОЛЖНЫ (MUST) идти через единый интерфейс «подписать байты
TBS ключом с данным идентификатором», возвращающий только подпись и идентификатор
алгоритма. Приватный ключевой материал НЕ ДОЛЖЕН (MUST NOT) проходить через этот
интерфейс ни в одном направлении; операций экспорта приватного ключа в интерфейсе
НЕ ДОЛЖНО (MUST NOT) существовать.

#### Scenario: Подпись через любой адаптер
- **WHEN** ядро передаёт TBS и идентификатор ключа адаптеру
- **THEN** наружу возвращаются только байты подписи и алгоритм; ключевой материал не появляется в памяти ядра

### Requirement: Адаптер PKCS#11 — токен и HSM одним кодом

Адаптер PKCS#11 ДОЛЖЕН (MUST) работать с любым PKCS#11-модулем (аппаратный токен,
HSM, включая ГОСТ-токены, отдающие PKCS#11-механизмы) без специализации по
вендору. PIN ДОЛЖЕН (MUST) запрашиваться только на время операции, затираться
(zeroize) после использования и НЕ ДОЛЖЕН (MUST NOT) попадать в логи, журнал
выпусков или аргументы командной строки.

#### Scenario: Один код — токен и HSM
- **WHEN** конфигурация указывает путь к PKCS#11-модулю HSM вместо токена
- **THEN** подпись выполняется тем же адаптером без ветвления по типу устройства

#### Scenario: PIN не утекает
- **WHEN** операция подписи завершена (успехом или ошибкой)
- **THEN** PIN затёрт в памяти и отсутствует в логах, журнале и списке процессов

### Requirement: Адаптер Vault Transit

Адаптер Vault/OpenBao ДОЛЖЕН (MUST) использовать только Transit-подпись готового
TBS. Выпуск через Vault PKI НЕ ДОЛЖЕН (MUST NOT) использоваться: `encoding/asn1`
в Go не разбирает OID-дуги больше int64, и расширения арки `2.25.<UUID>` через
Vault PKI выпустить нельзя.

#### Scenario: Подпись через Transit
- **WHEN** конфигурация указывает Vault Transit как хранилище ключа CA
- **THEN** ядро выполняет все проверки локально и отправляет на подпись только готовый TBS

### Requirement: Файловый бэкенд подписи

Файловый бэкенд ДОЛЖЕН (MUST) принимать ключ CA в формате PKCS#8 (PEM или DER),
включая зашифрованный `EncryptedPrivateKeyInfo` (PBES2). Поддерживаемые типы
ключей: ECDSA P-256, ECDSA P-384, RSA (подпись RSASSA-PKCS1-v1_5/SHA-256).
ГОСТ-ключи файловым бэкендом НЕ поддерживаются — для ГОСТ остаётся адаптер
PKCS#11; неподдерживаемый тип ключа ДОЛЖЕН (MUST) отвергаться типизированной
ошибкой.

На Unix-платформах файл ключа, доступный группе или остальным (`mode & 0o077 !=
0`), ДОЛЖЕН (MUST) отвергаться до чтения содержимого — и зашифрованный, и
незашифрованный. Незашифрованный ключ ДОЛЖЕН (MUST) приниматься только с
локализованным предупреждением в stderr при каждом старте; рекомендация
документации — зашифрованный PKCS#8, а для прода — PKCS#11/Vault (файловый
бэкенд — принятый риск; фиксируется в threat-model).

Пароль зашифрованного ключа ДОЛЖЕН (MUST) запрашиваться через pinentry, с
fallback на переменную окружения `TESSERA_ISSUER_KEY_PASSPHRASE`; пароль и
расшифрованный DER-буфер ДОЛЖНЫ (MUST) затираться (zeroize) после построения
ключа подписи и НЕ ДОЛЖНЫ (MUST NOT) попадать в аргументы командной строки,
логи или журнал выпусков. Ключ читается и расшифровывается один раз при
построении бэкенда (fail-fast на битом файле или неверном пароле).

Алгоритм подписи ДОЛЖЕН (MUST) выводиться из самого ключа (кривая EC / RSA).
Явно заданный `--algorithm`, не совпадающий с ключом, ДОЛЖЕН (MUST) отвергаться
ошибкой — молчаливое переопределение НЕ ДОЛЖНО (MUST NOT) происходить.
Идентификатор ключа (`--key`) для файлового бэкенда опционален; по умолчанию
берётся имя файла без расширения и используется в `/info`, сводке подтверждения
и журнале.

#### Scenario: Подпись зашифрованным ключом из файла
- **WHEN** оператор запускает выпуск с `--backend file --key-file ca.p8` и вводит верный пароль через pinentry
- **THEN** TBS подписывается алгоритмом, выведенным из ключа; пароль и расшифрованный буфер затёрты; в argv и логах пароля нет

#### Scenario: Файл ключа с широкими правами
- **WHEN** файл ключа доступен группе или остальным (например, mode 0644)
- **THEN** бэкенд отказывает до чтения содержимого, называя фактические права; подпись не выполняется

#### Scenario: Незашифрованный ключ принимается с предупреждением
- **WHEN** `--key-file` указывает на незашифрованный PKCS#8 с правами 0600
- **THEN** выпуск работает, но при старте в stderr печатается локализованное предупреждение о незашифрованном ключе

#### Scenario: Неверный пароль
- **WHEN** пароль из pinentry или `TESSERA_ISSUER_KEY_PASSPHRASE` не расшифровывает ключ
- **THEN** построение бэкенда завершается типизированной ошибкой до каких-либо операций выпуска

#### Scenario: Расхождение --algorithm с ключом
- **WHEN** оператор передал `--algorithm ecdsa-p384`, а в файле — ключ P-256
- **THEN** запуск отвергается ошибкой, называющей оба алгоритма; подпись ключом «не тем» алгоритмом не выполняется

#### Scenario: Неподдерживаемый тип ключа
- **WHEN** файл содержит ключ типа, не входящего в P-256/P-384/RSA (например, ГОСТ или Ed448)
- **THEN** бэкенд отвергает файл типизированной ошибкой с указанием поддерживаемых типов

### Requirement: Локальный агент issuer serve

Агент `issuer serve` ДОЛЖЕН (MUST) слушать только 127.0.0.1. Основной гейт
запросов подписи/информации — **парный токен сессии**, выданный при старте
агента: запрос без валидного токена ДОЛЖЕН (MUST) отвергаться. Origin — вторичный
гейт против cross-origin: если запрос НЕСЁТ заголовок Origin, он ДОЛЖЕН (MUST)
входить в allowlist, иначе запрос отвергается; отсутствие Origin (легитимный
случай same-origin GET, который браузеры не сопровождают Origin) само по себе
НЕ отвергает запрос — его гейтит токен. Такое разделение безопасно: cross-origin
страница не может ни прочитать вшитый токен (same-origin policy), ни поставить
заголовок токена на cross-origin запрос без preflight, который агент
контролирует; DNS-rebind несёт Origin атакующего и отсекается allowlist.

Агент ДОЛЖЕН (MUST) поддерживать выбор бэкенда подписи флагом `--backend
pkcs11|vault|file` (по умолчанию `pkcs11` — вызовы без флага работают как
прежде) с теми же флагами конфигурации бэкендов, что у выпускающих подкоманд
CLI. Гейты агента (127.0.0.1, парный токен, Origin-allowlist, подтверждение
оператора) НЕ ДОЛЖНЫ (MUST NOT) зависеть от выбранного бэкенда. Секрет бэкенда
(PIN PKCS#11, пароль файла ключа) ДОЛЖЕН (MUST) вводиться на стороне агента;
токен Vault читается из окружения агента; кабинет НЕ ДОЛЖЕН (MUST NOT)
принимать, хранить или передавать секреты бэкенда.

Раздача статики кабинета (SPA) на том же 127.0.0.1-origin, что и `/sign`, —
поведение агента **по умолчанию**, когда кабинет доступен (встроен в бинарь
фичей сборки либо указан каталогом). Отдельный флаг отключает раздачу (чистый
режим моста — для случая, когда SPA раздаётся снаружи); если встроенного кабинета
нет и каталог не указан, агент падает в режим моста. Когда агент раздаёт кабинет:
(1) его собственный origin автоматически входит в allowlist (явный
`--allow-origin` не требуется); (2) агент вшивает парный токен сессии в отданную
страницу, чтобы оператор не переписывал адрес и токен вручную. Раздача статики
НЕ ДОЛЖНА (MUST NOT) ослаблять гейты `/sign`: парный токен и подтверждение
оператора остаются обязательными для подписи. Маршруты статики НЕ ДОЛЖНЫ
(MUST NOT) отдавать ничего за пределами набора ассетов кабинета (защита от path
traversal).

Токен, вшитый в отданную страницу, доступен по `GET /` до какого-либо гейта (иначе
браузер не получил бы его для первого запроса) — значит в режиме раздачи парный
токен НЕ является секретом против других локальных процессов на том же хосте,
дотягивающихся до loopback-порта. Анти-локальный-процесс-инвариант в этом режиме
держит НЕ секретность токена, а обязательное подтверждение оператора: локальный
сосед с прочитанным токеном может вызвать диалог подтверждения, но не завершить
подпись без явного «да» оператора. На однопользовательском рабочем месте (целевой
сценарий) сторонних локальных процессов нет.

#### Scenario: Чужой Origin
- **WHEN** запрос к агенту приходит с Origin, отсутствующим в allowlist
- **THEN** запрос отвергается до какого-либо обращения к бэкенду подписи

#### Scenario: Same-origin запрос без Origin
- **WHEN** кабинет, отданный агентом, шлёт same-origin GET (`/info`) — браузер не сопровождает его заголовком Origin — с валидным парным токеном
- **THEN** запрос обслуживается: отсутствие Origin само по себе не отказ, гейтом выступает токен

#### Scenario: Отсутствует парный токен
- **WHEN** запрос содержит допустимый Origin, но парный токен сессии отсутствует или неверен
- **THEN** запрос отвергается; событие фиксируется в выводе агента

#### Scenario: Секрет бэкенда запрашивается локально
- **WHEN** бэкенду нужен секрет: PIN слота — при операции подписи, пароль файла ключа — при старте агента
- **THEN** секрет запрашивается в терминале/pinentry агента; в HTTP-обмене с кабинетом секреты бэкенда не фигурируют ни в какой момент

#### Scenario: Агент поверх Vault Transit
- **WHEN** агент запущен с `--backend vault --vault-addr <…> --key <…>` и валидным `VAULT_TOKEN` в окружении
- **THEN** цикл кабинета работает без изменений (те же гейты и подтверждение); подпись TBS уходит в Transit; токен Vault не фигурирует в HTTP-обмене с кабинетом

#### Scenario: Агент поверх файлового ключа
- **WHEN** агент запущен с `--backend file --key-file <…>`
- **THEN** применяются все требования файлового бэкенда (права файла, пароль, предупреждение о плейн-ключе), а гейты и подтверждение оператора работают как с другими бэкендами

#### Scenario: Запуск без --backend совместим со старой формой
- **WHEN** агент запущен старой командой `issuer serve --module <…> --key <…>` без `--backend`
- **THEN** используется бэкенд PKCS#11, поведение идентично прежнему

#### Scenario: Кабинет отдан агентом same-origin
- **WHEN** агент запущен с раздачей кабинета и оператор открывает его localhost-адрес
- **THEN** кабинет открывается с того же origin, что и `/sign`, с уже подставленными адресом агента и парным токеном; подпись всё равно требует токена в запросе и явного подтверждения оператора

#### Scenario: Статика не обходит гейты подписи
- **WHEN** к агенту, раздающему кабинет, приходит запрос `/sign` без парного токена (или запрос на файл вне набора ассетов кабинета)
- **THEN** `/sign` без токена отвергается как обычно, а запрос файла вне набора ассетов не отдаёт ничего из файловой системы

### Requirement: Подтверждение операции оператором на стороне агента

Агент `issuer serve` ДОЛЖЕН (MUST) перед обращением к модулю подписи разобрать
полученный TBS разделяемым кодом (`tessera_ext`), отобразить оператору в
собственном интерфейсе (терминал/pinentry — тот же доверенный канал, что и
ввод PIN) ключевые поля операции: тип (лист | CA | CRL), subject, срок
действия, роли/рамки/binding'и — и получить явное подтверждение. Без
подтверждения модуль подписи НЕ ДОЛЖЕН (MUST NOT) вызываться. Валидный парный
токен НЕ ДОЛЖЕН (MUST NOT) заменять подтверждение: токен аутентифицирует
кабинет, подтверждение авторизует конкретную операцию. TBS, который не
разбирается разделяемым кодом, ДОЛЖЕН (MUST) отвергаться до подтверждения.

#### Scenario: Зловредный процесс с украденным токеном
- **WHEN** запрос с валидным Origin и парным токеном приходит без ведома оператора
- **THEN** операция отображается в интерфейсе агента и без явного подтверждения оператора подпись не выполняется

#### Scenario: Оператор отклоняет операцию
- **WHEN** оператор отвечает отказом на запрос подтверждения
- **THEN** запрос отвергается с аудит-записью, модуль подписи не вызывается

#### Scenario: Нечитаемый TBS
- **WHEN** присланный TBS не разбирается разделяемым кодом расширений
- **THEN** запрос отвергается до запроса подтверждения — что нельзя показать оператору, то нельзя и подписать

### Requirement: Кроссплатформенность стороны выпуска

CLI `issuer` и агент `issuer serve` ДОЛЖНЫ (MUST) работать на Linux, macOS и
Windows: платформо-специфичные механизмы (путь PKCS#11-модуля, права файлов,
запуск pinentry) ДОЛЖНЫ (MUST) быть абстрагированы, а инварианты безопасности
(127.0.0.1, Origin, парный токен, подтверждение операции) ДОЛЖНЫ (MUST)
сохраняться на каждой платформе. Агент работает как foreground-процесс на время
сессии выпуска; фоновая демонизация/автостарт стороной выпуска НЕ
предоставляются. Enforcement-сторона (агент Tessera на устройствах) остаётся
Linux-only и данным требованием не затрагивается.

#### Scenario: Выпуск с macOS/Windows рабочего места
- **WHEN** диспетчер запускает `issuer serve` на macOS или Windows с PKCS#11-модулем своего токена
- **THEN** цикл выпуска работает идентично Linux: те же проверки, тот же протокол кабинета, тот же порядок подтверждения

### Requirement: Локализация операторских поверхностей

Операторские поверхности стороны выпуска ДОЛЖНЫ (MUST) быть локализованы
минимум на русский и английский: сводка подтверждения в pinentry/терминале,
сообщения `issuer serve`, пользовательский вывод CLI. Локаль
выбирается из конфига или переменных окружения (`TESSERA_ISSUER_LANG`, затем
`LANG`), fallback — английский. Технические идентификаторы (role_id, OID,
subject в RFC 4514, имена полей протокола) НЕ ДОЛЖНЫ (MUST NOT) переводиться.
Локализация НЕ ДОЛЖНА (MUST NOT) менять состав отображаемых полей сводки —
переводятся подписи, не данные.

#### Scenario: Русская локаль оператора
- **WHEN** оператор с `LANG=ru_RU.UTF-8` получает диалог подтверждения выпуска
- **THEN** подписи полей сводки (операция, subject, срок, роли, рамки) — на русском; значения полей — без изменений

#### Scenario: Неизвестная локаль
- **WHEN** локаль окружения не входит в поддерживаемые
- **THEN** поверхности отображаются на английском (fallback), операция не блокируется
