# issuer-signing Specification

## Purpose

Сторона подписи выпуска: единый интерфейс «подписать байты TBS ключом слота»
(ключ не покидает токен/HSM/Vault), адаптеры PKCS#11 (включая выделенный ключ
реестра), Vault Transit и файловый (PKCS#8; ключ в памяти процесса — осознанный
компромисс), кроссплатформенность стороны выпуска (CLI) и локализация
операторских поверхностей. Открытый `issuer` — CLI; локальный агент подписи и
веб-кабинет выпуска поставляются отдельно.

## Requirements
### Requirement: Единый интерфейс подписи готового TBS

Все операции подписи ДОЛЖНЫ (MUST) идти через единый интерфейс «подписать байты
TBS ключом с данным идентификатором», возвращающий только подпись и идентификатор
алгоритма. Приватный ключевой материал НЕ ДОЛЖЕН (MUST NOT) проходить через этот
интерфейс ни в одном направлении; операций экспорта приватного ключа в интерфейсе
НЕ ДОЛЖНО (MUST NOT) существовать.

#### Scenario: Подпись через любой адаптер
- **WHEN** ядро передаёт TBS и идентификатор ключа адаптеру
- **THEN** наружу возвращаются только байты подписи и алгоритм; ключевой материал не появляется в памяти ядра

### Requirement: Адаптер PKCS#11 — токен и HSM одним кодом

Адаптер PKCS#11 ДОЛЖЕН (MUST) работать с любым PKCS#11-модулем (аппаратный токен,
HSM, включая ГОСТ-токены, отдающие PKCS#11-механизмы) без специализации по
вендору. PIN ДОЛЖЕН (MUST) запрашиваться только на время операции, затираться
(zeroize) после использования и НЕ ДОЛЖЕН (MUST NOT) попадать в логи, журнал
выпусков или аргументы командной строки.

#### Scenario: Один код — токен и HSM
- **WHEN** конфигурация указывает путь к PKCS#11-модулю HSM вместо токена
- **THEN** подпись выполняется тем же адаптером без ветвления по типу устройства

#### Scenario: PIN не утекает
- **WHEN** операция подписи завершена (успехом или ошибкой)
- **THEN** PIN затёрт в памяти и отсутствует в логах, журнале и списке процессов

### Requirement: Выделенный ключ реестра в PKCS#11-бэкенде

PKCS#11-бэкенд ДОЛЖЕН (MUST) поддерживать необязательный **выделенный ключ
реестра** — отдельное конфигурируемое поле (второй `CKA_LABEL` в том же
модуле), которым внешний фронтенд подписи подписывает экспортированные реестры
устройств. Ключ реестра ДОЛЖЕН (MUST) быть ECDSA P-256; бэкенд ДОЛЖЕН (MUST)
проверять это при инициализации, читая `CKA_KEY_TYPE` (должен быть `CKK_EC`) и
`CKA_EC_PARAMS` (должен называть кривую P-256) с токена, и отвергать не-P-256
ключ типизированной ошибкой до каких-либо операций подписи (fail-closed).

Ключ реестра НЕ ДОЛЖЕН (MUST NOT) использоваться для подписи TBS выпуска:
сопоставление алгоритма подписи возвращает P-256 для метки ключа реестра и
настроенный алгоритм — для ключа выпуска, поэтому домены подписи (выпуск и
реестр) не смешиваются даже при общем модуле. CLI открытого репозитория ключ
реестра НЕ конфигурирует — это библиотечная возможность бэкенда для внешних
фронтендов подписи.

#### Scenario: Не-P-256 ключ реестра отвергается при инициализации
- **WHEN** бэкенд инициализируется с ключом реестра, который на токене оказывается P-384 (или не-EC)
- **THEN** инициализация завершается типизированной ошибкой до любой операции подписи (fail-closed)

#### Scenario: Ключ реестра не подписывает TBS выпуска
- **WHEN** запрашивается подпись TBS ключом выпуска, при сконфигурированном отдельном ключе реестра
- **THEN** используется настроенный алгоритм выпуска; ключ реестра в подписи TBS выпуска не участвует и всегда сопоставлен P-256

### Requirement: Адаптер Vault Transit

Адаптер Vault/OpenBao ДОЛЖЕН (MUST) использовать только Transit-подпись готового
TBS. Выпуск через Vault PKI НЕ ДОЛЖЕН (MUST NOT) использоваться: `encoding/asn1`
в Go не разбирает OID-дуги больше int64, и расширения арки `2.25.<UUID>` через
Vault PKI выпустить нельзя.

#### Scenario: Подпись через Transit
- **WHEN** конфигурация указывает Vault Transit как хранилище ключа CA
- **THEN** ядро выполняет все проверки локально и отправляет на подпись только готовый TBS

### Requirement: Файловый бэкенд подписи

Файловый бэкенд ДОЛЖЕН (MUST) принимать ключ CA в формате PKCS#8 (PEM или DER),
включая зашифрованный `EncryptedPrivateKeyInfo` (PBES2). Поддерживаемые типы
ключей: ECDSA P-256, ECDSA P-384, RSA (подпись RSASSA-PKCS1-v1_5/SHA-256).
ГОСТ-ключи файловым бэкендом НЕ поддерживаются — для ГОСТ остаётся адаптер
PKCS#11; неподдерживаемый тип ключа ДОЛЖЕН (MUST) отвергаться типизированной
ошибкой.

На Unix-платформах файл ключа, доступный группе или остальным (`mode & 0o077 !=
0`), ДОЛЖЕН (MUST) отвергаться до чтения содержимого — и зашифрованный, и
незашифрованный. Незашифрованный ключ ДОЛЖЕН (MUST) приниматься только с
локализованным предупреждением в stderr при каждом старте; рекомендация
документации — зашифрованный PKCS#8, а для прода — PKCS#11/Vault (файловый
бэкенд — принятый риск; фиксируется в threat-model).

Пароль зашифрованного ключа ДОЛЖЕН (MUST) запрашиваться через pinentry, с
fallback на переменную окружения `TESSERA_ISSUER_KEY_PASSPHRASE`; пароль и
расшифрованный DER-буфер ДОЛЖНЫ (MUST) затираться (zeroize) после построения
ключа подписи и НЕ ДОЛЖНЫ (MUST NOT) попадать в аргументы командной строки,
логи или журнал выпусков. Ключ читается и расшифровывается один раз при
построении бэкенда (fail-fast на битом файле или неверном пароле).

Алгоритм подписи ДОЛЖЕН (MUST) выводиться из самого ключа (кривая EC / RSA).
Явно заданный `--algorithm`, не совпадающий с ключом, ДОЛЖЕН (MUST) отвергаться
ошибкой — молчаливое переопределение НЕ ДОЛЖНО (MUST NOT) происходить.
Идентификатор ключа (`--key`) для файлового бэкенда опционален; по умолчанию
берётся имя файла без расширения и используется в журнале выпусков.

#### Scenario: Подпись зашифрованным ключом из файла
- **WHEN** оператор запускает выпуск с `--backend file --key-file ca.p8` и вводит верный пароль через pinentry
- **THEN** TBS подписывается алгоритмом, выведенным из ключа; пароль и расшифрованный буфер затёрты; в argv и логах пароля нет

#### Scenario: Файл ключа с широкими правами
- **WHEN** файл ключа доступен группе или остальным (например, mode 0644)
- **THEN** бэкенд отказывает до чтения содержимого, называя фактические права; подпись не выполняется

#### Scenario: Незашифрованный ключ принимается с предупреждением
- **WHEN** `--key-file` указывает на незашифрованный PKCS#8 с правами 0600
- **THEN** выпуск работает, но при старте в stderr печатается локализованное предупреждение о незашифрованном ключе

#### Scenario: Неверный пароль
- **WHEN** пароль из pinentry или `TESSERA_ISSUER_KEY_PASSPHRASE` не расшифровывает ключ
- **THEN** построение бэкенда завершается типизированной ошибкой до каких-либо операций выпуска

#### Scenario: Расхождение --algorithm с ключом
- **WHEN** оператор передал `--algorithm ecdsa-p384`, а в файле — ключ P-256
- **THEN** запуск отвергается ошибкой, называющей оба алгоритма; подпись ключом «не тем» алгоритмом не выполняется

#### Scenario: Неподдерживаемый тип ключа
- **WHEN** файл содержит ключ типа, не входящего в P-256/P-384/RSA (например, ГОСТ или Ed448)
- **THEN** бэкенд отвергает файл типизированной ошибкой с указанием поддерживаемых типов

### Requirement: Кроссплатформенность стороны выпуска

CLI `issuer` ДОЛЖЕН (MUST) работать на Linux, macOS и Windows:
платформо-специфичные механизмы (путь PKCS#11-модуля, права файлов, запуск
pinentry) ДОЛЖНЫ (MUST) быть абстрагированы, а инварианты безопасности
(проверки рамок до подписи, обращение с секретами бэкенда) ДОЛЖНЫ (MUST)
сохраняться на каждой платформе. Enforcement-сторона (агент Tessera на
устройствах) остаётся Linux-only и данным требованием не затрагивается.

#### Scenario: Выпуск с macOS/Windows рабочего места
- **WHEN** диспетчер запускает CLI `issuer` на macOS или Windows с PKCS#11-модулем своего токена
- **THEN** цикл выпуска работает идентично Linux: те же проверки, те же бэкенды, тот же формат артефактов

### Requirement: Локализация операторских поверхностей

Операторские поверхности стороны выпуска ДОЛЖНЫ (MUST) быть локализованы
минимум на русский и английский: пользовательский вывод CLI `issuer`
(результатные сообщения об операциях выпуска, статусах журнала и ошибках).
Локаль выбирается из флага `--lang`, затем переменных окружения
(`TESSERA_ISSUER_LANG`, затем `LANG`), fallback — английский. Технические
идентификаторы (role_id, OID, subject в RFC 4514, crlNumber, серийники) НЕ
ДОЛЖНЫ (MUST NOT) переводиться.

#### Scenario: Русская локаль оператора
- **WHEN** оператор с `LANG=ru_RU.UTF-8` запускает выпуск через CLI
- **THEN** результатные сообщения — на русском; технические идентификаторы — без изменений

#### Scenario: Неизвестная локаль
- **WHEN** локаль окружения не входит в поддерживаемые
- **THEN** вывод CLI отображается на английском (fallback), операция не блокируется
