## Requirements
### Requirement: Формат набора тегов устройства

Набор тегов устройства ДОЛЖЕН (MUST) быть отображением `key → value`, где `key` и `value` —
непустые UTF8-строки, и на один `key` приходится ровно один `value`. Повторяющийся ключ в
источнике ДОЛЖЕН (MUST) трактоваться как ошибка формата (fail-closed), а не как «последний
выигрывает». Имена тегов непрозрачны для Engine: Engine НЕ ДОЛЖЕН (MUST NOT) присваивать особую
семантику конкретным именам ключей (`region`, `class` и пр.) — любые ключи обрабатываются
единообразно как данные.

#### Scenario: Повторяющийся ключ в источнике тегов
- **WHEN** источник тегов содержит один и тот же `key` дважды
- **THEN** набор тегов отвергается как malformed (fail-closed), вход по групповому листу не продолжается

#### Scenario: Неизвестный ключ тега
- **WHEN** набор содержит ранее не встречавшийся ключ (например, `vendor`)
- **THEN** он обрабатывается как обычные данные без изменения кода Engine (generic-обработка)

### Requirement: Источник и доверие тегов

В managed-режиме теги ДОЛЖНЫ (MUST) доставляться тем же подписанным манифестом и под тем же
монотонным `bundle_version`, что база ролей (`role-store`): подпись над байтами проверяется,
откат `bundle_version` отвергается (anti-rollback), битая подпись → fail-closed всей базы тегов.
В standalone-режиме источник тегов — локальный файл, доверие которому ДОЛЖНО (MUST) опираться на
права ФС (паритет с standalone `role-store`). Engine ДОЛЖЕН (MUST) читать собственные теги только
из доверенного источника, не из произвольного локального конфига.

#### Scenario: Откат manifest с тегами
- **WHEN** предъявлен подписанный манифест с `bundle_version` меньше уже виденного
- **THEN** манифест отвергается (anti-rollback), прежний набор тегов сохраняется

#### Scenario: Битая подпись manifest
- **WHEN** подпись манифеста не проходит проверку
- **THEN** база тегов не применяется (fail-closed)

### Requirement: Отсутствие тегов

Если у устройства нет применённого набора тегов, Engine ДОЛЖЕН (MUST) трактовать любое
требование `requireTags` как неудовлетворимое → reject (fail-closed для группового делегирования).
Per-host вход без конверта делегирования в цепи (лист с `host_binding=sha256:<host>`, серты без
`delegation_constraints`) НЕ ДОЛЖЕН (MUST NOT) затрагиваться отсутствием тегов.

#### Scenario: Групповой лист на устройстве без тегов
- **WHEN** цепь содержит CA с `requireTags`, а у устройства нет применённых тегов
- **THEN** конверт неудовлетворим → reject (fail-closed)

#### Scenario: Per-host вход без рамок на устройстве без тегов
- **WHEN** лист привязан `host_binding=sha256:<этот host>` и ни один серт цепи не несёт `delegation_constraints`
- **THEN** отсутствие тегов не влияет, вход проверяется как обычно
